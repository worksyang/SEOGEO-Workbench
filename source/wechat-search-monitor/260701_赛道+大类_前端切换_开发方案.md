# 260701 赛道(track)+大类(category) 前端切换 开发方案

## 一、需求重述

系统目前只有 `group`（分组）这一层归属，所有关键词都平铺在“港险”这个隐含的赛道里。未来需要支持：

1. **两个新归属字段**
   - `category`（大类）: 如“资产配置”。
   - `track`（赛道）: 如“港险 / 保费融资 / IUL / CRS / 光通信 / 存储”。
   - 一词可多赛道。
   - 未归类的词，用户在前端手动选择 category/track。

2. **三处前端切换入口**
   - 监控首页顶部：加赛道/大类切换 tab，按 track 过滤当前展示的关键词与账号。
   - “刷新全部关键词”弹窗：加赛道 tab，只刷新当前赛道下的关键词。
   - 关键词管理页面：加赛道/大类切换或筛选入口。

3. **Agent 检索护栏**
   - 通过 `ranking_hits → snapshot → keyword → track` 反查，让 Agent 只在某 track 的命中子集里检索。

4. **非需求（明确不做）**
   - 不改页面视觉、不重做渲染逻辑。
   - 不引入 K 线或行情数据。
   - 不改造调度器：定时刷新仍全量刷。

## 二、核心判断：为什么改动可控

### 2.1 页面是通用渲染器，不绑定“港险”

- `monitor.js:4` 只有 `const DATA_URL = '/api/monitor-data'`；页面只消费 payload，渲染器对赛道无感知。
- 切换赛道 = 换一份 payload，**不是**换一套页面。

### 2.2 最干净的过滤点已天然存在

`monitor_service.py:121 load_monitor_payload()` 在**请求时**把派生数据 `monitor-data.json` 和配置 `keywords.json` 合并：

```python
# monitor_service.py:57-74 _apply_keyword_config_scope
for group in payload.get("groups", []):
    for item in group.get("keywords", []):
        keyword_id = item.get("keyword_id")
        merged = dict(monitor_by_id.get(keyword_id) or _empty_keyword_payload(...))
        merged["keyword"] = item.get("keyword_text", ...)
        merged["keyword_bucket"] = merged.get("keyword_bucket") or group.get("label") or DEFAULT_BUCKET
        keywords.append(merged)
```

**结论**：把 track/category 加到 `keywords.json` 的每个 keyword 上，然后在这里按 `track` 参数过滤，即可按赛道返回 payload。**不需要重建派生层**，也不需要按 track 单独 build monitor-data。

### 2.3 调度器与 track 无关

- `scheduler_service.py:92-121`：调度器只收集 `keyword_ids` 后 POST 到 `/api/refresh-all`。
- `api.py:407-447`：`refresh_all` 对 `keyword_ids` 是纯透传过滤。
- **本期不动调度器**。未来若要“按赛道定时刷”，只需调度器侧按 track 圈 `keyword_ids`，接口签名都不用改。

### 2.4 文章不带赛道，赛道靠命中反查

- `articles.json` 文章本体无赛道字段（正确设计）。
- 完整溯源链已存在：`article ← ranking_hit → snapshot → keyword_id → track`。
- `ranking_hits.json` 41825/41825 条全部能反查到 keyword_id。
- 因此 Agent 检索只需先按 track 圈出 `article_id` 子集，再搜索即可，**不污染全局文章库**。

## 三、数据层设计

### 3.1 字段定义

```json
{
  "keyword_id": "kw_71d5a377",
  "keyword_text": "友邦财富盈活",
  "enabled": true,
  "note": "",
  "created_at": "2026-06-08T19:56:35",
  "updated_at": "2026-06-09T01:18:35",
  "category": "资产配置",
  "tracks": ["港险"]
}
```

- `category`: string, 单值。未设置时为 `null` 或 `"未分类"`。
- `tracks`: list[string], 多值。允许一词跨赛道。

### 3.2 为什么 track/category 存在 keywords.json，而不是 SQLite keyword_settings

现状：
- **结构性归属**（group） → `keywords.json`（配置主表）。
- **人工状态**（topic / bucket / note / pin） → `SQLite keyword_settings`（运行时状态表）。

track/category 属于**结构性归属**：它决定关键词属于哪个赛道/大类，跟 group 同一级别，应该随配置版本化、导出、迁移。塞进 SQLite 会造成“配置和归属分离”，违背 AGENTS.md“事实与人工状态分离”原则。

topic/bucket 之所以进 SQLite，是因为它们可以在运行中不断被 AI/用户修正，是**派生后的人工标注**。track/category 是**分类元数据**，应跟随 keywords.json。

### 3.3 存量迁移

`keyword_config_repo.py:193 _normalize_payload()` 是天然迁移点。读取 keywords.json 后自动补字段：

```python
# _normalize_payload 中循环每个 keyword item 时
item.setdefault("category", None)  # 或按 group 推断一个默认值
item.setdefault("tracks", [])
```

**本期策略**：
- 存量 137 词首次写入时 `tracks` 自动补 `["港险"]`，`category` 自动补 `"资产配置"`（约定）。
- 后续由用户在关键词管理页面手动修正。

> ⚠️ 记一笔：如果未来想按 group 自动推断 track（比如“保费融资词”→“保费融资”），建议在脚本里做一次性映射，但**不要**把这个推断逻辑长期放进 `_normalize_payload`，否则手动修改后会被覆盖。

### 3.4 是否有必要为 track/category 建配置表

**本期不建议**。因为：
- track/category 数量很少（< 20）。
- 允许前端自由输入/选择，减少约束。
- 后续若要做“track 固定枚举 + 顺序/图标”，再升级为 `data/config/tracks.json` 和 `categories.json` 也不迟。

**实现草图**：

```python
def load_monitor_payload(track_filter: str | None = None) -> dict:
    data = _monitor_data_repo().load()
    data = _apply_keyword_config_scope(data, track_filter=track_filter)
    ...

def _apply_keyword_config_scope(data: dict, track_filter: str | None = None) -> dict:
    payload = _keyword_config_repo().load()
    monitor_by_id = {item.get("keyword_id"): item ...}
    window_days = int(data.get("window_days") or 15)
    keywords = []
    for group in payload.get("groups", []):
        for item in group.get("keywords", []):
            keyword_id = item.get("keyword_id")
            tracks = item.get("tracks") or []
            if track_filter and track_filter not in tracks:
                continue
            merged = dict(monitor_by_id.get(keyword_id) or _empty_keyword_payload(item, group, window_days))
            merged["keyword_id"] = keyword_id
            merged["keyword"] = item.get("keyword_text", merged.get("keyword", ""))
            merged["category"] = item.get("category")
            merged["tracks"] = tracks
            merged["keyword_bucket"] = merged.get("keyword_bucket") or group.get("label") or DEFAULT_BUCKET
            keywords.append(merged)
    return {**data, "keywords": keywords}
```

## 六、Agent 检索护栏方案

Agent 检索时带 `track` 参数，后端执行：

```python
def list_articles_by_track(track: str, limit: int = 1000):
    # 1. 找所有属于该 track 的 keyword_id
    kw_ids = {kw["keyword_id"] for kw in config_repo.list_keywords() if track in kw.get("tracks", [])}
    # 2. 找这些 keyword 的所有 snapshot
    snap_ids = {s["snapshot_id"] for s in snapshots if s["keyword_id"] in kw_ids}
    # 3. 找命中这些 snapshot 的 article_id
    article_ids = {h["article_id"] for h in ranking_hits if h["snapshot_id"] in snap_ids}
    # 4. 返回 article 子集给 Agent
    return [a for a in articles if a["article_id"] in article_ids][:limit]
```

> 注：这是一个**逻辑描述**，不是新 API。实际实现时，可以把这套过滤挂到 Agent 调用的搜索函数入口，也可以作为独立的 `/api/articles?track=xxx` 接口。

## 八、风险与待确认事项

| 风险/待确认 | 说明 | 处理方式 |
|---|---|---|
| `track` 字段到底存在哪 | 有建议存 SQLite settings 表，有建议存 keywords.json | **本方案选定 keywords.json**，理由见 3.2 |

## 十一、结语

这个方案的核心思想是：**“赛道切换”只做一件事——在请求时把 payload 按 track 过滤。** 它不动派生层 build 逻辑、不动账号评分、不动调度器、不动页面渲染。前端看到的仍然是同一套 monitor-data，只是内容按赛道换了。

这样可以把“前端切换”和“评分/热度跨赛道隔离”这两个复杂度拆开。前者先上，后者等真正出现 IUL/CRS/光通信等第二、第三条赛道后再评估。
