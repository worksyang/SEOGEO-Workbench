# Agent 语义投影与每日播报协议

## 1. 目的

系统保留现有原始事实层、标准化数据层和人类看板；新增一层 **Agent 语义投影**，让每日播报不再从网页和全量 JSON 中临时拼装判断。

原则：

1. 同一事实层，分别服务人类界面与 Agent。
2. Agent 默认只读万级 Token 的观察包；完整数据仅用于核查。
3. 每个信号都要说明观测范围、时间窗口、置信度、证据 ID 与禁止外推事项。
4. 系统只输出事实、可复算指标和候选信号；行业解释、选题判断仍由 Agent 明确标注为判断。
5. 同一事件只有首次出现、状态变化、方向反转或证据显著增强时，才允许再次进入日报。

## 2. 文件协议

派生输出位于 `data/agent/`：

| 文件 | 用途 |
|---|---|
| `manifest.json` | 数据新鲜度、完整度、算法版本、读取顺序和文件哈希 |
| `daily_brief.json` | 每日默认输入：24 小时文章、账号异动、关键词信号、内容聚集、词表外关联词、候选事件 |
| `daily_brief.md` | 与 JSON 同源的人类可读核查版 |
| `evidence/<evidence_id>.json` | 单篇文章、单个账号或单个关键词的按需下钻证据包 |
| `data/state/agent_claims.json` | 已经推送过的事件账本，控制语义去重 |
| `data/agent/decisions/*.json` | 每次日报实际采用/压制了哪些候选事件的决策记录 |

`data/agent/` 与事件账本均为可重建运行状态，不进入 Git。

## 3. 读取顺序

每日播报 Agent 固定按以下顺序工作：

1. 读取 `AGENTS.md`，确认事实层边界。
2. 读取 `data/agent/manifest.json`；数据过期、批次不完整时停止强结论。
3. 读取 `data/config/agent_metric_dictionary.json`，理解指标口径。
4. 读取 `data/agent/daily_brief.json` 与 `MEMORY.md`。
5. 仅对准备写入报告的候选事件读取 `evidence_id` 对应证据包；必要时才读文章正文。
6. 写执行记录、推送正文和事件决策 JSON。
7. 调用 `scripts/agent_claims.py apply` 写回事件账本，再推送。

禁止默认读取：

- `normalized/monitor-data.json` 全量文件；
- 全量历史记录；
- 网页 DOM、悬浮卡片或截图；
- 无关的大型 Markdown。

## 4. 指标使用边界

| 指标 | 可以回答 | 不能回答 |
|---|---|---|
| 账号分 | 15 天内账号的经典文章、覆盖、持续性与广度 | 账号商业价值、内容质量真相 |
| 时效分 | 最近 3 天冲入 Top3 的规模与新进能力 | 长期经营能力 |
| 当天分 | 今天在监控词下谁的搜索表现更强 | 长期水平 |
| 趋势 | 最近 3 天相对此前 7 天的阅读需求代理信号 | 今天比昨天的真实搜索量 |
| 常态阅读 | 当前关键词的阅读基线估计 | 微信真实总搜索量 |
| 15 日阅读增量 | 有限时间切片下的估算增量 | 每篇文章的真实新增阅读 |
| 外部热度 | 外部搜索需求辅助信号 | 微信搜索当天必然上涨 |
| 阅读/点赞 | 文章当前可观测互动指标 | 推荐曝光、读者年龄、资产或转化 |

阅读估算必须同时看 `observed_share`、`estimated_share`、`coverage_ratio`、`confidence_level` 和模型版本。单独一个数值不能当作实测事实。

## 5. 事件账本

候选事件的稳定键由「信号类型 + 主体」生成。账本记录：

```text
candidate_id
kind / subject / direction
first_reported_at / last_reported_at
last_fingerprint / last_priority / last_state
evidence_ids / report_path
```

事件状态：

```text
emerging（首次） → updated（证据或强度显著变化）
→ persistent（持续但无重要变化） → reversed（方向反转）
→ reappeared（沉寂后重新出现）
```

`persistent` 且最近三天已经推送过的事件默认压制。是否写入日报由 Agent 决定，但写入后必须在决策 JSON 中留下候选 ID。

## 6. 输出质量门禁

1. 每个核心判断都必须附至少一个 `evidence_id`。
2. 不得由阅读、点赞推断年龄、资产、人群画像或推荐曝光。
3. 不得把监控词表内的变化写成全行业事实；必须标记为监控样本信号。
4. 不得把“词表外关联词”写成市场新增词；它只是待核查候选。
5. 算法版本变化后不得直接比较新旧分数。
6. 新观察账号必须显示观察天数，不能直接认定为新号、改名号或矩阵号。
7. 推荐流选题只能基于文章互动、跨账号内容聚集等代理信号，不能写成真实推荐流量事实。

## 7. 验收指标

1. 默认输入小于 25,000 Token。
2. 全部核心结论可下钻到事实证据。
3. 每个估算指标都带数据质量字段。
4. 7 天内相同事件无状态变化时不重复推送。
5. 每次生成的数据包有可验证的 schema、生成时间与算法版本。
