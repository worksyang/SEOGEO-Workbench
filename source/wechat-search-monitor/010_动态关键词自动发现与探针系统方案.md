# 动态关键词自动发现与探针系统方案

> 目标：在每日约 500 次搜索资源的约束下，让系统持续发现新词，但不把模型猜测、系统补全词或一次性事件词直接污染正式监控词库。

## 先看这一页：整套系统其实只有四件事

你不需要理解后面的评分公式，只需要记住这四步：

| 步骤 | 做什么 | 谁负责 | 结果 |
|---|---|---|---|
| **1. 找探针词** | 每日晨报找出正在升温的文章，从原文提取少量可能的核心词 | 现有“港险行业晨报”Claude Task | 每天最多 3 个升温方向、5 个探针词 |
| **2. 找真实词** | 搜索探针，从微信关联词中寻找更接近真实搜索表达的词；下拉词只辅助 | 确定性程序 | 候选真实词 |
| **3. 观察 15 天** | 通过验真的新词先每天刷新；前7天标为早期观察，满15个有效快照日后才允许降频 | 关键词监控系统 | 建立稳定基线 |
| **4. 降频或淘汰** | 有持续价值的词按 1/3/7/15 天降频；无证据、过期事件词归档 | 现有刷新策略 + 候选状态机 | 词库可持续增长 |

一句话概括：

```text
晨报负责“猜得有依据”
→ 微信关联词负责“证明有人这样搜”
→ 15天观察负责“证明不是一天噪声”
→ 分频和归档负责“控制长期成本”
```

### 现有每日播报如何接入

不另建 Claude 任务，直接复用每天 08:00 的“港险行业晨报”：

1. 晨报照常写行业变化，不改变主要内容；
2. 末尾增加一个很短的“今日探针词”附录；
3. 每天最多选择 3 个升温文章/主题，总探针词不超过 5 个；
4. 每个探针必须附文章原文依据；
5. 探针只写入候选文件，**不直接加入正式词库、不启动搜索、不改变刷新频率**。

晨报只负责第一步。第二至第四步必须由程序按关联词、跨日期、反向命中文章和观察天数处理，不能交给晨报凭感觉决定。

### 当前已经启用

四步闭环已于 2026-07-12 上线。你只需要看四个结果：

1. 晨报今天提出了哪些探针；
2. 哪些探针找到了真正的关联词；
3. 哪些关联词通过反向验证进入15天每日观察；
4. 哪些成熟词被降频或归档。

晨报猜错词不会直接污染正式词库：探针每天最多5个，探真搜索每天最多30次；仅下拉词会标为 `suggestion_only`，只有关联词通过反向验证后才会进入 active。

## 一、核心判断

这套系统不能只做“Claude 读文章后自动加词”。那只是把人工猜词换成模型猜词，无法证明用户真的这样搜索。

正确闭环是：

```text
文章事实信号
→ 规则识别值得探测的热文/内容聚集
→ Claude 从原文中提取 1～3 个探针词
→ 轻量搜索探针
→ 收集关联词与下拉词
→ 搜索候选真实词，反向验证能否命中源文章及同主题文章
→ 连续 2～3 轮积累事实证据
→ 达标进入 active；否则继续观察、拒绝或过期
```

需要明确区分三种词：

| 类型 | 含义 | 是否消耗常规监控资源 |
|---|---|---:|
| `seed/probe` | 人或 Claude 猜出的探针，只用于寻找真实表达 | 仅消耗有限探针预算 |
| `candidate` | 在关联词/搜索结果中得到部分验证的候选词 | 消耗有限验证预算 |
| `active` | 已达到确定性门槛的正式监控词 | 按 1/3/7/15 天策略刷新 |

## 二、上线状态

这不是从零开发。当前系统已有：

1. `agent_projection_service.py`：近 24 小时文章、内容聚集、词表外候选、证据包；
2. `article_metric_observations.json`：文章阅读等指标的多次观测事实；
3. `ranking_hits.json`：一篇文章被哪些关键词命中、排名第几；
4. `snapshot_terms.json`：每次搜索得到的关联词和下拉词；
5. `keyword_batch_runner.py`：支持 `fetch_depth=0` 的轻量探针搜索；
6. 关键词 1/3/7/15 天分频和每日 500 次预算；
7. 本机 Claude Code 支持 `-p`、JSON Schema、预算上限和无会话运行。

2026-07-12 已补齐：

- 关联词与下拉词分开保存；下拉词仅为 `suggestion_only`；
- 探针、候选、证据三张独立状态表；
- 自动轻量探针搜索、候选反向命中验证和晋级；
- 新词15天每日观察及1/3/7/15天自动降频；
- 商业价值、低活跃和15天频率联合归档；
- 500次资源拆分为常规450、探真30、人工预留20。

后续增强项仍包括更精细的文章同龄增速和加速度指标；它们只影响晨报挑选探针的优先级，不影响当前“关联词验真—观察—降频—归档”闭环正确运行。

## 三、哪些信号可以触发自动发现

自动发现不应只靠“阅读量高”，建议同时从五条通道发现：

### 1. 单篇文章开始爆发

需要计算文章年龄校准后的指标，而不是直接比较累计阅读量：

- `read_velocity`：相邻两次观测的阅读增量 ÷ 间隔小时；
- `age_bucket_percentile`：与发布后 0～6h、6～24h、1～3d 等同龄文章相比的增速百分位；
- `acceleration`：最近一段增速是否高于上一段；
- `stable_keyword_hit_count`：被多少个观察满 7 天的稳定关键词命中；
- `rank_strength`：按搜索排名加权，Top1 权重最高；
- `cross_run_persistence`：是否跨多轮仍被命中。

不能只依赖新加词命中。若文章全部由观察不足 7 天的新词发现，应降低触发权重。

### 2. 同主题多账号集中发布

如果 24 小时内多个独立账号发布同一实体、产品、事件或问题，即使单篇阅读量不高，也值得提取探针：

- 独立文章数；
- 独立账号数；
- 被稳定关键词命中的数量；
- 搜索结果中的最佳排名；
- 内容标题中的共同短语。

### 3. 词表外关联词异常增加

关联词是比下拉词更强的候选来源，但必须去重：

- 每个“父关键词 × 日期”只计一次 primary 成功快照；
- 单独统计 `related` 和 `suggestion`；
- 统计来自多少不同父词、多少不同日期；
- 计算最近 3 天相对此前 7 天的增量。

### 4. 标题短语异常增加

从近期文章标题中抽取重复实体和动作短语。例如多个账号同时出现“分红实现率”“趸交”“保费融资”“返佣作废”，可以成为探针来源。

标题短语本身不能直接进入 active；它只负责触发探针。

### 5. 已有关键词的覆盖缺口

当同一篇文章被多个相邻关键词命中，但词库中没有一个能准确概括文章核心主题时，说明现有词库可能只是在外围命中，需要寻找更精准的中心词。

## 四、热文触发分

建议使用透明公式输出“探针触发优先级”，不输出业务解释：

```text
article_probe_priority =
  25% × 同龄阅读增速百分位
  + 15% × 阅读加速度
  + 20% × 稳定关键词命中覆盖
  + 15% × 搜索排名强度
  + 15% × 同主题独立账号密度
  + 10% × 跨轮持续性
  - 新词口径依赖惩罚
  - 数据质量惩罚
```

硬性前提：

- 至少有两次有效指标观测，或者有明确的多账号内容聚集；
- 文章正文或摘要可读取；
- 不是纯导航、公司介绍、账号词或明显无关内容；
- 同一篇文章 48 小时内最多触发一次 Claude 提取。

每天最多选择 10 篇文章，每篇最多提取 3 个探针。

## 五、Claude `-p` 的职责与边界

### Claude 只负责

1. 从标题、摘要、正文中提取 1～3 个可能的搜索探针；
2. 给探针标注类型：实体、产品、事件、动作、问题；
3. 返回探针所依据的文章原文片段；
4. 输出结构化 JSON。

### Claude 不负责

- 不直接写 SQLite；
- 不直接修改 active 词库；
- 不直接启动批跑；
- 不决定候选晋级；
- 不生成文章中没有出现、也无法从原文直接推出的长尾词；
- 不把“自然不自然”作为用户真实搜索的证明。

建议单独建立“关键词雷达提取器”，不要复用当前权限很宽、还负责晨报推送的 Claude 任务。

建议调用形态：

```bash
claude -p \
  --input-format text \
  --output-format json \
  --json-schema "$SCHEMA" \
  --max-budget-usd 0.30 \
  --no-session-persistence \
  --permission-mode dontAsk
```

输入是代码提前生成的小型 `discovery_packet.json`，包含文章事实和正文片段；Claude 无需工具权限。程序只接受通过 JSON Schema 校验的输出。

Claude 输出示例：

```json
{
  "article_id": "art_xxx",
  "probes": [
    {
      "text": "港险分红实现率",
      "type": "question_or_metric",
      "source_quote": "各家保司产品分红实现率...",
      "reason": "文章核心反复出现的明确指标名"
    }
  ]
}
```

## 六、探针如何找到真正的搜索词

### 第一轮：搜索 Claude 探针

使用 `fetch_depth=0`，只抓搜索结果、关联词和下拉词，不抓正文。记录：

- 探针是否命中源热文；
- 源热文排名；
- 是否命中同主题其他文章；
- 搜索结果文章集合；
- 关联词；
- 下拉词。

### 第二轮：从关联词中挑候选真实词

候选优先级：

1. 在多个不同探针的关联词中重复出现；
2. 跨日期出现；
3. 关联词位置靠前；
4. 搜索后能反向命中源热文；
5. 能同时命中同主题的多篇文章；
6. 不是当前 active、人工归档或噪声词。

下拉词只作为辅助证据。**仅有下拉词、没有关联词证据的词，不允许自动晋级。**

### 第三轮：搜索候选真实词做反向验证

搜索候选词后验证：

- 源热文是否进入 Top10，Top5 加分；
- 是否命中至少 2 篇同主题文章；
- 搜索结果集合与探针结果集合的 Jaccard 相似度；
- 是否跨两轮仍有稳定结果；
- 是否只命中单一账号的营销文章。

## 七、候选词指标与晋级门槛

每个候选词保存以下可复算字段：

| 指标 | 含义 |
|---|---|
| `related_occurrence_count` | 去重后的关联词出现次数 |
| `related_parent_probe_count` | 来自多少个不同探针/父词 |
| `related_date_count` | 跨多少个日期出现 |
| `best_related_position` | 关联词最佳位置 |
| `suggestion_occurrence_count` | 下拉词出现次数，仅作辅助 |
| `source_article_best_rank` | 搜索候选词后源文章最佳排名 |
| `cluster_article_hit_count` | 命中的同主题文章数 |
| `cluster_account_hit_count` | 命中的独立账号数 |
| `result_set_jaccard` | 与源探针搜索结果的文章集合相似度 |
| `validation_round_count` | 已完成验证轮数 |
| `first_seen_at / last_seen_at` | 首次和最近证据时间 |

建议自动晋级的硬门槛：

1. 不能仅见于下拉词；
2. `related_parent_probe_count >= 2`；
3. `related_date_count >= 2`；
4. 搜索候选词能在 Top10 命中源文章；
5. `cluster_article_hit_count >= 2`；
6. 至少完成 2 轮验证；
7. 不命中导航词、账号词、过期月份词、无关噪声和人工归档名单；
8. 综合事实分 `>= 70`。

建议事实分：

```text
candidate_validation_score =
  30% × 关联词多父词/多日期证据
  + 25% × 源文章反向命中强度
  + 20% × 同主题文章与账号覆盖
  + 15% × 多轮持续性
  + 10% × 关联词位置与搜索结果一致性
  - 下拉词单源惩罚
  - 导航/月份/营销噪声惩罚
```

对于强时效事件，可以设“快速观察”通道：同日来自至少 3 个独立父探针、源文章 Top5、命中至少 3 篇同主题文章时，进入 `temporary_active`，但 14 天后必须复核，不能直接成为永久核心词。

## 八、候选状态机

```text
proposed
→ probe_queued
→ probed
→ validating
→ validated
→ promoted

旁路：
rejected / expired / cooldown / manually_archived
```

建议字段：

```text
candidate_id
candidate_text
candidate_type
status
source_article_ids
source_probe_texts
proposed_at
first_probed_at
last_validated_at
next_validation_at
expires_at
validation_round_count
validation_score
rejection_reason_code
promoted_keyword_id
claude_run_id
evidence_json
```

候选池必须与正式关键词注册表分离。被拒绝或过期的词不删除，保留证据和冷却期，避免系统反复浪费资源测试同一个噪声词。

## 九、晋级后的生命周期

新晋级词：

1. 前 15 天设为每日刷新，保证建立基线；
2. 15 天后进入现有自动分频体系；
3. 活跃且换新明显：1 天；
4. 中等变化：3 天；
5. 稳定：7 天；
6. 长期稳定：15 天；
7. 事件词在 14/30 天到期复核，无持续证据则归档；
8. 人工锁定的频率和人工归档状态优先于自动规则。

## 十、每日 500 次资源分配

建议初始预算：

| 用途 | 每日上限 |
|---|---:|
| 常规到期关键词 | 430 |
| 新探针搜索 | 30 |
| 候选词复验 | 20 |
| 人工/紧急预留 | 20 |
| **合计** | **500** |

规则：

- 常规到期词优先，未跑过的新 active 词仍保持最高优先级；
- 每篇热文最多 3 个探针，每天最多 10 篇；
- 同义或近重复探针先合并，避免重复搜索；
- 探针和候选复验默认 `fetch_depth=0`；
- 20:00 后若探针预算未使用，可归还常规刷新；
- 即使常规词积压，也至少保留 10 次/天的发现预算，避免系统完全失去新词发现能力；
- 每日自动晋级 active 最多 5 个，防止一次异常批次扩大词库。

## 十一、用当前数据说明规则为什么必要

截至 2026-07-11 23:45 的当前观察包：

- `香港保险趸交之王`：关联词 62 次，来自 45 个监控词，适合进入探针验证；
- `港险产品对比`：关联词 15 次，来自 16 个监控词，适合验证；
- `友邦环宇盈活优势`：21 次全部来自下拉词，不能自动晋级；
- `香港友邦保险公众号`：虽有大量关联词证据，但属于导航词，应规则拒绝；
- `香港保险6月优惠`：虽仍在关联词出现，但已过期，应规则拒绝或归档。

这说明“出现次数高”不是充分条件，必须同时看来源类型、跨父词、跨日期、反向命中、主题覆盖和噪声规则。

## 十二、推荐上线顺序

### 阶段 1：观察模式，运行 7 天

- 改进热文增速和词表外候选计算；
- 自动生成 `discovery_packet`；
- Claude 自动提取探针；
- 生成候选报告；
- **不自动搜索、不自动加 active**。

验收：人工查看 Claude 探针是否来自原文、是否足够短、是否明显重复或营销化。

### 阶段 2：自动探针，人工晋级，运行 14 天

- 每日最多自动搜索 30 个探针；
- 自动完成关联词收集和反向验证；
- 候选进入状态机；
- 人工点击晋级或拒绝。

验收：统计探针有效率、仅下拉词比例、反向命中率、人工通过率和每个有效词消耗的搜索次数。

### 阶段 3：受控自动晋级

- 满足硬门槛后自动进入 15 天每日观察；
- 每天最多晋级 5 个；
- 保留人工撤销、锁定和归档；
- 每周输出自动晋级准确率和资源消耗报告。

## 十三、最终建议

> 应该上 Claude `-p`，但把它放在“语义探针生成器”位置，而不是“关键词库管理员”位置。

系统真正的智能不来自 Claude 能猜多少词，而来自这套闭环能持续回答：

1. 为什么这篇文章值得探测；
2. 探针来自文章哪句话；
3. 微信实际返回了哪些关联词；
4. 哪个候选词能反向命中源文章和同主题文章；
5. 候选经过几轮、多少父词、多少日期验证；
6. 为什么晋级、为什么拒绝、消耗了多少搜索资源。

这样词库可以持续增长，但每日压力不会线性增长；预测词、真实候选词、正式监控词也不会再混在一起。
