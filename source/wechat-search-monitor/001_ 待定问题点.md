# 待定问题点

> 状态：开放中  
> 目的：把当前已经识别到、但不适合现在就定死的问题单独拎出来，避免它们混进主架构里影响开发推进。

## 1. 搜索词管理怎么设计

这是当前最明确的待定项。

现状：

- 系统已经有“主关键词”
- 每次搜索还能稳定拿到“搜索下拉词”和“相关搜索”
- 这些词对选题判断很重要，也建议入库

但还没有定的是：

### 1.1 搜索词管理要不要单独做页面

可选方向：

- 单独做一个“词库管理”页面
- 仍放在现有关键词视角中，通过扩展模块展示

### 1.2 相关词和主词是什么关系

还未定清：

- 相关词只是搜索观察信号
- 还是未来也能一键升级成正式监控关键词

### 1.3 搜索词是否需要层级结构

例如：

- 主词
- 相关词
- 下拉词
- 候选词
- 已升级为正式监控词

这套层级未来可能会影响数据模型和页面结构，但当前不建议先做。

### 1.4 搜索词管理的建议结论

当前建议：

- 先把“搜索下拉词 / 相关搜索”稳定入库
- 暂时只作为关键词的附属信号使用
- 不马上做独立页面
- 等第一版监控系统跑稳，再单独设计“词库管理”

## 2. 账号唯一标识何时补 `wechat_biz`

现状：

- 当前默认仍按账号名称工作
- 未来账号改名会带来拆分问题

方向已经明确：

- 预留 `wechat_biz` 字段
- 等上游模块具备能力后补入

当前结论：

- 不阻塞第一版开发
- 但相关字段要预留

## 3. 低价值账号是否需要清理

现状：

- 系统没有“观测账号池”的概念
- 账号是随着关键词搜索自然进入系统的

当前判断：

- 低命中账号先静态保留，不必急着清理
- 以后如有必要，再设计：
  - 冷账号归档
  - 长期未命中清理
  - 重点账号标记

## 4. 正文内容未来是否需要全文检索

当前决定：

- 正文继续以 Markdown 文件落地
- 数据层只保存路径和相关元信息

待定点：

- 未来如果要做全文检索、相似文章搜索、正文级 AI 分析缓存，是否引入：
  - SQLite FTS
  - 独立全文索引
  - 向量库

当前不做。

## 5. 这份清单的使用方式

当前建议：

- 已定的事进“架构计划”
- 未定但已识别的问题进本清单
- 开发时不主动扩散到这些问题上，除非它们开始阻塞当前实现
