## 0. 先给未来接手者一句话结论

这个项目不是“做一个监控页面”这么简单，它真正做的是三件事：

1. 把微信搜索结果这一类原本松散、临时、很难沉淀的事实，稳定转成可复用的数据资产。
2. 在“关键词主视角”和“账号副视角”之间建立一套统一的解释模型，让页面、字段、历史回看、评分体系不互相打架。
3. 在不引入大模型进入主链路的前提下，用纯规则、纯结构化方法，把“今天的榜单”慢慢演进为“可持续研究的行业观测系统”。

如果你把它误判成一个静态网页项目、一个前端展示项目，或者一个简单的 Markdown 解析项目，你后面一定会做错。

这个系统的核心，从来都不是 HTML，而是“如何定义事实、如何定义派生、如何定义研究价值”。

## 1. 当前系统已经发展到什么阶段

截至本文写作时，系统已经从最初的假页面，演进到一个可以真实运行的 Flask 项目，且已经具备如下能力：

1. 可以解析微信搜索结果 Markdown。
2. 可以沉淀结构化事实层 JSON。
3. 可以生成页面派生层 `monitor-data.json`。
4. 可以通过 Flask 启动页面，而不是直接打开本地 HTML。
5. 页面已经具备两个明确视角：
   - 关键词视角
   - 账号透视
6. 页面支持按日期和快照时间点回看历史。
7. 页面支持抽屉查看文章正文 Markdown，并做 Markdown 渲染。
8. 控制层已经支持关键词置顶、`topic` 人工归并、`keyword_bucket` 人工分类。
9. 账号评分体系已经从 V1 进化到 V2。
10. 已完成一轮大规模关键词压力测试，并把“护航搜索”的方法论沉淀为了独立 Skill。

当前一份真实重建结果的大致规模是：

1. `snapshots = 307`
2. `keywords = 169`
3. `accounts = 632`
4. `articles = 1366`
5. `ranking_hits = 3201`

这说明系统已经不是 demo，而是一个开始承受真实数据复杂度的早期产品。

## 2. 最初的需求其实并不清楚，这一点非常关键

如果未来有人只看今天的代码，很容易误以为这个系统从一开始就很清楚：

1. 主视角是关键词。
2. 副视角是账号。
3. 时间按 15 天窗口看。
4. 账号分需要看持续性和广度。

这是错觉。

真实过程恰恰相反：这个系统一开始最大的价值，不是代码，而是用一个“假页面”把需求一步一步逼出来。

最初的纠结集中在三个问题上：

1. 这个系统到底是一个关键词监控器，还是一个账号监控器。
2. 用户真正想看的，到底是“今天谁排在前面”，还是“哪些账号值得长期研究”。
3. 页面应该围绕“文章排行榜”展开，还是围绕“账号影响力”展开。

这三个问题如果一开始就强行拍板，很可能会选错方向。  
所以项目最早的假页面并不是浪费，它承担了非常重要的需求澄清作用。

换句话说，最早的 HTML 不是 UI 原型，而是“需求逼近装置”。

## 3. 需求认知是怎么一步步变化的

### 3.1 第一阶段：视角被现状绑架，误以为自己想看“账号”

用户最初的潜意识其实被现实条件带偏了。

当时只有一个账号在重点观察，因此整个思维会自然滑向：

1. 我想盯这个账号。
2. 我想看这个账号在哪些关键词里排得高。
3. 我想知道这个账号最近表现如何。

这会让系统很容易被设计成一个“账号中心页面”，而不是“关键词监控系统”。

这不是用户想错了，而是现实输入太单一，导致视角发生偏移。

### 3.2 第二阶段：明确承认“关键词才是主轴”

在讨论过程中，用户明确意识到一个更底层的事实：

1. 这个系统的大目标是选题。
2. 选题的入口一定是关键词，而不是账号。
3. 账号的价值，是帮助理解“这个领域里谁值得研究”，但它不是主轴本体。

这一步是整个项目最重要的一次需求校准。

因为一旦承认“关键词才是主视角”，后面很多设计都会发生连锁变化：

1. 页面默认入口应该是关键词，而不是账号。
2. 数据建模的第一实体应该是 `keyword -> snapshot -> ranking_hit`，而不是先建账号主表再去挂数据。
3. 当天榜单的解释权，应该归属于某个关键词快照，而不是某个账号总览。

### 3.3 第三阶段：又发现账号不能被降级成附属品

但问题没有因此结束。

如果系统完全以关键词为主，用户很快又会遇到新的困惑：

1. 我知道某个关键词今天谁在前面了，然后呢？
2. 我怎么知道这里面的哪些账号值得长期盯？
3. 我怎么把“一天的榜单结果”转换成“长期研究名单”？

于是项目进入第三次认知调整：

1. 主视角仍然是关键词。
2. 但账号视角不能缺席。
3. 最终系统不是二选一，而是“一份事实数据，两种分析入口”。

这就是后来确定下来的核心信息架构：

1. 关键词视角负责回答：今天这个词谁强，哪些文章值得看。
2. 账号视角负责回答：最近这个领域里，哪些号值得长期研究。

这一点后来被证明是对的。  
如果当时坚持只做一种视角，后面的整个系统都会很别扭。

## 4. 假页面为什么不是假的，它其实是需求的第一版解释器

很多工程项目会把“假数据页面”视为过渡产物，觉得以后一定会推倒。  
这个项目不是这样。

这里的假页面有两个非常关键的作用：

1. 它先把“用户想怎么看数据”逼出来。
2. 它再反向逼出“后端到底必须提供什么字段”。

例如，最早页面里出现的一些东西，后来都倒逼出了真实字段设计：

1. 关键词热度分
2. 账号视角的 15 天历史
3. 当前榜单和历史透视并存
4. 某个账号点击后跳去账号页
5. 某篇文章点击后打开抽屉正文
6. 同一天多个抓取时间点的切换

如果没有这些“先画出来”的页面元素，字段设计很容易停留在抽象层：

1. 文章表怎么设计
2. 快照表怎么设计
3. 同一天多次抓取该如何回看
4. 账号页的数据究竟来自哪里

所以这个项目的一个重要经验是：

不要低估高保真假页面的价值。  
它不是装饰，它是“需求建模工具”。

## 5. 真正的大转折：从“写死前端假数据”转向“事实层 + 派生层”

项目进入工程化阶段后，第一件真正需要被定清楚的事，不是 UI，而是数据分层。

当时已经看得很清楚：

1. 页面里很多字段都是派生出来的，不是真实事实。
2. 如果直接围绕页面字段建数据库，后面几乎一定返工。
3. 必须先定义“什么是事实”，再定义“什么是页面结果”。

于是系统逐步形成了三个层次：

### 5.1 事实层

只存客观抓取结果，不做业务判断。

典型实体：

1. `keyword`
2. `crawl_snapshot`
3. `snapshot_term`
4. `account`
5. `article`
6. `ranking_hit`

事实层回答的问题只有一个：

`某个关键词，在某个抓取时间点，第 N 名是谁，是哪篇文章，属于哪个账号。`

### 5.2 派生层

由事实层计算出来，允许反复重建。

典型字段：

1. 关键词热度分
2. 账号影响力分
3. 15 天趋势
4. 今日与昨日变化
5. 页面列表排序

派生层最重要的原则是：

它不应成为新的事实源。  
它只是当前业务解释方式的快照。

### 5.3 控制层

由人工维护、不会从抓取数据自然长出来的东西。

例如：

1. 关键词置顶
2. `topic` 人工归并
3. `keyword_bucket` 人工分类
4. 启用停用状态
5. 备注

控制层如果和事实层混在一起，后面每次重建数据都会互相污染。

这一层后面落到了 SQLite 状态表里，这也是 Flask 化之后非常关键的一步。

## 6. 为什么时间最小单位必须是 `captured_at`

这个问题看似技术，实际上是整个系统能否支撑历史回看的关键。

最早非常容易犯的错误是：

1. 每天抓一次，那就按“天”存。
2. 页面也是看日期，那就用日期做主键。

但这个思路很快被否掉了，因为现实业务里存在三个事实：

1. 同一天可能定时抓取一次。
2. 同一天也可能手动复查一次或多次。
3. 补抓也可能发生在同一天或跨天。

如果只按“天”建模，就会立刻出现这些问题：

1. 同一天多个快照会互相覆盖。
2. 页面无法回看“今天上午”和“今天晚上”的差异。
3. 账号页和关键词页的历史解释会变得模糊。

因此项目最终做出的一个非常重要的决定是：

1. 时间最小锚点是 `captured_at`。
2. `snapshot_id` 是一次快照的身份标识。
3. `day` 只是展示层的聚合维度。

这也带来了后面一整套设计：

1. 关键词页先选日期，再选当天的具体抓取时间点。
2. 同一天多个快照全部保留，不覆盖。
3. `is_primary` 只是“当天主快照”的标签，不是事实本体替代品。

这一步非常关键。  
如果未来有人想简化成“每天一条记录”，请谨慎，因为这几乎等于把系统的历史分辨率砍掉。

## 7. Markdown 事实源并不干净，这是整个解析链路最难的地方之一

很多人会低估这个项目的解析难度，因为表面看起来只是“把 Markdown 读一读”。  
真实情况复杂得多。

项目里的 Markdown 不是统一格式，而是至少混杂了以下几类：

1. 搜索结果型 Markdown
2. AI 监控报告型 Markdown
3. Push 推送型 Markdown
4. Top10 对比文本
5. 单篇文章正文 Markdown

真正可作为事实源的，只有搜索结果型 Markdown。  
这一步如果判错，整个事实层都会被污染。

### 7.1 最大的现实问题不是解析难，而是“文件长得像，但语义不同”

例如：

1. 某些文件首行是 `# 关键词`
2. 某些文件首行是 `# 微信搜索监控报告`
3. 某些文件同样有时间、关键词、文章列表，但实际上是派生分析

如果不先明确“哪些文件是一级事实源，哪些只是派生层”，后面数据库一定越来越乱。

### 7.2 `source/` 软链接和顶层重复文件带来了去重问题

项目里存在两种文件来源：

1. `source/` 软链接指向权威数据目录
2. 项目根目录和 `微信搜索结果/` 下还有拷贝、副本、历史整理文件

这会带来一个非常现实的问题：

同一个快照可能被扫描到不止一次。

因此项目后来采用的去重方式不是“按文件路径唯一”，而是按：

`keyword + captured_at`

这一点看起来小，但非常重要。  
因为原始文件组织本身不是规范数据库，它只是人和脚本混合留下来的工作目录。

### 7.3 老格式没有 URL，这直接冲击文章唯一性

这是一个很典型、也很容易被忽视的坑。

项目早期历史文件里，有一部分搜索结果 Markdown 根本没有文章 URL。  
这意味着：

1. 你不能简单把 URL 当作文章主键。
2. 你也不能因为没 URL 就放弃入库。

项目后来采取的策略是：

1. 第一版文章唯一键先按 `account_name + title` 做稳定键。
2. 如果没有真实 URL，则先用 `placeholder://账号/标题` 占位。
3. 后续一旦抓到真实 URL，再回填 `raw_url / normalized_url`。

这一步的经验非常重要：

不要用“理想上最标准的唯一键”压现实数据。  
在现实系统里，更重要的是“先稳定合并同一篇文章”，而不是追求理论完美主键。

### 7.4 正文匹配也不是一对一简单映射

正文 Markdown 的来源和榜单 Markdown 的来源并不总是完美对应。

因此正文匹配逻辑后来被单独抽层，先做文章正文索引，再做文章内容匹配，而不是在主解析器里硬写。

这里涉及几个现实问题：

1. 同标题文章可能有多个版本。
2. 某些正文文件文件名不稳定。
3. 某些正文带原文链接，某些不带。
4. 特殊字符、全半角、标点都会影响匹配。

因此项目后面把这部分拆成：

1. [article_content_indexer.py](/Users/works14/.claude/监控/wechat-ybxhyyh-top3/app/ingest/content/article_content_indexer.py:1)
2. [article_content_matcher.py](/Users/works14/.claude/监控/wechat-ybxhyyh-top3/app/ingest/content/article_content_matcher.py:1)

这一步的本质是承认：

“正文定位”不是事实解析的附属小动作，而是一件独立的工程问题。

## 8. 为什么正文最终没有进数据库，而是继续保留 Markdown

这个决策非常重要，因为很多系统走到这里时，都会本能地想把正文也塞进 SQLite。

项目最终没有这样做，原因很清晰：

1. 正文体量大。
2. 正文本身已经以 Markdown 文件形式存在。
3. 后续 AI 分析、人工阅读、单文复用，本来就更适合直接基于文件。
4. SQLite 更适合存路径、状态、抓取时间、校验信息，而不是把所有 Markdown 正文塞进去。

于是形成的设计是：

1. 正文继续作为外部 Markdown 文件存在。
2. 结构化层只保存 `content_file_path`、`content_status` 等元信息。
3. 页面抽屉按需读取正文，再做 Markdown 渲染。

这既保留了工程简洁性，也保留了正文资产的原生可用性。

如果未来需要全文检索，再讨论 FTS 或独立全文索引，而不是现在过早复杂化。

## 9. 从静态 HTML 到 Flask，不是为了“上框架”，而是为了把状态和事实分开

用户后来明确提出一个方向：

1. 这个业务未来一定会是正式 Python 项目。
2. 未来会有账号密码。
3. 静态文件模式虽然能看，但不适合作为长期底座。

这时有一个很容易做错的地方：

很多人会把“Flask 化”理解成“把页面搬到 Flask 里就行了”。

项目这次重构不是这样做的。  
真正的目标是借 Flask 把边界拉开。

### 9.1 当前结构的真正意义

现在的 `app/` 目录，不是为了好看，而是为了把职责拆开：

1. `ingest/` 负责摄取和重建
2. `repositories/` 负责状态持久化
3. `services/` 负责业务编排
4. `web/` 负责路由和 API
5. `templates/` 和 `static/` 负责页面表现

这件事背后的工程目标只有一个：

不要再让 Parser、聚合、Web、控制状态写在一个文件里。

### 9.2 为什么 `normalized/*.json` 没被废掉

很多人上了 Flask 之后，会想把 JSON 中间层干掉。  
这个项目没有这么做。

原因很现实：

1. JSON 非常适合调试。
2. JSON 适合肉眼检查和回归验证。
3. JSON 让重建链路可见，不会把所有问题都藏进数据库。

所以当前系统采取的是非常实用的双轨思路：

1. `normalized/*.json` 继续作为可重建缓存和检查产物
2. SQLite 只承载控制层状态

这是一种很合理的渐进式工程策略，不要轻易推翻。

### 9.3 真正推动 Flask 化的，不是页面，而是“人工状态不能被重建覆盖”

例如：

1. 关键词置顶
2. `topic` 人工归并
3. `keyword_bucket` 人工分类

这些状态如果写回事实层 JSON，会在每次重建时被冲掉。  
所以它们必须进入独立状态表。

这件事，是本轮 Flask 重构的真正价值。

## 10. 关键词管理是如何从“没有”变成“控制层实体”的

最早系统只有关键词文本本身，并没有“关键词配置”这个概念。  
后来实际使用中，逐步逼出了三类人工控制需求：

1. 关键词置顶
2. `topic` 管理
3. `keyword_bucket` 管理

### 10.1 关键词置顶不是 UI 装饰，而是业务优先级表达

用户很快意识到：

1. 有些关键词即便分数暂时不高，也必须长期放在前面。
2. 热度排序并不能完全表达主观关注度。

于是“置顶”变成了系统里第一个明确的人工状态字段。

### 10.2 `topic` 是第一层人工归并

它解决的问题是：

1. 同一个产品在不同关键词表述下，应该算同一个产品方向。
2. 否则账号页里的“广度”会因为词根变体而被虚增。

这一步一开始看似只是页面优化，后来证明它其实是评分体系的前置条件。

### 10.3 `keyword_bucket` 是第二层人工归并

当项目发展到账号分 V2 时，仅有 `topic` 已经不够了。

因为 `topic` 只回答：

`这是不是同一个产品`

但它不能回答：

`这是不是同一种搜索意图`

于是系统引入了 `keyword_bucket`，用人工词库分类把关键词分进：

1. 热门单品
2. 保司入口词
3. 单品对比词
4. 提领成交词
5. 风控审查词
6. 保费融资词
7. 香港杠杆寿

## 11. 账号分 V1 为什么会出现，而且为什么它不是“错算法”

很多人看到 V2 之后，会本能地把 V1 当成失败版本。  
这是不准确的。

V1 的出现有非常合理的背景。

当时项目已经明确意识到旧问题：

1. 单篇文章打中很多变体词，会把账号分冲得很虚。
2. 只看今天的榜单，没有持续性概念。
3. 只看某一个词，也无法看出一个号是否跨多个产品方向稳定出现。

于是 V1 的方向是：

1. 用 `topic` 做产品词归并
2. 用每天累积分数形成基础分
3. 乘上持续性
4. 再乘上产品广度

这套模型最大的价值不是“完全正确”，而是它第一次把系统从：

`今天谁蹭词多`

推进到了：

`最近 15 天谁整体比较强`

所以 V1 的历史定位是：

1. 它解决了旧世界的问题。
2. 它不是终局。
3. 它是迈向 V2 的必要过渡层。

## 12. 账号分 V1 为什么会变得不公允

项目后期出现了一个很典型的业务证据：

1. 某些账号明明只有 1 篇文章、1 天命中，却因为切中了很多相关词，账号分很高。
2. 反而一些跨多天、跨多文章、跨多个产品方向持续上榜的强账号，分数被压在后面。

这说明系统已经不是“完全错”，而是：

1. 排序方向大体可信
2. 但分差尺度不公允

V1 的主要问题后来被拆解成几类：

### 12.1 `topic` 不是搜索类别

它只能折叠产品词根，但不能区分：

1. 热门单品词
2. 对比词
3. 提领词
4. 风控词
5. 保费融资词

于是同一篇文章切中多个相近角度时，系统容易把它误判成“覆盖很广”。

### 12.2 广度被奖励了两次

1. `base_score` 本身因为命中更多主题就上涨
2. `breadth_multiplier` 又整体再乘一遍

这会让头部账号在大爆发日进一步被放大。

### 12.3 时间窗口没有明显区分最近和更早

15 天总和式的窗口，容易让：

1. 老号的历史残留过强
2. 黑马最近 2-3 天的爆发不够灵敏

### 12.4 乘法结构让头部更夸张

这不是说乘法一定错，而是它在当前业务里会造成：

1. 本来已经很强的账号，被继续整体放大
2. 分差看起来过于夸张
3. 用户主观上会产生“不公允感”

所以 V2 的核心，不是“调小几个系数”，而是“改结构”。

## 13. 账号分 V2 真正改变了什么

V2 最重要的不是数字，而是价值观被冻结了。

### 13.1 价值观先于公式

用户后来明确拍板了几件事：

1. 想奖励覆盖广的账号。
2. 想奖励持续命中的账号。
3. 黑马如果连续 2-3 天很强，应该有能力进前五。
4. 7 天不上榜的老号应该明显掉下去。
5. 不给老号发功勋费。
6. 当前不引入高客单价加权。
7. 当前不让大模型参与实时分类。

这组判断非常重要。  
因为它定义了系统到底在评什么，而不是只定义“怎么算”。

### 13.2 V2 的结构性改变

V2 不是简单调参，而是改了四件大事：

1. 引入 `keyword_bucket`
2. 同天同篇同主题的去重更严格
3. 时间衰减明显偏向最近 7 天
4. 广度奖励从乘法改成加法

### 13.3 现在的 V2 更接近什么

它更像在回答：

`最近 15 天里，哪些账号在最近仍然很活跃、跨多个产品方向、跨多种搜索角度、而且不是靠同一篇文章刷出来的。`

这比 V1 更贴近“长期研究价值”。

### 13.4 V2 并不是最终答案

当前 V2 仍然是一个工作版本，而不是永恒公式。  
它的意义是：

1. 已经把结构改到了比较对的位置
2. 后续主要应做调参，而不是再重构方向

这一点对未来 AI 很关键。  
如果以后继续优化账号分，优先调以下层次：

1. 先看 `keyword_bucket` 是否还缺分类
2. 再看时间衰减是否过强或过弱
3. 再看产品广度和角度广度加分是否过大
4. 最后才微调具体系数

不要重新退回“多关键词命中 = 高价值”的旧逻辑。

## 14. 大规模搜索压力测试为什么是项目里的一个真正里程碑

一个系统在样本很少时，看起来总是容易“差不多对”。  
真正的问题，往往在规模上来之后才暴露。

未来如果有 AI 要再跑长时间关键词批量搜索，不应该重新从零摸索，而应该直接复用这个 Skill 和这套方法论。

## 15. 实现过程中真正踩过的坑

这一节非常重要，因为它记录的是“如果重新来一次，哪些坑必须主动绕开”。

### 15.1 不要过早把页面字段当数据库字段

页面里像：

1. `today_prev`
2. `score`
3. `coverage_days`
4. `history`
5. `hit_days`

这些看起来很自然，但它们全是派生层。  
如果一开始把这些反写进事实层，后面解释体系一变，整个底座都要返工。

### 15.2 不要把“天”误当成最底层时间单位

这会直接毁掉多次抓取、历史回看和手动复查能力。

### 15.3 不要把正文一股脑塞进 SQLite

这只会让早期调试和内容复用变得更麻烦。

### 15.4 不要把人工判断混进解析层

`topic` 和 `keyword_bucket` 是控制层判断，不应写死在 Parser 里。  
否则未来每次改一个分类，都要重跑解析逻辑，边界会变得很脏。

### 15.5 不要低估“老格式历史数据”的破坏力

真实项目里，最麻烦的从来不是新数据，而是历史残留。

在这个系统里，历史样本带来的问题包括：

1. 没有 URL
2. 文件命名不一致
3. 目录结构不统一
4. 同一个快照存在多个副本
5. 正文文件定位不稳定

如果未来再导入历史目录，应该默认它“不规范”，而不是默认它“符合当前标准”。

### 15.6 不要把“Flask 化”误解成“只是启动服务”

Flask 在这里真正解决的是状态和职责分离，而不是单纯的 HTTP 启动。

### 15.7 不要让 AI 总结进入主链路过早

用户非常明确地压住了一件事：

1. 当前阶段追求的是公允性和可解释性
2. 不是追求大模型赋能后的花哨价值

这很重要。  
如果未来有人过早把“今日战况总结”“词语分类”“战场识别”全部交给大模型，系统很可能会失去简洁性和可控性。

## 8. 关键词热度分 V2 → V4（2026-06-15）

**问题**：V2 公式对"无热度的词"做了补偿——把原本用于热度的 50% 权重转嫁给广度（账号数 30%→55%）和丰度（文章数 20%→45%）。结果一个搜索量几乎为零的词（比如财富盈活环宇）靠账号多+文章多就能排到第 3。

**最终口径**：

- 微信搜索量（WSO）和抖音搜索量（DSO）不是两个独立排名指标，而是合成一个"热度值"。
- 合成方式：基础为 `heat_base = WSO归一化 × 50% + DSO归一化 × 50%`，再加单通道高分补偿：`heat = heat_base + (max(WSO归一化, DSO归一化) - heat_base) × 50%`。
- 任一通道有数据就算有热度，进入热度段；缺失通道按 0 参与基础合成，但高分通道会获得补偿加成。
- 两个通道都没有数据，才算无热度，进入后段按丰度排序。
- 广度（追踪账号数）只展示，不参与排序。

**改动**（三处协同）：

- `app/ingest/builders/monitor_builder.py::_compute_kw_score`
  - 删除两段式权重补偿。
  - 新公式：`total = 1.0 + heat`（有热度时，∈[1,2]）/ `total = richness`（无热度时，∈[0,1)）。
  - `heat_base = wso_n × 0.5 + dso_n × 0.5`；`heat = heat_base + (max(wso_n, dso_n) - heat_base) × 0.5`；`has_heat = bool(wso_val or dso_val)`。
  - 广度（tracked_accounts）保留为展示字段，但不再进入 `total`。
- `app/services/monitor_service.py::_sort_keywords`
  - 排序键改用 `segment > inner_metric`：置顶段最前；非置顶有热度段按 `heat` 倒排；无热度段按 `richness` 倒排。
- `app/static/js/monitor.js::renderList`
  - 前端排序同步后端（置顶 > 有热度 > 无热度）。
  - score tag 显示分 = `min(100, round(heat*100))` 或 `min(100, round(richness*100))`，0-100 区间直观。
  - tooltip 文案改为说明 V4 新语义：有热度时微信/抖音 50:50；无热度时按丰度进入后段。
  - 详情面板"热度分解读"重做：仅显示当前主导维度（热度或丰度），广度条置灰标"仅展示"。
