# 关键词监控系统开发计划

> 状态：v1 草案  
> 说明：这是当前阶段的开发计划，不是最终工程排期。重点是让后续 AI Agent 或人工接手时，知道先做什么、后做什么，以及哪些事现在不要提前做。

## 1. 开发目标

第一版目标不是做完全部能力，而是跑通这条主链路：

`上游微信搜索结果 -> 标准 JSON -> 前端页面读取真实数据`

并确保：

- 关键词视角可用
- 账号透视可用
- 时间点/快照回看可用
- 字段逻辑不返工

## 2. 当前已确认的前提

- 主视角：关键词
- 副视角：账号
- 时间最小锚点：`captured_at`
- 榜单看快照，趋势看窗口
- 同一天允许多次抓取，全部保留
- 榜单上限：上游返回多少就存多少
- 正文继续以 Markdown 文件保存，不直接入 SQLite
- 搜索建议词 / 相关搜索要入结构化层
- 第一版优先走轻量链路，再考虑 SQLite

## 3. 推荐开发顺序

## 3.1 第一阶段：冻结数据契约

目标：

- 把现有架构计划收敛成字段级数据字典
- 明确每个字段的来源、类型、是否可空、是否派生

产出：

- `字段数据字典 v1`

完成标准：

- 不再讨论“这个字段到底是不是原始字段”
- 上游字段和前端字段能一一对应

## 3.2 第二阶段：定义标准 JSON 中间层

目标：

- 在不上 SQLite 的前提下，先定义一套标准 JSON 结构
- 这套 JSON 后续可以直接导入 SQLite

建议 JSON 分层：

- `keywords.json`
- `snapshots.json`
- `ranking_hits.json`
- `articles.json`
- `accounts.json`
- `snapshot_terms.json`

或者单关键词单文件结构，也可以，但必须满足：

- 人能直接读
- 机器能稳定解析
- 后续能一键重建数据库

完成标准：

- 拿一个关键词历史样本，可以稳定输出一份规范 JSON

## 3.3 第三阶段：写 Parser

目标：

- 把现有 Markdown / 上游 API 结果统一转换成标准 JSON

输入来源：

- 搜索结果页 Markdown
- 文章详情 Markdown
- 必要时补充上游 API 原始结果

核心任务：

- 解析关键词
- 解析 `captured_at`
- 解析搜索建议词 / 相关搜索
- 解析榜单文章
- 归一化 URL
- 识别正文文件路径

完成标准：

- 指定一个关键词目录，可以稳定生成结构化 JSON

## 3.4 第四阶段：让前端读取真实 JSON

目标：

- 替换掉当前 `monitor.html` 里的手写假数据
- 页面改为读取标准 JSON

范围：

- 关键词列表
- 关键词页快照榜单
- 日期选择 / 时间点回看
- 15 天趋势
- 账号透视
- 正文抽屉

完成标准：

- 选一个现有关键词，页面显示真实历史数据

## 3.5 第五阶段：引入 SQLite

目标：

- 当 JSON 结构稳定后，再正式写入 SQLite

注意：

- 这一步不是第一优先级
- 不要在 Parser 还没稳定前就急着上库

完成标准：

- 同一份 JSON 可导入 SQLite
- SQLite 中的数据能支撑前端查询

## 3.6 第六阶段：自动化与补抓

目标：

- 定时抓取
- 手动复查
- 失败重试
- 补抓历史

完成标准：

- 形成稳定的数据沉淀链路

## 4. 每一阶段要避免的坑

### 4.1 第一阶段不要急着写 SQL

如果字段语义还没稳定，先写 SQL 只会让后面返工更大。

### 4.2 第二阶段不要把页面字段直接当存储字段

页面里像：

- `score`
- `hit_days`
- `prev_day_rank`
- `prev_snapshot_rank`

这些都应视为派生层，不要直接反写进主存储。

### 4.3 第三阶段不要把正文直接塞进数据库

当前项目里，Markdown 文件本身就是最自然、最可复用的正文载体。

### 4.4 第四阶段不要边接真数据边改字段语义

字段语义和前端接线要分开做。

### 4.5 第五阶段不要过早优化“冷账号清理”

第一版先让数据沉淀稳定，冷账号归档不是当前瓶颈。

## 5. 建议的文件组织方向

这里只给方向，不是强制目录结构。

建议后续至少区分：

- `raw/`
  - 原始 Markdown 或上游输出
- `normalized/`
  - 标准 JSON 中间层
- `db/`
  - SQLite 数据库与迁移脚本
- `frontend/`
  - 页面
- `scripts/`
  - 解析与导入脚本
- `docs/`
  - 架构计划、字段字典、待定问题

当前项目可以先不移动现有文件，但后续最好逐步归位。

## 6. 第一版里最值得优先做的代码任务

如果直接开始写代码，我建议按这个顺序：

1. 定义标准 JSON 结构
2. 写搜索结果页 Parser
3. 写正文文件映射逻辑
4. 生成一个关键词的真实 JSON 样本
5. 让 `monitor.html` 读取这份 JSON

## 7. 当前明确后置的内容

以下内容先不要混入第一版：

- 词库管理独立页面
- AI 自动总结
- AI 选题聚类
- 事件流
- 阅读量 / 点赞数
- 冷账号清理机制
- 账号唯一标识 `wechat_biz` 的强依赖

## 8. 下一步最合理的动作

下一步建议直接做：

**编写《字段数据字典 v1》**

因为一旦这份字典写完，后面：

- Parser
- JSON 结构
- SQLite
- 前端接线

都会顺很多。

## 9. 这份计划给后续 AI Agent 的提醒

如果后续由别的 AI Agent 接手，优先遵守这些约束：

- 不要跳过标准 JSON 直接强上数据库
- 不要把正文整段塞进 SQLite
- 不要把前端派生字段写回主事实层
- 不要把“天”当成最底层时间单位
- 不要提前解决还未阻塞的待定问题
