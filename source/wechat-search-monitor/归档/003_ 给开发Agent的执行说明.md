# 给开发 Agent 的执行说明

> 面向：后续接手本项目进行实际开发的 AI Agent  
> 当前默认执行者：DeepSeek 或其他代码型 Agent  
> 目的：减少误判、避免跳步骤、避免在字段未冻结前直接进入重开发

## 1. 你接手前必须先读的文件

请严格按这个顺序阅读：

1. [数据字段与架构计划.md](./数据字段与架构计划.md)
2. [待定问题点.md](./待定问题点.md)
3. [开发计划.md](./开发计划.md)
4. [monitor.html](./monitor.html)

阅读目标：

- 先理解系统目标和字段原则
- 再理解哪些问题已经定了，哪些问题明确待定
- 再理解当前前端假页面已经验证了哪些交互

## 2. 当前阶段你要做的事

当前阶段不是“直接把整个系统开发完”，而是：

**先把数据层和前端真实读数的主链路跑通。**

本阶段主链路：

`上游微信搜索结果 / 本地 Markdown -> 标准 JSON -> monitor.html 读取真实 JSON`

## 3. 当前阶段你不要做的事

请不要主动扩散到这些方向：

- 不要直接重构整个前端 UI
- 不要改页面整体风格
- 不要直接把正文整段塞进 SQLite
- 不要跳过标准 JSON 中间层直接上数据库
- 不要把派生字段写回主事实层
- 不要提前做词库管理页面
- 不要提前引入 AI 总结 / 聚类 / 事件流
- 不要为“冷账号清理”写逻辑
- 不要因为未来可能有 `wechat_biz`，就阻塞当前开发

## 4. 当前已经确认的关键约束

这些约束默认成立，除非用户明确推翻：

- 主视角是关键词，副视角是账号
- 榜单看快照，趋势看时间窗口
- 时间最小锚点是 `captured_at`
- 同一天允许多次抓取，并全部保留
- 榜单上限遵循“上游返回多少就存多少”
- 正文继续保存在 Markdown 文件中，不直接入 SQLite
- 搜索建议词 / 相关搜索需要入结构化层
- 文章唯一身份优先用 `normalized_url`
- 页面默认命中次数理解为 `hit_days`

## 5. 你现在应优先产出的内容

按顺序产出：

1. 《字段数据字典 v1》
2. 标准 JSON 结构定义
3. Parser 脚本
4. 至少一个关键词的真实 JSON 样本
5. 让 `monitor.html` 读取真实 JSON

## 6. 你的第一阶段工作范围

建议你只完成这几个任务：

### 6.1 写字段数据字典

至少包含：

- 字段名
- 所属实体
- 数据类型
- 是否可空
- 来源字段
- 是否派生
- 是否前端直接使用

### 6.2 定义标准 JSON 中间层

要求：

- 可人工审阅
- 可稳定重建
- 可后续导入 SQLite

### 6.3 写 Parser

优先支持：

- 搜索结果页 Markdown
- 文章详情 Markdown

最少要能解析出：

- `keyword`
- `captured_at`
- `search_suggestions`
- `related_searches`
- `rank`
- `title`
- `summary`
- `account_name`
- `publish_time`
- `url`
- `content_file_path`

### 6.4 接前端真实 JSON

要求：

- 不改页面整体信息架构
- 不改现有时间选择逻辑方向
- 只把假数据替换成真实 JSON 读取

## 7. 你提交结果时必须说明的内容

每次提交都应该明确说明：

- 改了哪些文件
- 当前完成到开发计划的哪一阶段
- 还有哪些内容没做
- 有没有偏离当前架构约束
- 有没有发现上游字段不足

## 8. 如果你发现上游模块字段不够

不要直接自行发明字段。

请先把缺口归类成下面三种之一：

1. 阻塞型缺口
   例：拿不到 `captured_at`

2. 可回退缺口
   例：暂时没有 `snapshot_id`，但 `keyword + captured_at` 可以先唯一

3. 未来增强缺口
   例：暂时没有 `wechat_biz`

然后明确告诉用户：

- 缺什么
- 影响哪一层
- 有没有临时替代方案

## 9. 待定问题的处理方式

如果你碰到 [待定问题点.md](./待定问题点.md) 里的问题：

- 不要擅自定死
- 不要为了这些问题中断主链路
- 只有当它明确阻塞当前阶段任务时，才需要抛给用户确认

## 10. 当前最推荐的执行顺序

你现在最合理的执行顺序是：

1. 先写《字段数据字典 v1》
2. 再定义 JSON 结构
3. 再写 Parser
4. 再让前端读取真实 JSON
5. 最后才考虑 SQLite

## 11. 简化判断原则

如果你在实现时犹豫，优先用这三条做判断：

1. 先保留原始事实，不要过早写派生结论
2. 先走轻量中间层，不要急着上数据库
3. 先保证关键词页跑通，再扩散到更复杂能力
