# 项目 Flask 重构方案与预期

更新时间：2026-06-06 23:17 CST

## 1. 背景

当前项目已经完成了两件关键工作：

1. 微信搜索结果 Markdown 已能被解析成结构化 JSON。
2. 假页面已经被打通为真实数据页面，关键词视角与账号视角都已可用。

当前主要问题不是“页面不能看”，而是“代码结构还不够稳定”。如果继续在单个 `monitor.html` 和单个 `parse_search_md.py` 上叠加功能，未来一旦加入 Flask、SQLite、账号密码、手动触发抓取和后台配置，代码会快速熵增。

因此，这一轮重构的目标不是重做功能，而是建立一个长期可扩展的结构底座。

## 2. 本轮重构目标

本轮只做 Flask 最小骨架，不追求一次到位。

### 目标

1. 页面由 Flask 启动，不再依赖单页直读本地文件。
2. 前端通过 API 读取监控数据，而不是直接读取 `normalized/monitor-data.json`。
3. 解析逻辑、聚合逻辑、Web 路由、人工配置状态开始拆开。
4. 为“关键词置顶”“后续登录”“SQLite 状态持久化”预留清晰位置。

### 本轮不做

1. 不强行把所有逻辑一次迁成大型工程。
2. 不在本轮把正文写进数据库。
3. 不在本轮引入完整用户系统和权限模型。
4. 不在本轮加入任务队列、Celery、异步任务框架。
5. 不重做页面视觉风格。

## 3. 架构原则

### 3.1 分三层

#### 事实层

只保存客观抓取结果，不包含人工判断：

- keyword
- snapshot
- snapshot_term
- account
- article
- ranking_hit
- 原始 Markdown 文件路径

事实层的核心要求：

1. 可重复导入。
2. 可校验来源。
3. 可在未来重建派生层。

#### 派生层

由事实层计算出来，允许随时重建：

- 关键词热度分
- 账号影响力分
- 15 天趋势
- 最新快照榜单
- 前后变化
- 页面用的聚合结构

当前的 `normalized/monitor-data.json` 就属于这一层。

#### 控制层

由人工或系统状态驱动，不可混入事实层：

- 关键词置顶
- 关键词启用/停用
- 关键词备注
- 白名单/黑名单
- 用户账号与权限
- 操作日志

### 3.2 职责隔离

以下职责必须分开：

1. Markdown 解析
2. 结构化实体构建
3. 页面聚合视图构建
4. Web API
5. 人工配置状态持久化
6. 未来的调度与重建任务

### 3.3 Parser 不进路由

Flask 路由不直接做大段解析工作。

正确方式是：

1. 解析脚本或服务先生成结构化数据。
2. Web 层只读取结构化结果或数据库。
3. 未来手动“重建”功能也只触发服务，不把业务逻辑写在路由函数里。

## 5. 当前仓库到未来结构的映射

### 当前

- `微信搜索结果/`：原始 Markdown 与正文 Markdown
- `scripts/parse_search_md.py`：解析 + 实体构建 + 派生聚合 + CLI 混在一起
- `normalized/*.json`：中间层和前端派生层
- `monitor.html`：模板、样式、脚本混在一个文件里

### 未来

- `微信搜索结果/` 暂时保留，作为原始事实输入目录
- `scripts/parse_search_md.py` 逐步拆成：
  - `app/ingest/parsers/search_md_parser.py`
  - `app/ingest/builders/monitor_builder.py`
  - `scripts/rebuild_data.py`
- `normalized/` 继续保留，但定位改为：
  - 调试缓存
  - 中间导出物
  - 回归验证样本
- `monitor.html` 拆到：
  - `app/templates/monitor.html`
  - `app/static/css/monitor.css`
  - `app/static/js/monitor.js`

## 6. 数据持久化建议

### 6.1 第一阶段

先允许两套存储共存：

1. `normalized/*.json`
2. `SQLite`

这样做的原因：

1. JSON 便于调试和肉眼核对。
2. SQLite 便于保存配置状态和后续 API 查询。
3. 可以渐进迁移，不必一次性切断现有链路。

### 6.2 哪些适合先入 SQLite

优先入以下内容：

- keyword_settings
- users
- audit_logs
- 手动配置状态

事实层是否立刻全部入 SQLite，可以放到第二步。

### 6.3 关键词置顶字段建议

建议独立一张状态表，不要混进事实数据。

```text
keyword_settings
- keyword_id
- keyword_text
- is_pinned
- pin_order
- is_active
- note
- updated_at
```

排序规则建议：

1. `is_pinned desc`
2. `pin_order asc`
3. `score desc`

## 7. API 方向

第一阶段只需要极少数接口：

### 页面/API

- `GET /`
- `GET /api/monitor-data`
- `GET /api/keywords`

### 关键词控制

- `POST /api/keywords/<keyword_id>/pin`
- `POST /api/keywords/<keyword_id>/unpin`
- `POST /api/keywords/<keyword_id>/settings`

### 后续扩展

- `POST /api/rebuild`
- `POST /api/import-history`
- `GET /api/articles/<article_id>`
- `GET /api/accounts/<account_id>`

## 8. 前端改造原则

1. 页面结构和视觉尽量保持稳定。
2. 先把内联 JS/CSS 拆出来，不急着换前端框架。
3. 页面只调用 Flask API，不再直接读本地 JSON 文件。
4. 关键词视角和账号视角继续共用一份监控数据模型。

## 9. 预期收益

完成这一轮后，项目会获得下面几个确定性收益：

1. 页面启动方式统一，使用体验稳定。
2. “关键词置顶”这类控制状态有明确归属，不会被重建覆盖。
3. 后续接登录、权限、手动重建时，不需要拆掉现有大半代码。
4. Parser、聚合、Web 的边界更清楚，AI Agent 后续接手时不容易越改越乱。
5. 以后从 JSON 逐步过渡到 SQLite，会更平滑。

## 10. 本轮重构的验收标准

### 必达

1. Flask 可以启动首页。
2. 页面可以正常展示真实监控数据。
3. 前端通过 API 获取数据。
4. 页面可以设置关键词置顶，并且重启后不丢。
5. 现有页面风格不被明显破坏。

### 可延后

1. 登录系统
2. 全量 SQLite 化
3. 手动触发重建页面
4. 更完整的后台管理页面

## 11. 实施顺序

建议按这个顺序推进：

1. 落地项目级约束文件
2. 搭 Flask 最小骨架
3. 拆 `monitor.html` 为模板 + 静态资源
4. 前端改成 API 驱动
5. 加关键词置顶状态存储
6. 回归测试
7. 再决定是否推进 SQLite 事实层化

## 12. 结论

这轮重构的重点不是功能数量，而是边界。

只要先把下面三件事彻底分开，后面就会顺很多：

1. 抓取事实
2. 统计派生
3. 人工控制状态

Flask 只是承载这三层的外壳，不应该反过来侵入这三层的职责。
