# 字段数据字典 v1

> 状态：v1，配合 [002_ 数据字段与架构计划.md](./002_%20数据字段与架构计划.md) 一起使用  
> 范围：第一版只覆盖"上游 Markdown → 标准 JSON"主链路所需字段

## 0. 实际数据形态（先于字典）

### 0.1 真正可用的"搜索结果型 Markdown"

满足以下两个条件之一即视为"搜索结果型"：

- 首行匹配 `^# (?P<keyword>.+)$`，且紧接行匹配 `^- 时间：(?P<captured_at>\d{4}-\d{2}-\d{2} \d{2}-\d{2}-\d{2})$`
- 或文件名匹配 `^\d{6}(_\d{6})?_(搜索结果_)?(?P<keyword>.+)\.md$` 且文件内出现 `#### 文章列表`

第一版 Parser 的权威输入只有这一类。

### 0.2 第一版明确忽略的文件

| 类型 | 例子 | 原因 |
|---|---|---|
| 监控报告型 | `260524_080005_wechat-ybxhyyh-top3.md` | 首行是 `# 微信搜索监控报告：…`，属 AI 写的分析报告 = 派生 |
| Push 短文 | `260605_*_push.md`、`260606_push_content.md` | 推送脚本草稿，派生层 |
| Top10 对比 | `260605_*_财富盈活-top10对比.txt` | 派生分析 |
| 文章正文 | `260606_7年回本…登場！.md` 等 | 不是榜单事实，仅作 `article_content.source_file_path` 候选 |

### 0.3 数据源目录

- `source/` 是软链接 → `~/Documents/震凯小助理/震凯的总结/友邦微信监控`，**权威落地处**
- 项目根目录有大量从 `source/` 拷出来的副本，Parser 必须去重（按 `keyword + captured_at` 唯一）
- `微信搜索结果/` 是另一批样本，也参与解析

### 0.4 当前实际样本能覆盖的关键词

第一版只有两个关键词有真实搜索结果数据：

- 友邦环宇盈活
- 友邦财富盈活

monitor.html 当前显示的"宏利卓越隽升"在原始数据中**不存在**，第一版不会生成它。

## 6. `article`（文章元数据）

| 字段 | 类型 | 可空 | 来源 | 派生 | 前端用 | 说明 |
|---|---|---|---|---|---|---|
| `article_id` | string | 否 | `art_<md5(account_name + title)前12位>` | 是 | 否 | 第一版稳定主键 |
| `normalized_url` | string | 否 | `raw_url` 去 query/fragment 后的规范化形式 | 是 | 否 | URL 归一化结果，第一版不直接拿它做唯一主键 |
| `raw_url` | string | 否 | 列表中 `https://mp.weixin.qq.com/s/XXX` 一行 | 否 | 是 | |
| `title` | string | 否 | 列表中 `01. XXX` 行的标题 | 否 | 是 | 最新一次抓取的标题（覆盖式） |
| `account_id` | string | 否 | 列表中 `公众号：XXX` 字段查表得到 | 是 | 是 | |
| `published_at` | datetime | 是 | 列表中 `时间：YYYY-MM-DD HH:MM` | 否 | 是 | 公众号发文时间 |
| `summary` | string | 是 | 列表中 `文章简介：XXX` | 否 | 是 | |
| `first_seen_at` | datetime | 否 | 该文章首次入榜的 `captured_at` | 是 | 否 | |
| `last_seen_at` | datetime | 否 | 该文章最近一次入榜的 `captured_at` | 是 | 是 | |
| `content_status` | enum | 否 | `available`（找到正文 .md）/ `missing`（未找到） | 是 | 是 | |
| `content_file_path` | string | 是 | 项目内匹配到的正文 .md 路径（相对项目根） | 是 | 是 | 第一版按“URL 精确优先，标题规范化精确匹配次之”定位正文 |

**第一版文章唯一键**：

- `canonical_article_key = normalize(account_name) + "::" + normalize(title)`
- `article_id = art_<md5(canonical_article_key)前12位>`
- 原因：老样本存在“先抓到无 URL 榜单，后补抓到真实 URL”的情况；若直接按 URL 建主键，会把同一篇文章拆成两条，污染 `article_count`

**`normalized_url` 规则**：取 `https://mp.weixin.qq.com/s/<id>` 前缀，丢弃 query 和 fragment。若抓取时没有真实 URL，则先落 `placeholder://账号/标题`；后续一旦拿到真实 URL，用真实值回填 `article.raw_url / normalized_url`。

**`content_file_path` 匹配规则（第一版）**：

1. 如果本条 hit 带真实文章 URL，优先用 URL 去正文 Markdown 顶部 `链接：...` 精确匹配
2. 若没有真实 URL，再用标题做规范化（全半角统一、去空白和常见标点）后精确匹配正文标题 / 文件名
3. 命中多个时取最近修改的一个
4. 命中失败 → `content_status = "missing"`

## 12. 已知限制

1. **账号合并按名字精确匹配**，改名会拆为两条；待 `wechat_biz` 上线后修
2. **`trigger_type` 是启发式推断**，08:00 ± 1.5h 即视为定时抓取
3. **monitor.html 现有"宏利卓越隽升"在原始数据中不存在**，第一版 JSON 不含该关键词
4. **历史 5 月份的"监控报告型 Markdown"未参与解析**，因此 5/23 - 6/2 期间没有真实榜单，只有少数老格式 `260516_友邦环宇盈活.md` 系列覆盖
5. **`source/` 与项目根有大量重复文件**，Parser 用 `(keyword, captured_at)` 去重，重复时优先取 `source/` 下版本
6. **第一版文章主键按“账号 + 标题”稳定键合并**；若同一账号长期多次发布完全同标题但不同内容的文章，会被视为同一篇
7. **正文映射已升级为“URL 精确优先 + 标题规范化精确匹配”**；若正文文件标题被人工改写过大，仍可能匹配失败，但不会再做高风险前缀串文
