## 一、先给结论：这不是“6 个长得差不多的页面”

AIDSO 的关键词体系表面上看是统一外壳，实际上是**同一套导航壳 + 多套业务字段模型**。

统一的部分有：

1. 顶部导航条一致。
2. 频道条并非完全固定。主关键词体系里稳定看到 `全站找词 / DSO / RSO / KSO / WSO / BSO`，而 `TSO` 是另一条独立关键词线，页面顶端会直接显示 `30亿/日 / TSO`。
3. 搜索输入区大体一致：`单个 / 批量`、输入框、`挖掘 / 分词`。
4. 左侧都有“分词 / 全部”视图以及结果数量提示。
5. 表格类页面都带“导出”“列表配置”“屏蔽关键词”“添加关键词”等操作区。

真正不一致的部分有：

1. **URL 命名并不统一按简称走**，例如抖音不是 `KeywordDSO`，而是 **`KeywordDouyin`**。
2. **字段完全不统一**，不是简单的“月均搜索量换个名字”。
3. **tab 数量不统一**：
   - 全站找词没有右侧 4 tab 结构。
   - DSO 有 5 个主 tab，还多一个“电商词热搜榜”榜单页。
   - BSO 只有 3 个主 tab，没有行业词。
   - KSO 有 4 个主 tab外加“搜索飙升榜”。
4. **付费墙不统一**：
   - RSO 行业词实测会弹完整会员购买弹窗。
   - DSO 行业词同样会被会员弹窗拦截。
   - BSO 当前搜索页没有行业词 tab，因此不存在同一类拦截点。
5. **指标口径不统一**：
   - WSO 用“月均搜索量 / 月均点击量”。
   - DSO 用“月覆盖人次 / 7日搜索人次 / 下拉词月覆盖”。
   - KSO 用“近7天搜索量 / 下拉词近7天搜索量”。
   - BSO 用“搜索指数 / 下拉词月搜索”。
   - RSO 还有独有字段“市场出价”。
   - 全站找词是聚合视图，直接并列抖、快、红、微、哔多平台指标。

> **一句话判断**：后续抓取方案不能再把它理解成“WSO 模板复制 5 份”，而必须理解成“统一壳下的多平台异构数据模型”。

## 三、页面骨架：哪些是共性，哪些从这里开始分叉

跨板块共用的页面骨架大致如下：

```text
┌──────────────────────────────────────────────────────────────┐
│ 顶部全局导航：DSO / GEO / 首页 / 关键词 / 达人&素材 / AI工具 │
├──────────────────────────────────────────────────────────────┤
│ 频道条：全站找词 / DSO / RSO / KSO / WSO / BSO              │
│ 搜索模式：单个 / 批量                                        │
│ 搜索输入：输入框 + 挖掘 + 分词                               │
├─────────────────────┬────────────────────────────────────────┤
│ 左侧：分词 / 全部   │ 右侧：tab 或聚合列表                   │
│ 结果数量            │ 核心指标筛选                           │
│ 长尾词/分词树       │ 表头 / 表格 / 导出 / 列表配置          │
├─────────────────────┴────────────────────────────────────────┤
│ 页脚 + 客服悬浮 + 安全验证                                  │
└──────────────────────────────────────────────────────────────┘
```

分叉点主要在右侧主数据区：

| 板块 | 右侧结构类型 |
|---|---|
| 全站找词 | 单层聚合表格，无 4 tab 结构 |
| DSO | 5 个主 tab + 1 个榜单页 |
| RSO | 4 个主 tab |
| KSO | 4 个主 tab + 1 个榜单页 |
| WSO | 4 个主 tab |
| BSO | 3 个主 tab |

左侧“分词 / 全部”视图在多个板块都存在，但左侧展示的词树内容、付费墙触发点、结果数规模会随平台变化。

## 五、全站找词板块详解

### 5.1 真实入口

- URL：`https://dso.aidso.com/KeywordAll/searchWord?keyword=奶茶`
- 定位：不是某个平台自身词库，而是**同一关键词在多平台的横向聚合视图**。

### 5.2 页面结构特点

全站找词没有 DSO/RSO/WSO 那种“搜索词 / 相关词 / 下拉词 / 行业词”tab。它更像一个总控面板：

1. 左侧仍然有分词/全部视图与长尾词列表。
2. 右侧直接是一张横向比较表。
3. 表头不是同平台不同指标，而是**同一个关键词在不同平台的对照列**。

### 5.3 当前实测字段

搜索 `奶茶` 时，右侧表头为：

1. `关键词`
2. `字数`
3. `全站下拉词数量`
4. `抖·日均`
5. `快·日均`
6. `红·日均`
7. `微·月均`
8. `哔·搜索指数`

### 5.4 当前样本数据

以 `奶茶` 为例：

- `奶茶`：字数 `2`，全站下拉词数量 `30.4w`，抖 `8.02w`，快 `1.56w`，红 `3.24w`，微 `1.73w`，哔 `10.27w`
- `奶茶店`：`3 / 4.5w / 3795 / 675 / 2301 / 3100 / 7328`
- `七杯奶茶`：`4 / 71 / 8.50w / 767 / 1259 / 100 / 1027`
- `珍珠奶茶`：`4 / 1.0w / - / - / - / 1000 / 1.05w`

### 5.5 这一板块的意义

它不是用来做字段深挖的主战场，而是用来回答：

1. 同一个词在哪个平台更热。
2. 同一个词是否只在某个平台成立。
3. 哪些词有跨平台扩散潜力。

### 5.6 对抓取设计的启发

全站找词不适合简单映射到某一个平台的“标准字段表”。更合理的是：

1. 保留一个 `aggregate_keyword_snapshot` 概念。
2. 将 `douyin_daily`、`kuaishou_daily`、`xhs_daily`、`wechat_monthly`、`bso_index` 视为**异构列**。
3. 空值 `-` 不能被误写成 `0`，因为它经常表示“该平台当前无对应值”。

## 六、DSO 板块详解

### 6.1 真实入口与主特征

- URL：`https://dso.aidso.com/KeywordDouyin/searchWord?keyword=奶茶`
- 页面显示：`抖·关键词`
- 当前是全站里**最复杂**的一类板块之一。

### 6.2 主 tab 结构

DSO 已实测到以下入口：

1. `搜索词`
2. `相关词`
3. `下拉词`
4. `行业词`
5. `电商词`
6. `电商词热搜榜`（独立榜单页，不是普通 tab 表格）

### 6.3 搜索词页字段

以 `奶茶` 搜索词页为例，表头字段有：

1. `月覆盖人次`
2. `7日搜索人次`
3. `字数`
4. `下拉词数量`
5. `下拉词月覆盖`
6. `类型`
7. `竞争度`

样本数据：

- `奶茶`：月覆盖人次 `5373.55w`，7日搜索人次 `平均:7.54w`，字数 `2`，下拉词数量 `30.4w`，下拉词月覆盖 `2.57亿`
- `奶茶店`：`484.67w / 平均:3489 / 3 / 4.5w / 2404.69w / 蓝海词`
- `一点点奶茶`：类型出现 `黑马词 / 同行买词 / 蓝海词`

### 6.4 DSO 的字段特点

1. **覆盖口径替代“搜索量”口径**：不是“月均搜索量”，而是“月覆盖人次”。
2. `7日搜索人次` 列有时是 `平均:xxxx` 格式，有时空。
3. `下拉词月覆盖` 可能到 `亿` 级。
4. `竞争度` 在首屏文本里不总是有明确中文值，存在视觉渲染优先的情况。
5. 标签字段放在 `类型` 列里，而不是 WSO/RSO 的 `推荐理由` 列。

### 6.5 DSO 的相关词页与下拉词页

继续直达 `correlationWord / sugWord` 后，可以把 DSO 三个 tab 的列名差异明确下来：

1. **搜索词页**
   - `月覆盖人次 / 7日搜索人次 / 字数 / 下拉词数量 / 下拉词月覆盖 / 类型 / 竞争度`
2. **相关词页**
   - 页头核心指标显示：`相关词月覆盖 / 竞争度 / 类型`
   - 表头为：`7日搜索人次 / 字数 / 相关词月覆盖 / 类型 / 竞争度`
   - 当前 `奶茶` 样本下，相关词页实测出现了表头和左侧词树，但主表数据区为空，这本身要记成一种状态。
3. **下拉词页**
   - 表头为：`月覆盖人次 / 7日搜索人次 / 字数 / 类型 / 竞争度`
   - 相比搜索词页，**不再显示** `下拉词数量 / 下拉词月覆盖`
   - 操作区还出现了 `剧本 / 写作 / 做图`

这说明 DSO 的 `相关词` 和 `下拉词` 不是简单复用搜索词字段，而是各自裁掉了一部分列。

### 6.6 DSO 的冷门词异常状态

搜索 `安盛盛利2` 时，出现了大量：

- `点击收录`
- `数据更新中，点击收录完整信息，收录完成即可查看。`

这说明 DSO 里除了普通“无数据”外，还存在另一类状态：

> **词已出现在列表，但详细指标尚未完成平台收录，需显式点击触发或等待更新。**

这对爬虫极重要，因为：

1. 不能把这种状态等同于 `0`。
2. 不能等同于“关键词不存在”。
3. 需要单独标记为 `collect_required` 或 `status = pending_index`。

### 6.7 DSO 行业词付费墙

点击 `行业词` 后，页面直接出现会员拦截弹窗，核心文案包括：

1. `权限不足，升级会员版本可查看更多数据`
2. `旗舰版 / 企业版`
3. `您当前是个人版会员`
4. 特权清单中出现：
   - `DSO关键词-行业词库`
   - `DSO关键词-详情-电商`
   - `DSO关键词-榜单-筛选`

这说明 DSO 的行业词和部分深度能力是权限受控的。

4. `订单量`
5. `销售指数`

可切换分类包括：

- `3C数码家电`
- `图书教育`
- `美妆`
- `钟表配饰`
- `医疗健康`
- `个护家清`
- `鲜花园艺`
- `珠宝文玩`
- `二手商品`
- `礼品文创`
- `服饰内衣`
- `智能家居`
- `母婴宠物`
- `本地生活`

### 6.9 DSO 榜单层：不是一个榜单，而是一整组榜单产品

本轮继续深挖后，确认 DSO 榜单层不是单页，而是至少 8 个并列入口：

1. `小蓝词榜`
2. `电商词热搜榜`
3. `电商词飙升榜`
4. `电商词热销榜`
5. `行业词热搜榜`
6. `行业词高跑量榜`
7. `行业词飙升榜`
8. `话题总榜`
9. `话题飙升榜`

其中字段模型至少已经分成 4 类：

1. **小蓝词榜模型**
   - 字段：`月覆盖人次 / 7日搜索人次 / 竞争度 / 视频数 / 点赞总量 / 搜索点击率 / 搜索成交率`
   - 还有 `排名` 列与 `写作` 操作。
   - 样本词包括：`修车躺板`、`纯棉吊带`、`烤羊腿`。
2. **电商榜模型**
   - 字段：`月覆盖人次 / 7日搜索人次 / 搜索点击率 / 订单量 / 销售指数 / GPMS指数 / 搜索成交率`
   - 支持 `分类`、`周榜 / 月榜`、日期区间。
   - 样本词包括：`领丰金官方旗舰店`、`菜百金条`、`华为官方旗舰店`。
3. **行业榜模型**
   - 字段：`搜索热度 / 消耗指数 / 月覆盖人次 / 7日搜索人次 / 字数 / 竞争度 / 行业`
   - 样本词包括：`1-3岁幼儿餐碗`、`hiv自测试纸`、`intoyou`、`iqooneo11`。

RSO 是目前已勘察板块里唯一把**商业投放价格信号**显式暴露出来的搜索词页。  
因此它比 WSO/KSO 更接近“自然搜索 + 商业投放混合判断”。

点击 `行业词` 后，实测会触发完整会员弹窗，包含：

1. `权限不足，升级会员版本可查看更多数据`

3. 当前账号是 `个人版会员`
4. 特权表中出现 `RSO监测达人 10`

所以 RSO 至少有以下事实成立：

1. 行业词不是“空数据”，是**权限锁定**。
2. 个人版不能把它当成普通 tab 使用。
3. 后续抓取必须把该状态记录为 `locked`。

### 7.6 标签特征

RSO 搜索词页常见标签包括：

- `蓝海词`

- `高点击词`

没有在当前样本中像 KSO 那样大规模出现 `低成本词`。

### 7.7 RSO 的相关词页与下拉词页

1. **相关词页**
   - 表头：`月均搜索量 / 7日搜索人次 / 字数 / 竞争程度 / 推荐理由 / 市场出价`
   - 与搜索词页相比，**去掉了** `下拉词数量`
   - 样本里高频出现：`同行买词 / 黑马词 / 高点击词`
2. **下拉词页**
   - 实测表头与相关词页一致，也是：
     `月均搜索量 / 7日搜索人次 / 字数 / 竞争程度 / 推荐理由 / 市场出价`
   - 说明在 RSO 中，`相关词` 与 `下拉词` 更像同一套列表模型，只是数据集不同
3. **额外现象**
   - `下拉词` 页结果量显著更大，当前 `奶茶` 样本页提示 `7000+个搜索结果`
   - 左侧词树能直接把 `推荐 / 联名 / 发色 / 免单 / 文案` 这类内容消费向分词暴露出来

### 7.8 RSO 详情页层

`/RsoKeyWordDetail/detail?keyword=奶茶` 继续实测后，可确认：

1. 顶部卡片字段：`月均搜索量 / 7日平均搜索 / 下拉词数量 / 推荐理由 / 竞争程度 / 市场出价`

3. 下拉词区字段：`关键词 / 月覆盖人次`

   - `关键词 / 月覆盖人次 / 7日搜索人次 / 推荐理由 / 竞争程度 / 市场出价 / 操作`

这里说明 RSO 不只是搜索页独有 `市场出价`，详情页层也保持了这套商业化字段口径。

## 九、TSO 板块详解

### 9.1 真实入口与和其他板块的根本差异

- URL：`https://dso.aidso.com/KeywordTSO/searchWord?keyword=奶茶`
- 页头不是六频道条，而是直接显示：`30亿/日 / TSO`
- 搜索区比其他板块多一个前置输入：`请选择国家`

这意味着 TSO 不是简单复制 DSO/RSO 的平台，而是带**国家维度**的另一套产品。

### 9.2 搜索页 tab 与字段

实测具备：

1. `搜索词`
2. `相关词`
3. `下拉词`
4. `创意灵感话题榜`

搜索页字段：

1. `搜索指数`
2. `字数`
3. `下拉词数量`
4. `下拉词月搜索`

继续补到国家维度后，还能确认：

1. **相关词页**
   - 表头：`搜索指数 / 字数`
2. **下拉词页**
   - 表头同样是：`搜索指数 / 字数`
3. **国家切换后结果规模会变化**
   - `makeup` 在未选国家时是 `5000+`
   - 切到 `巴西` 后降为 `500+`
   - `下拉词` 页结果则进一步缩到 `59个`

### 9.3 当前无国家条件下的典型空态

搜索 `奶茶` 且未选择国家时，搜索词/相关词/下拉词页都能出现：

> `无官方数据，请搜索其他词`

这说明 TSO 的“空”不是单纯关键词冷门，而可能是：

1. 国家未选择
2. 当前国家下无官方收录
3. 该词在 TSO 数据源中不可用

但这不是 TSO 的全貌。继续实测后，已经确认：

1. 国家下拉真实可展开，当前列表至少包含：`美国 / 巴西 / 沙特阿拉伯 / 阿拉伯联合酋长国 / 墨西哥`
2. 选择 `巴西` 后，同一关键词 `makeup` 的结果数会从 `5000+` 变成 `500+`
3. 这说明国家不是视觉装饰，而是**真实改变结果集规模与样本词分布的主维度**

### 9.4 创意灵感话题榜

点击 `创意灵感话题榜` 后，进入独立榜单页：

- 路由：`/KeywordTSOList/creativeIdeaTopics`
- 结构：
  1. `创意灵感话题榜 / 电商词榜 / TSO-关键词库`
  2. 分类维度：`健康 / 运动户外 / 餐饮美食 / 生活服务 / 游戏 / 商务服务 / 新闻资讯 / ...`
  3. 时间维度：`近7天 / 近30天`
  4. 列：`话题信息 / 作品数 / 热度趋势`

当前样本：

- `#2016throwback`
- `#2025recap`
- `#2026`
- `#35mm`
- `#adventcalendar`

这表明 TSO 更像“跨境/海外趋势 + 话题灵感”产品，而不是单纯的站内关键词库。

继续深挖后还能确认两点：

1. `创意灵感话题榜` 右上导航不是装饰，实测还有 `电商词榜 / TSO-关键词库`
2. 榜单字段比之前记录得更细，当前应写成：
   - `排序 / 话题信息 / 作品数 / 热度趋势 / 操作`
   - 部分词条带 `新上榜` 标记
   - 同一 hashtag 可伴随分类一起展示，如 `#animal / 宠物生活`

### 9.5 批量模式

TSO 也支持 `单个 / 批量` 切换。切到批量后，同样出现“个人版最多输入 5 个词 / 文件上传”一类批量入口，说明它沿用了全站统一的批量框架。

### 9.6 TSO 详情页层

本轮继续确认到 TSO 确实存在独立详情页：

- 路由：`/TsoKeyWordDetail/detail?keyword=makeup`

2. 图表区多了 `国家` 选择，而不是地区省份这类国内维度
3. 下拉词区字段：`关键词 / 搜索指数`
4. 二级页只有：
   - `详情`
   - `相关词列表`
5. `相关词列表` 页为 `相关词TOP100`，字段是：
   - `关键词 / 搜索指数 / 字数 / 操作`

这说明 TSO 当前更像“国家维度 + 海外关键词 / 相关词”的轻量详情模型，没有像 DSO 那样扩展到电商和作品层。

## 十二、跨板块字段差异矩阵

这部分是后续抓取改造的核心。

| 逻辑含义 | 全站找词 | DSO | RSO | KSO | WSO | BSO |

## 十四、单位解析规则与字段规范化建议

跨板块抓取如果不先定义解析规则，后面所有横向对比都会失真。

### 14.1 常见数值格式

| 原始格式 | 含义 | 建议规范化结果 |
|---|---|---|
| `10.27w` | 10.27 万 | `102700` |
| `5373.55w` | 5373.55 万 | `53735500` |
| `2.57亿` | 2.57 亿 | `257000000` |
| `100-250` | 区间值 | 保留原字符串，同时可拆 `min=100,max=250` |
| `平均:7.54w` | 带前缀说明的均值 | 原串保留，另提取数值 `75400` |
| `1.29` | 价格/出价 | Decimal，不要当流量值 |
| `67.34%` | 百分比 | 规范化为 `0.6734` 或保留 `%` 原串并附 float |

### 14.2 不能直接转数字的字段

1. `竞争程度 / 竞争度`：有些板块显示 `高/低`，有些更偏视觉渲染。
2. `类型 / 推荐理由`：多标签并列，不是单值。
3. `订单量`：是区间档位，不是精确数值。
4. `点击收录`：是状态，不是数值。

### 14.3 建议的统一存储方式

每个指标至少保留三列：

1. `raw_text`
2. `normalized_number`
3. `unit`

这样既不丢页面事实，也能做后续排序和统计。

## 十六、空值与异常状态汇总

这部分必须单列，因为“空”在这个站不是一种状态，而是多种业务语义。

| 状态类型 | 页面表现 | 业务含义 | 示例 |
|---|---|---|---|
| 数字 0 | `0` | 该指标有字段，但当前值为 0 | DSO 冷门词的若干下拉词数量 |

## 二十三、"7 日搜索人次"折线图深层数据获取说明（2026-06-10 增量）

本章专门补讲搜索词表格里"7 日搜索人次"列的真实数据形态。它**不是一个纯文本单元格，而是一个内嵌的迷你 ECharts 折线图**。这是 010 文档其它章节没有覆盖到的"隐藏字段"。

### 23.1 字段真实结构

表格里"7 日搜索人次"列的单元格 HTML 实际是：

```html
<td class="el-table_1_column_3 is-center el-table__cell">
  <div class="cell">
    <div class="searchLeft" style="height: 70px;">
      <div class="searchLeft_data">
        <div class="searchLeft_top">平均:6635</div>
        <div class="lineEcharts" style="height: 70px;">
          <div class="chart_base" _echarts_instance_="ec_1781081242795">
            <div style="position: relative; overflow: hidden; width: 102px; height: 70px;">
              <svg width="102" height="70">
                <path d="M10 44.595 C... " stroke="#6D14FF"/>
                <!-- 7 个圆点 path，每个数据点一个 -->
              </svg>
            </div>
          </div>
        </div>
      </div>
    </div>
  </div>
</td>
```

每个数据点（每天 1 个，共 7 个）由一个独立的 `<path transform="matrix(2,0,0,2,X,Y)">` 绘制的小圆点表示。

### 23.2 可以拿到的数据

| 数据 | 可拿到 | 方式 |
|---|---|---|
| 平均值（如"平均:6635"） | ✓ | 读 `.searchLeft_top` 文本 |
| 7 个数据点的视觉坐标（x,y 像素） | ✓ | 读 SVG 里 7 个圆点 path 的 transform matrix |
| 折线形状（上升/下降/波动） | ✓ | 读 SVG path d 属性 + 圆点 y 坐标 |
| 每一天的真实搜索人数（数值） | ✗ | 见下 |

### 23.3 拿不到真实数值的原因

ECharts 实例在 AIDSO 的闭包内：

1. `window.echarts` 不存在 — 验证：`typeof window.echarts === 'undefined'`
2. `echarts.getInstanceById(ec_xxx)` 返回 null
3. Vue 组件实例的 `__vue__` 链上没有暴露 option 数据
4. 整个 HTML 449k 字符里没有 `__INITIAL_STATE__`、`preloadedState`、gridData、tableData 等内嵌数据
5. hook `fetch` 和 `XMLHttpRequest` 后 0 个匹配 `trend|7day|line|chart` 的请求 — 说明 7 日数据是被 SSR 注入到 hydration 数据里的，但是被组件闭包隔离

### 23.4 视觉坐标 → 真实数值的反推尝试

折线图容器 70px 高，102px 宽，每个圆点 y 坐标范围在 [37, 56] 之间（基于试管婴儿、三代试管、私人银行 3 个实测样本）。但**没有 yAxis 的真实 min/max 暴露**，因此无法从 y 像素反推真实搜索人数。

折线图横坐标从 10 像素到 92 像素，分 7 个等距点，对应 7 天。

唯一可确认的硬约束是：**7 个圆点 y 像素之和对应"平均:XXXX"×7 的真实数值之和**。

### 23.5 异步加载特性

1. **行级懒加载**：刚打开搜索结果页时，前 3-4 行已经渲染（含折线图），其它行需要等待；`get_text` 抓取到的所有 20 行是**网络层数据**，但 DOM 实际只渲染了 3-4 行
2. **折线图渲染**：每个 `.chart_base` 都有 `_echarts_instance_` ID，证明是按 cell 实例化 ECharts，渲染速度相对独立
3. **超时**：少数词的折线图加载失败时，`.searchLeft_top` 显示空值或 " - "

### 23.6 实战数据采集策略

因为拿不到真实 7 天数值，最佳折中方案是：

1. **优先采集"平均值"**：所有可渲染的折线图都有 `平均:XXX` 文本，是稳定可读数据
2. **辅助采集"趋势指纹"**：从 SVG 圆点 transform 提取 7 个 y 像素，计算 std/mean 比值，给出"平稳/波动/上升/下降"四档趋势标签
3. **过滤条件**：跳过 `.searchLeft_top` 为空或 ` - ` 的行
4. **批量采集时机**：页面 navigate 后等待 5-8 秒再 evaluate，避免行级懒加载未完成

### 23.7 反常识结论

**"7 日搜索人次"列在 AIDSO 客户端的折线图本质上是个"展示型字段"，不是"可数据化字段"。**

它对人类用户的价值是"一眼看出趋势"，对程序化采集的价值只剩"平均值 + 趋势指纹"。这意味着：

1. AIDSO 卖的不是"7 日搜索人次"这个字段，而是把 7 日数据折算成了**月覆盖人次**这个汇总字段
2. 想拿 7 日真实序列，唯一的路径是 AIDSO 后台 API（需付费会员权限）或屏幕 OCR + 视觉识别
3. 后续做关键词监控项目时，"7 日搜索人次"应当被建模成 `{avg, trend_shape, trend_std}` 三元组，而不是 7 个数字数组

### 23.8 与抖音实际"看后搜"行为的契合

折线图本质适合回答一个问题："这个词最近 7 天是上升还是下降？"在抖音的"看后搜"生态里，**7 天趋势曲线是判断"这个词是不是被某条爆款视频短期炒起来"的唯一可靠信号**。一个稳定的真实需求词，7 天曲线应该是平缓的；一个事件驱动型词（俞敏洪南极、全红婵劳力士、汪小菲接女儿），7 天曲线应该有一个明显尖峰。

但因为 AIDSO 不暴露 7 天数值，我们在 011 报告里只能用"平均值"做对比。在监控系统 V2 演进时，**需要专门为"7 日趋势指纹"设计采集通道**——这条数据比"月覆盖人次"更能区分"真实需求 vs 事件泡沫"。

## 二十四、对第二十三章的修正：真实 hover 可拿到 7 日精确数值（2026-06-10 当日补充）

第二十三章的"23.3 拿不到真实数值的原因"和"23.7 反常识结论"在 2026-06-10 当天被推翻。下面记录这一关键修正。

### 24.1 推翻点

用户提醒："鼠标悬浮上去是有具体数值的，不需要复杂计算。"验证后发现：**只要用真实鼠标 hover 在 ECharts 折线图的 7 个数据点 marker（圆圈）上，tooltip 就会显示 `YYYY-MM-DD value` 格式的真实文本**，可以被读取。

### 24.2 实测样本：香港保险关键词的 7 日精确序列

通过 7 次真实 hover（每次 hover 一个数据点 marker），从 AIDSO 抖音搜索词页面拿到了香港保险关键词的完整 7 天数据：

| 日期 | 真实搜索人次 |
|---|---|
| 2026-05-14 | 549 |
| 2026-05-15 | 487 |
| 2026-05-16 | 347 |
| 2026-05-17 | 435 |
| 2026-05-18 | 454 |
| 2026-05-19 | 606 |
| 2026-05-20 | 463 |

算术平均 = (549+487+347+435+454+606+463) / 7 = 3341 / 7 = **477.28**

AIDSO 列表里展示的"平均:477"是四舍五入后的真实算术平均，不是估值。

### 24.3 为什么之前会判断"拿不到"

| 失败尝试 | 现象 | 根因 |
|---|---|---|
| `window.echarts` / `window.echarts.getInstanceByDom` | undefined | ECharts 全局对象未暴露 |
| `div[_echarts_instance_].getOption()` | undefined | ECharts 实例存在 WeakMap 里，没挂在 div 属性上 |
| `dispatchEvent(MouseEvent('mousemove'))` 触发 hover | tooltip 不更新 | zrender 用 PointerEvent 监听，标准 MouseEvent 没走完它的事件链 |
| `dispatchEvent(PointerEvent('pointermove'))` | tooltip 不更新 | 同上，必须经过真实浏览器事件循环 |

结论：ECharts 实例本身的 option 数据确实无法直接拿到，**但真实事件可以触发它的 tooltip**，tooltip 文本又是普通 DOM 可读的。所以"7 日精确序列"是可拿到的，只是必须走"真实 hover"路径。

### 24.4 实操采集方法（4 步走通）

任何 Agent 在 user-browser MCP（或类似带真实 hover 的浏览器工具）下，按以下 4 步可以批量拿到任一关键词的 7 日精确数据：

**第 1 步：导航到目标页面**
```text
navigate https://dso.aidso.com/KeywordDouyin/searchWord?keyword=XXX
```
等待 5-8 秒（让行级懒加载完成折线图渲染）。

**第 2 步：给每个图表的 7 个 marker 加唯一 className**
```js
// 在 user-browser 的 evaluate 里执行：
(() => {
  const divs = document.querySelectorAll('div[_echarts_instance_]');
  divs.forEach((div, chartIdx) => {
    const paths = div.querySelectorAll('path');
    let n = 0;
    paths.forEach((p) => {
      const d = p.getAttribute('d');
      // 数据点 marker 的 d 属性是 "M1 0A1 1 0 1 1 1 -0.0001" 模式
      if (d && d.startsWith('M1 0A1 1')) {
        p.setAttribute('data-chart-idx', chartIdx);
        p.setAttribute('data-day-idx', n);
        p.setAttribute('class', 'dp-marker c' + chartIdx + '-d' + n);
        n++;
      }
    });
  });
  return { totalMarkers: document.querySelectorAll('.dp-marker').length };
})()
```

**第 3 步：用真实 hover 工具依次 hover 每个 marker**
```text
hover  .c0-d0
hover  .c0-d1
hover  .c0-d2
hover  .c0-d3
hover  .c0-d4
hover  .c0-d5
hover  .c0-d6
```
每次 hover 后，evaluate 读 tooltip DOM：
```js
(() => {
  const tts = [];
  document.querySelectorAll('div').forEach(d => {
    const s = d.getAttribute('style') || '';
    if (s.includes('display: block') && /2026-/.test(d.textContent)) {
      tts.push(d.textContent.replace(/\s+/g, ' ').trim());
    }
  });
  return tts.slice(0, 3);
})()
```
返回格式：`["2026-05-14 549"]`、`["2026-05-15 487"]` 等。

**第 4 步：解析并入库**
每行格式：`日期\t数值`，正则 `(\d{4}-\d{2}-\d{2})\s+(\d+)` 解析。

### 24.5 注意事项

1. **等待时间**：用户提到"通常需要等待一会"——经验值：每次 hover 后等 500ms-1s 让 tooltip 渲染稳定。
2. **行级懒加载**：AIDSO 不是 20 行一次性渲染，前 3-4 行先行渲染，剩下的要等。如果只看到 3 个 ECharts 实例，是正常的，要等几秒。
3. **可见性过滤**：tooltip DOM 用 `display: block` 标识可见状态，hover 后才会从 `display: none` 切换。
4. **滚动加载**：要拿 20 行全部数据，需要先滚动到底让所有行渲染完成，再开始 hover 流程。
5. **图表与行对应关系**：每个 `div[_echarts_instance_]` 对应一行（一个关键词），与可见行的 td 顺序一致。
6. **频率控制**：高频 hover 可能触发 AIDSO 反爬，建议每 7 次 hover 后等待 2-3 秒再继续下一行。

### 25.7 实勘修正摘要（与原方案差异）

| 原方案字段 | 修正后 | 原因 |
|---|---|---|
| `top_city: "深圳"` | `city_level: "三线城市"` | API 实际返回城市等级，不是具体城市名 |
| `week_avg_search` 直接取 | 需从 trend API 最近 7 天 pv 算均值 | detail API 不返回 |
| `down_keyword_month_covercount` 带"亿"/"w"后缀 | 纯数字 | API 返回原始数字，前端才格式化 |
| 无 `age_profile` | 新增 | DSO 独有，年龄段画像 |
| 无 `gender_profile` | 新增 | DSO 独有，性别画像 |
| 无 `relation_word_count` | 新增 | DSO 独有，相关词数量 |
| 无 `is_eco` | 新增 | DSO 独有，是否有电商数据 |
| `tags` 直接中文 | 双存储：`type_enums` + `tags` | API 返回枚举名，同时存中文标签便于展示 |
| 无 `type_enum_map` | 新增在顶层 | 枚举→中文映射，避免每条数据重复 |
