# 2026-07-13 监控系统性能优化与测试交接

## 1. 本次目标与不变边界

本次只优化微信 Top3 监控系统的读取、传输和前端渲染链路，不改变抓取事实、不改变评分算法、不改变现有 Markdown 文章与搜索结果的存储格式。`normalized/monitor-data.json` 仍是完整事实快照；SQLite、内存缓存、分片接口只承担索引与加速职责。

优化顺序严格按以下低风险路径执行：

1. mtime/size 内存缓存与 JSON 预序列化；
2. gzip、ETag、304；
3. 账号评分 Tooltip 懒加载；
4. 关键词/账号详情分片、账号列表分页；
5. 完整自动化测试、开发端口浏览器验收、生产重启。

## 2. 为什么旧版慢

旧版首页每次打开都走 `/api/monitor-data`：

- 后端每次读取并解析约 164MB 的 `normalized/monitor-data.json`；
- 再合并 SQLite 关键词注册表、人工状态、阅读增量；
- Flask `jsonify` 每次重新序列化，实际 HTTP 响应约 219MB；
- 无 gzip，`Cache-Control: no-store`，刷新无法使用 ETag/304；
- 前端为了首页列表同时下载全部关键词 runs、全部账号 topics/keywords、三套六边形评分依据；
- 1279 个账号评分 Tooltip 全部预生成 HTML；
- 账号视角一次性创建 1279 行 DOM；
- Tooltip 使用全局 `mousemove + elementFromPoint` 热循环；
- 单个关键词的换新热力图也会重新下载整份 219MB 数据。

旧生产基线：

| 项目 | 旧值 |
|---|---:|
| `monitor-data.json` | 171,709,511 bytes |
| 旧 `/api/monitor-data` HTTP 响应 | 219,267,646 bytes |
| 旧接口 TTFB | 2.72–3.22 秒 |
| gzip | 无 |
| 缓存 | `no-store` |
| 账号首批 DOM | 1279 行 |

核心瓶颈不是磁盘本身：读取文件约 0.014 秒，主要时间消耗在重复 JSON 解析、数据合并、序列化、大对象传输和浏览器建 DOM。

## 3. 后端改动

### 3.1 新增 `app/services/monitor_fast_service.py`

`MonitorFastStore` 是新的进程内只读加速层：

- 首次启动解析、合并一次完整监控数据；
- 后续请求先检查源签名，未变化直接复用内存对象和序列化字节；
- 源签名包含：
  - `normalized/monitor-data.json`
  - `data/state/app.db`
  - `data/state/app.db-wal`
  - `normalized/keyword_read_deltas.json`
  - `normalized/article_metric_observations_meta.json`
- 任一文件 mtime/size 改变后，下一次请求自动原子重载；
- 缓存完整 JSON bytes、轻量 bootstrap bytes、bootstrap gzip、弱 ETag；
- 完整 gzip 第一次需要时生成，之后复用；
- 建立 keyword/account ID 索引；
- 关键词详情、账号详情使用 200 项 LRU，缓存 raw/gzip/ETag；
- 账号详情额外预计算 `_today_article_ids` / `_today_article_titles`，避免前端为了标记今日文章扫描所有关键词快照。

### 3.2 新接口

| 接口 | 用途 |
|---|---|
| `GET /api/monitor-data` | 保留旧完整数据语义，供兼容调用 |
| `GET /api/monitor-data/bootstrap` | 首页关键词摘要 + 账号摘要 |
| `GET /api/monitor-data/keyword/<keyword_id>` | 单个关键词完整 runs/accounts/文章详情 |
| `GET /api/monitor-data/account/<account_id>` | 单个账号完整 topics/keywords/评分依据 |

四个接口均支持：

- `Accept-Encoding: gzip`
- `ETag`
- `If-None-Match → 304`
- `Vary: Accept-Encoding`
- `Cache-Control: no-cache, must-revalidate`

这里的 `no-cache` 不是“不缓存”，而是允许浏览器保存响应、再次使用前先用 ETag 验证；未变化时只返回 304。

### 3.3 bootstrap 裁剪原则

关键词摘要保留：

- 列表/排序/筛选指标；
- history_best/history_hits；
- 精简阅读增量；
- `turnover_runs`：保留快照时间和文章 ID，足够首页计算换新率；
- 不包含完整 runs、文章正文元数据和账号透视明细。

账号摘要保留：

- 三套分数及昨日值、变化值；
- 历史趋势、连击、覆盖数量、文章数量；
- `topic_names` / `keyword_names`；
- 不包含三套 hexagon/parts、完整 topics、完整 keywords 和文章集合。

### 3.4 其他后端接入

- `app/__init__.py`：Blueprint 注册后预热 FastStore；监控接口不再被全局 `no-store` 覆盖；指纹静态文件允许正常验证缓存。
- `app/config.py`：新增阅读增量和观测元数据路径配置。
- `app/services/monitor_service.py`：已有内部调用优先复用 FastStore 完整对象。
- `app/services/keyword_manage_service.py`：关键词管理统计复用内存关键词索引，不再为约 385KB 的管理响应重新解析 171MB 文件。
- `run.py`：显式启用 Flask threaded 模式，详情、图片等并发请求不会串行阻塞；默认关闭 Flask debug，只有显式 `FLASK_DEBUG=1` 才开启。

## 4. 前端改动

### 4.1 首页改为 bootstrap

`app/static/js/monitor.js`：

- 首页只请求 `/api/monitor-data/bootstrap`；
- 关键词管理数据与 bootstrap 并行请求；
- `loadGroups()` 复用已经取得的 `KM_DATA`，删除首屏重复 `/api/keyword-manage` 请求；
- 建立四个 Map：
  - `KEYWORD_BY_ID`
  - `KEYWORD_BY_NAME`
  - `ACCOUNT_BY_ID`
  - `ACCOUNT_BY_NAME`
- 高频定位由数组 `.find()` 改为 Map；
- `window.__WX_PERF__` 记录 bootstrap 和详情请求耗时，方便后续排查。

### 4.2 关键词/账号详情懒加载

- 点击关键词时才请求该关键词详情；
- 点击账号时才请求该账号详情；
- 同一详情的并发请求使用 Pending Map 合并；
- 返回后合并到原摘要对象；
- 使用 mode + 当前选中项 stale guard，快速切换时旧请求不会覆盖新详情；
- 加载失败只影响当前详情区域，不破坏左侧列表。

### 4.3 Tooltip 懒加载

旧版每个账号行都把完整评分看板 HTML 写进 `data-score-tooltip-html`，1279 个账号会产生大量重复字符串和 DOM 属性。

新版只保留：

```html
data-account-id="..."
data-score-tooltip-mode="score|timeliness|today"
```

首次 hover/focus：

1. 立即显示“正在加载完整评分依据”；
2. 请求账号详情；
3. 详情到达后生成评分看板；
4. 第二次查看直接命中内存。

同时删除全局 `mousemove` 和 `elementFromPoint` 热循环，改为 `mouseover/mouseout/focusin/focusout` 事件委托。

### 4.4 账号分页

- 初始只渲染 100 个账号；
- “再加载”每次增加 100；
- 显示“已显示 N / 筛选结果总数”；
- 搜索、切换账号分/时效分/当天分、切换视角、重新加载数据时重置为 100；
- 分页状态使用 `mode + sort + filter` key，避免每次普通重渲染都错误重置。

### 4.5 换新热力图

`app/static/js/keyword-turnover.js` 不再请求完整 `/api/monitor-data`：

- URL 有 `keyword_id`：直接请求关键词详情；
- 只有关键词文本：先请求 bootstrap 找 ID，再请求关键词详情；
- 支持 ETag/304；
- 页面功能与原有 2D/3D 换新分析保持一致。

### 4.6 CSS

- 账号列表和行增加 layout/style containment；
- 新增加载更多按钮；
- 新增详情/Tooltip 加载与错误状态；
- 未改变原有视觉体系和响应式结构。

## 5. 实测结果

真实数据：298 个关键词、1279 个账号。

| 项目 | 优化后 |
|---|---:|
| 完整 compact JSON | 121,993,493 bytes |
| 完整 gzip | 8,043,427 bytes |
| bootstrap raw | 6,156,977 bytes |
| bootstrap gzip | 486,030 bytes |
| bootstrap 热 TTFB | 0.7–1.7 ms |
| bootstrap gzip 总传输时间（本机） | 2.7–3.9 ms |
| `/api/keyword-manage` TTFB | 6–10 ms |
| 单关键词详情 TTFB（样本） | 9.7 ms |
| 单账号详情 TTFB（样本） | 2.7 ms |
| 完整接口热 TTFB | 0.7–1.8 ms |
| 浏览器完整刷新到 298 词与详情可用 | 约 1.15 秒 |
| 账号初始 DOM | 100 行 |

相对旧版首屏 HTTP 体积：

- 219.3MB → 0.486MB gzip；
- 首屏传输约减少 99.78%；
- 后端热 TTFB 从约 3 秒降到约 1 毫秒量级；
- 浏览器不再一次构建 1279 个账号行和 1279 份评分看板 HTML。

## 7. 后续维护注意事项

1. **不要把 Markdown 文章存储迁入数据库。** Markdown 仍是正文与抓取证据，接口只返回路径和展示元数据。
2. 修改 monitor-data 字段时，先判断它属于：
   - 首页列表必需：加入 bootstrap 白名单；
   - 仅详情必需：保留在 detail，禁止塞回 bootstrap；
   - 换新计算必需：优先加入精简 `turnover_runs`，不要恢复完整 runs。
3. 新增会影响页面显示的数据源文件时，必须把该文件加入 `_source_signature()`，否则内存缓存不会自动失效。
4. 不要恢复 `Cache-Control: no-store`；监控接口应保持 ETag 验证模式。
5. 不要在账号行预生成完整 Tooltip HTML，也不要恢复全局 mousemove 热循环。
6. 账号列表如果增加新筛选维度，要把该维度加入 `accountPageStateKey`。
7. 详情异步请求必须保留 stale guard，防止用户快速切换时显示错对象。
8. 生产由 launchd `com.local.wechat-monitor-8765` 管理；开发测试只能使用 8767/8768 并设置 `ZK_MONITOR_DISABLE_SCHEDULER=1`。

## 8. 生产重启与回滚原则

生产重启：

```bash
launchctl kickstart -k gui/501/com.local.wechat-monitor-8765
```

重启前先确认 `/api/refresh-all/status` 没有活跃批次。重启后检查端口、bootstrap、关键词详情、账号详情和浏览器首页。

如需回滚，只回滚本文件列出的性能相关代码；不要 reset/checkout 整个仓库，因为项目中存在大量并行未提交开发改动。
