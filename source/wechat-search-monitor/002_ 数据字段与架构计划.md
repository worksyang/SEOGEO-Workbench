# 关键词监控系统：数据字段与架构计划

> 状态：工作草案，不是最终定稿  
> 目的：先把字段逻辑、时间语义、实体边界定清楚，避免后期前端、解析脚本、历史回看、账号透视全部返工。

## 1. 这份计划解决什么问题

当前页面方向已经基本成立：

- 主视角：关键词
- 副视角：账号
- 页面目标：先看某个关键词当下谁排前、哪些文章值得研究；再判断哪些账号值得长期盯

现在真正需要先定的，不是 UI，而是数据底座：

- 什么是原始事实
- 什么是派生结果
- 时间到底按“天”建，还是按“抓取时间点/快照”建
- 上游微信搜索模块至少要吐出哪些字段

如果这些前期定错，后面会出现这些问题：

- 同一天多次抓取无法回看
- 账号命中次数语义混乱
- 历史趋势和当天榜单互相打架
- 页面字段写死在前端，无法沉淀成正式数据

## 2. 第一原则

### 2.1 原始层只存“抓到了什么”

数据库不应该围绕页面字段来建，也不应该直接围绕“账号综合分”“15 天趋势”“todayPrev”这类结果字段来建。

数据库应该围绕这条事实来建：

`某次抓取 / 某个关键词 / 第 N 名 / 某篇文章 / 某个账号`

### 2.2 派生层再算“这代表什么”

以下字段都不建议作为核心原始字段写死：

- `todayRank`
- `todayPrev`
- `history`
- `score`
- `hitDays`
- `hitSnapshots`
- `coverageDays`
- `top3Days`

这些都应该通过原始事实再计算出来。

### 2.3 时间最小锚点必须是“抓取时间点”

不能只按“天”建模，因为未来一定会出现：

- 定时抓取
- 手动复查
- 补抓

这里更准确的说法不是“批次”，而是：

**每次抓取完成后形成一个快照，这个快照有一个唯一时间点 `captured_at`。**

也就是说：

- 真正的时间最小锚点是 `captured_at`
- `day` 只是展示层的聚合维度
- `snapshot_id` 只是那次快照的身份标识，不是比时间点更底层的概念

如果后面代码里仍然保留 `run` 这个词，它应当只表示“这次快照的执行记录”，不应再被理解成比时间点更基础的单位。

一句话原则：

**数据库按 snapshot/captured_at 存，页面按 day 看，必要时允许钻到具体时间点。**

### 2.4 必须区分三种状态

这三个状态绝不能混为一个值：

1. 没跑成功
2. 跑成功但没上榜
3. 跑成功并且上榜第 N 名

## 3. 系统事实基准

现阶段建议采用以下事实源优先级：

### 3.1 一级事实源

微信搜索结果列表页 Markdown / API 返回结果。

这是最核心的数据源，因为它天然回答：

- 这个关键词
- 这次抓取
- 第几名
- 是哪篇文章
- 属于哪个账号

### 3.2 二级事实源

文章详情 Markdown / API 返回正文。

这个数据源的价值是补齐：

- 正文
- 原文链接
- 内容抽屉
- 后续 AI 分析

### 3.3 非核心事实源

`report` / `push` / `top10对比` 这类文件。

这些更适合视为派生内容，不建议作为主库事实源。

## 4. 核心实体设计

下面是建议的核心实体。这里先讲概念，不先卡死数据库类型。

### 4.1 `keyword`

关键词本体。

建议字段：

| 字段 | 含义 | 是否必需 |
|---|---|---|
| `keyword_id` | 内部主键 | 是 |
| `keyword_text` | 关键词原文 | 是 |
| `is_active` | 是否仍在监控 | 是 |
| `notes` | 备注 | 否 |
| `created_at` | 创建时间 | 是 |
| `updated_at` | 更新时间 | 是 |

### 4.2 `crawl_snapshot`

一次真实抓取完成后形成的快照。这个表是整个时间语义的核心。

建议字段：

| 字段 | 含义 | 是否必需 |
|---|---|---|
| `snapshot_id` | 快照主键 | 是 |
| `keyword_id` | 对应关键词 | 是 |
| `captured_at` | 实际抓取完成时间，精确到分钟或秒 | 是 |
| `snapshot_date` | 快照所属日期 | 是 |
| `timezone` | 时区 | 是 |
| `trigger_type` | `scheduled` / `manual` / `backfill` | 是 |
| `is_primary` | 该日是否作为主快照 | 否 |
| `status` | `success` / `failed` / `partial` | 是 |
| `result_limit` | 本次抓取目标榜单上限，如 10/11 | 否 |
| `result_count` | 实际抓到多少条 | 否 |
| `source_name` | 来源模块名，例如微信搜索 API | 否 |
| `source_version` | 上游模块版本或解析版本 | 否 |
| `error_message` | 失败原因 | 否 |
| `recorded_at` | 入库时间 | 是 |

备注：

- 这里的主语义是“快照”
- `captured_at` 比 `snapshot_id` 更关键，因为它是真正的时间锚点

### 4.3 `snapshot_term`

保存某次抓取带回来的搜索建议词和相关搜索词。

建议字段：

| 字段 | 含义 | 是否必需 |
|---|---|---|
| `term_id` | 主键 | 是 |
| `snapshot_id` | 对应快照 | 是 |
| `term_type` | `suggestion` / `related` | 是 |
| `position` | 在列表中的顺序 | 是 |
| `term_text` | 词本身 | 是 |

### 4.4 `account`

公众号本体。

建议字段：

| 字段 | 含义 | 是否必需 |
|---|---|---|
| `account_id` | 内部主键 | 是 |
| `canonical_name` | 归一化后的公众号名 | 是 |
| `first_seen_at` | 首次出现时间 | 否 |
| `last_seen_at` | 最近出现时间 | 否 |
| `is_focus` | 是否重点账号 | 否 |
| `notes` | 备注 | 否 |
| `wechat_biz` | 公众号唯一标识，当前可空，后续预留 | 否 |

### 4.5 `account_alias`

处理公众号改名或历史别名。

建议字段：

| 字段 | 含义 | 是否必需 |
|---|---|---|
| `alias_id` | 主键 | 是 |
| `account_id` | 指向 account | 是 |
| `alias_name` | 抓到的原始名称 | 是 |
| `first_seen_at` | 首次使用该别名 | 否 |
| `last_seen_at` | 最近使用该别名 | 否 |

### 4.6 `article`

文章元数据本体。

建议字段：

| 字段 | 含义 | 是否必需 |
|---|---|---|
| `article_id` | 主键 | 是 |
| `normalized_url` | 归一化 URL | 是 |
| `raw_url` | 原始 URL | 是 |
| `title` | 标题 | 是 |
| `account_id` | 归属账号 | 是 |
| `published_at` | 发布时间 | 否 |
| `summary` | 列表页摘要 | 否 |
| `first_seen_at` | 首次入榜时间 | 否 |
| `last_seen_at` | 最近入榜时间 | 否 |
| `content_status` | 正文是否已抓到 | 否 |

备注：

- 第一版文章主键建议用稳定键：`normalize(account_name) + "::" + normalize(title)`
- `normalized_url` 仍然要保留，但更适合做“URL 归一化字段”，不适合在当前样本里单独承担唯一身份
- 原因：老样本存在“先无 URL、后补到真实 URL”的情况；若直接按 URL 建主键，会把同一篇文章拆成两条

### 4.7 `article_content`

正文内容单独放，避免主表过重。

建议字段：

| 字段 | 含义 | 是否必需 |
|---|---|---|
| `article_id` | 对应文章 | 是 |
| `content_markdown` | 正文 Markdown | 是 |
| `source_file_path` | 原始文件路径 | 否 |
| `content_hash` | 正文去重或变更校验 | 否 |
| `fetched_at` | 正文抓取时间 | 否 |
| `parser_version` | 正文解析器版本 | 否 |

备注：

- 当前阶段**不建议把正文 Markdown 直接入 SQLite**
- 更轻量、也更符合现有工作流的做法是：正文继续落地为 Markdown 文件，数据库里只保存路径、状态、抓取时间和校验信息
- 如果后面确实需要全文检索，再讨论是否引入 FTS 或独立内容库

### 4.8 `ranking_hit`

这是最关键的事实表。

含义是：

**某次抓取中，第 N 名命中了哪篇文章，属于哪个账号。**

建议字段：

| 字段 | 含义 | 是否必需 |
|---|---|---|
| `hit_id` | 主键 | 是 |
| `snapshot_id` | 对应快照 | 是 |
| `rank` | 榜单名次 | 是 |
| `article_id` | 对应文章 | 是 |
| `account_id` | 对应账号 | 是 |
| `title_raw` | 抓取当时的原始标题 | 否 |
| `summary_raw` | 抓取当时的原始摘要 | 否 |
| `account_name_raw` | 抓取当时的原始账号名 | 否 |
| `published_at_raw` | 抓取当时原始发布时间字符串 | 否 |
| `url_raw` | 抓取当时原始 URL | 否 |
| `created_at` | 入库时间 | 是 |

备注：

- 原始字段保留是为了审计和回溯
- 规范化字段则走 `article` / `account`

## 5. 页面需要的字段如何从数据库推导

### 5.1 关键词页当前快照榜单

来源：

- 指定 `keyword_id`
- 指定 `snapshot_id`
- 查询该 `snapshot_id` 下所有 `ranking_hit`

### 5.2 关键词页 15 天趋势

来源：

- 指定 `keyword_id`
- 取过去 15 天的“日代表快照”或“最新成功快照”
- 再聚合 `ranking_hit`

### 5.3 账号透视

来源：

- 过去 N 天内，该账号在所有关键词的 `ranking_hit`

### 5.4 命中次数

建议数据库支持同时计算三种定义：

| 指标 | 含义 | 默认是否用于页面 |
|---|---|---|
| `hit_snapshots` | 上榜了多少个快照时间点 | 否 |
| `hit_days` | 上榜了多少天 | 是 |
| `hit_articles` | 上榜了多少篇不同文章 | 否 |

当前页面建议默认显示 `hit_days`。

原因：

- 手动复查多时，`hit_snapshots` 会虚高
- `hit_days` 更接近“稳定存在感”

补充说明：

- 这里的“命中次数”只是一种页面统计语义
- **系统本身没有“先定义一批观测账号，再去看它有没有上榜”的监控概念**
- 账号是随着关键词搜索结果自然进入系统的：上榜一次就会进入库，之后安静留存即可
- 后续如果要做“低价值账号清理”或“冷账号归档”，那是独立机制，不影响当前主设计

### 5.5 `todayPrev`

这个字段以后不应作为单一概念存在。

建议拆成：

- `prev_snapshot_rank`：上一快照时间点相比
- `prev_day_rank`：前一日主快照相比

## 6. 关键词视角的时间语义

当前页面已经验证了一个重要方向：

- 榜单应该跟随 `snapshot`
- 趋势应该跟随 `day/window`

因此关键词页建议长期保持两层时间：

### 6.1 快照层

看“某一次抓取”的榜单。

适合：

- 今天下午手动复查
- 看某次波动是否真实发生

### 6.2 窗口层

看最近 7 天 / 15 天 / 30 天的趋势与盯号判断。

适合：

- 长期研究
- 判断哪些账号值得持续监控

## 7. 上游微信搜索模块建议输出的最小字段

你现在的微信搜索模块已经成熟，而且是 API 级别，这很好。  
这意味着这个系统尽量不要重写搜索逻辑，而是围绕它做结构化落地。

如果上游模块已经能吐出以下字段，那最好直接复用。  
如果缺少其中的关键字段，建议为了这个项目补一下。

### 7.1 `crawl_snapshot` 层最小字段

- `keyword`
- `captured_at`
- `trigger_type`
- `status`
- `result_count`
- `result_limit`

补充说明：

- 榜单上限不应在数据库层写死为 10 或 11
- 推荐原则：**上游返回多少，就存多少**
- 因此 `result_limit` 和 `result_count` 都应保留，便于后续从 Top10 扩展到 Top20

### 7.2 `snapshot_term` 层最小字段

- `search_suggestions[]`
- `related_searches[]`

### 7.3 `ranking_hit` 层最小字段

- `rank`
- `title`
- `summary`
- `account_name`
- `publish_time_raw`
- `url`

### 7.4 `article_content` 层最小字段

- `url`
- `content_markdown` 或 `content_file_path`

当前项目更推荐：

- `content_file_path`
- `content_hash`
- `fetched_at`

### 7.5 如果上游模块还能补这些，会更稳

- `snapshot_id` 或可稳定构造的快照唯一标识
- `fetch_started_at`
- `fetch_finished_at`
- `query_text`
- `parser_version`
- `raw_response_hash`

当前优先级补充建议：

- `snapshot_id` 可以晚一点做，只要 `keyword + captured_at` 在现阶段能稳定唯一也可以先跑起来
- `wechat_biz` 不是第一版硬依赖，但建议上游模块后续预留能力，因为它是解决账号改名问题的最佳长期方案

## 8. 当前阶段不建议提前做的事

以下内容先不要进第一版正式数据库：

- AI 自动总结字段
- AI 选题聚类结果
- 阅读量、点赞数、在看数
- 自动情绪判断
- 人工评分字段

这些都可以后加，但不应该反过来决定主库结构。

## 9. 推荐开发顺序

### 第一步

先把正式数据模型定下来。

目标：

- 明确表和字段
- 明确原始字段与派生字段的边界

### 第二步

写 Parser，把现有 Markdown / 上游 API 结果统一落成结构化 JSON。

### 第三步

先写入标准 JSON 中间层。

说明：

- 当前阶段优先走轻量链路：`上游 API / Markdown -> 标准 JSON`
- 这个中间层是为了调试、抽查、重建、回放
- 后续结构稳定后，再把同样字段批量写入 SQLite

### 第四步

让前端页面从结构化 JSON 读取，而不是继续吃手写假数据。

### 第五步

结构稳定后再接 SQLite、自动化调度、补抓与历史事件流。

## 10. 当前我建议先拍板的决定

如果只先定最关键的几条，我建议先确认这几个：

1. 原始事实单位是 `snapshot/captured_at + ranking_hit`
2. 页面默认展示的命中次数用 `hit_days`
3. 第一版文章唯一身份优先用“账号 + 标题”的稳定键，`normalized_url` 作为补充字段保留
4. 关键词页榜单按 `snapshot` 看，趋势按 `15天窗口` 看
5. 同一天允许多次抓取，且全部保留
6. `report/push/对比稿` 不进主事实库
7. 榜单上限遵循“上游返回多少就存多少”
8. 正文默认抓榜单前列文章，但正文本体继续以 Markdown 文件保存，不直接入 SQLite

## 11. 这份计划的定位

这不是不可更改的“架构圣经”，而是一份当前阶段尽量稳、尽量少返工的底稿。

后面可以改：

- 表名
- 字段名
- 是否分表
- 是否直接上数据库还是先落 JSON

但尽量不要轻易改的，是这几个底层原则：

- 原始层和派生层解耦
- 时间最小锚点是 `captured_at`
- 主事实是 ranking_hit
- 关键词视角和账号视角共用同一份底层事实
