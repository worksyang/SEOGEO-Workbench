# 内容监控、母文章、写作与发布系统总览

> 本文只记录当前代码中已经存在的产品、功能和边界，供 Agent 快速建立宏观认识。各系统目前分别开发、分别存储，尚未组成统一平台；未来可以通过标准化的“选题/关键词、Markdown、母文章、成稿批次”接口建立关联。

## 1. 全局链路

当前体系可以按职责分为四层：**发现选题 → 沉淀内容资产 → 承接写作 → 批量发布**。

| 层级 | 系统 | 当前作用 | 当前关系 |
|---|---|---|---|
| 发现选题 | 微信搜一搜关键词监控、小红书关键词监控、GEO 公域引用观察台 | 从搜索结果、排名、互动或 AI 引用中发现关键词、问题、文章和账号 | 三套系统独立 |
| 内容采集 | 公众号监控系统 | 按指定公众号持续抓文章并转为 Markdown | 不依赖关键词搜索结果 |
| 内容资产 | Wiki Viewer | 阅读、搜索、编辑和整理 Markdown，维护母文章及图片 | 读取本地 Markdown 知识库 |
| 写作生产 | WritingMoney | 铸造母文章，或让关键词匹配已有母文章后进入批量成稿 | 当前为静态 Demo |
| 发布执行 | 老写作与发布系统 | AI 改写、敏感词处理、Markdown 合成、多账号微信公众号发布 | 独立 CLI/自动化系统 |

## 2. 微信搜一搜关键词监控系统

**位置：** `/Users/works14/.claude/监控/wechat-ybxhyyh-top3`

这是以微信搜一搜结果页为事实源的关键词观察系统。基本事实是“某次抓取、某个关键词、第 N 名、某篇文章、某个公众号”，搜索结果与文章正文最终都可关联到本地 Markdown。它不是指定公众号全量监控：只有进入搜索结果的文章才会成为本系统事实。

| 关注面 | 当前功能 |
|---|---|
| 关键词管理 | 关键词启停、分类/赛道/大类、刷新频率、关键词发现与候选词管理 |
| 监控主页 | 汇总关键词、账号、最新快照、覆盖天数、文章数和趋势 |
| 排名快照 | 保存抓取时间、触发类型、搜索排名、相关文章、公众号、搜索建议词和相关搜索词 |
| 常态阅读 | 根据文章不同时间的阅读观测，估算稳定阅读水平和日常阅读增量 |
| 阅读增量 | 计算文章与关键词的阅读变化、趋势、动量及置信度 |
| 关键词分类 | 通过关键词注册表、赛道/大类和 bucket resolver 统一分类，不把分类写死在页面 |
| 账号分 | 15 天窗口观察历史覆盖、近期覆盖、稳定文章、持续上榜和主题/关键词广度 |
| 时效分 | 3 天窗口观察新文章冲榜、新进 Top3、连续冲榜和近期扩散 |
| 当天分 | 1 天窗口观察今日 Top3、今日关键词、今日文章、主题和排名质量 |
| 内容查看 | 通过 URL 或标题匹配正文 Markdown，支持文章详情、封面、阅读数据和来源审计 |

主要代码位于 `app/`；`app/ingest/` 负责解析、标准化和派生指标，`app/web/` 负责页面/API，`normalized/` 和监控 JSON 是标准化及前端聚合产物。系统按 `captured_at` 保存快照，页面再按天聚合。

## 3. 公众号监控与 Markdown 入库系统

**位置：** `/Users/works14/Documents/zkcode/250626_mpGUI`

这是指定公众号的独立监控系统，网页产品名为“公众号抓取控制台”。它连接内网 WeRSS，读取公众号列表、刷新指定公众号、拉取文章，再由本地工作流筛选、分类并转成 Markdown。它和微信搜一搜关键词监控没有数据耦合：即使文章从未进入搜索榜，只要属于被监控公众号，也可以被抓取。

| 关注面 | 当前功能 |
|---|---|
| 公众号管理 | 获取 WeRSS 公众号列表，设置是否选中、是否刷新、是否参与工作流及所属分类 |
| 微信登录 | 检查登录状态、获取二维码、完成扫码登录 |
| 抓取任务 | 创建、查看、取消后台任务，展示进度、日志和结果统计 |
| 文章范围 | 支持近 N 天工作流，也有指定公众号历史抓取脚本与 SOP |
| 去重 | 扫描已有 Markdown、已拒绝文章和标题，避免重复处理 |
| AI 分类 | 按提示词判断文章是否进入本地知识库，模型可配置并可探测 |
| Markdown 转换 | 不依赖 WeRSS 摘要正文，使用微信原文 URL 和 `SomeURL2MD/wechat_to_markdown.py` 转全文 |
| 图片与 OCR | `SomeURL2MD/` 提供图片下载、二维码检测、OCR 注释、图片处理等基础能力 |
| 本地状态 | 设置、账号标记、分类和任务状态保存在 `.web_console/app.db` |
| 默认入口 | 前端 `client/`，后端 `server/`，共享流程在 `workflow_service.py` 和 `article_workflow.py` |

默认网页地址为 `http://127.0.0.1:5173`，本地后端为 `127.0.0.1:28765`。`TEST/we-mp-rss` 是上游 WeRSS 的测试/参考副本，不是本系统默认 GUI。

## 4. 小红书关键词监控系统

这是微信关键词监控架构在小红书平台上的复刻版，数据事实来自 TikHub 小红书实时 API。主对象从“公众号文章”替换为“笔记与博主”，保留关键词、快照、排名、账号透视、三榜和六边形等产品结构。

| 关注面 | 当前功能 |
|---|---|
| 关键词管理 | 复用关键词 CRUD、启停、分类和监控页框架 |
| 搜索快照 | 按关键词获取小红书搜索笔记，保存排名、发布时间、作者和内容标识 |
| 笔记数据 | 标题、摘要、图文/视频类型、封面、原文链接及详情懒加载 |
| 互动数据 | 点赞、收藏、评论、分享及不同观测时间的互动增量 |
| 博主数据 | 昵称、头像、粉丝、关注、IP、认证、作品数和已发布笔记 |
| 账号分 | 15 天窗口：历史/近期覆盖、稳定笔记、持续经营、收藏矩阵和战场广度 |
| 时效分 | 3 天窗口：Top3、新笔记冲榜、新进 Top3、连续冲榜、收藏动能和关键词扩散 |
| 当天分 | 1 天窗口：今日 Top3、今日关键词/笔记/主题、排名质量和互动增长 |
| 热度指标 | 将收藏、点赞、评论、分享按权重折算为互动信号，并计算稳定热度、峰值和趋势 |
| 成本控制 | 搜索首抓与笔记/博主详情分离；详情在打开抽屉时按需拉取并缓存 |

入口是 `run.py`，核心代码在 `app/`，原始数据在 `data/raw/tikhub/xhs/`，标准化事实和页面聚合在 `normalized/`。项目上一级 `/Users/works14/Documents/zkcode/取数` 还包含多平台抓取实验和研究脚本，但小红书正式监控产品以 `xhs-keyword-monitor/` 为准。

## 5. Wiki Viewer：Markdown 与母文章编辑器

**位置：** `/Users/works14/Documents/output_md/wiki-viewer`

Wiki Viewer 是本地 Markdown 知识库的定制阅读与编辑界面，形态接近轻量 Obsidian。它主要服务母文章和知识资产整理，不负责关键词抓取，也不直接负责批量发布。

| 关注面 | 当前功能 |
|---|---|
| 文件导航 | 懒加载目录树、Markdown 文件列表和文件数量 |
| 搜索 | 全库文件名搜索，支持定位 Wiki 双链目标 |
| 阅读 | Markdown 渲染、Wiki Link、OCR 注释和插图建议块展示 |
| 编辑 | 全文编辑、保存、快捷键进入/退出编辑 |
| 图片查看 | 图片灯箱、原图查看和图片 URL 展示 |
| 单文删除图片 | 从当前 Markdown 删除指定图片引用 |
| 全库删除图片 | 查询相同图片被哪些文件引用，并批量删除全部引用 |
| 图片索引 | 维护图片引用次数、引用文件和重复图片关系 |
| OCR 辅助 | 建立 OCR 数据库、回填 OCR、检查远程图片 |
| 本地服务 | `server.py` 提供目录、搜索、读取、保存和图片管理 API |

这里既是知识库 Viewer，也是母文章人工维护入口：用户可以修改正文、删图、检查 OCR、整理双链和持续迭代底稿。

## 6. WritingMoney：母文章铸造与批量成稿 Demo

**位置：** `/Users/works14/Documents/output_md/wiki-viewer/WritingMoney`

WritingMoney 当前是静态可交互 Demo，目标是承接监控系统产生的关键词或选题。它把写作拆成两个独立模式，但共享母文章库。

### 6.1 母文章铸造

一个项目对应一个长期内容资产。流程包括：输入选题和写作目的、添加 URL 素材、从本地知识库推荐 Markdown、把素材标为必用/参考/不用、选择写作模板、确认写作方案、生成写作包，未来再交给 Agent 生成母文章。项目计划使用 `job.json` 保存阶段、状态、模板和输出路径。

### 6.2 批量成稿

一个项目对应一次生产批次，主视角是关键词而不是某篇母文章。每个关键词可以匹配零篇、一篇或多篇母文章；一篇母文章也可服务多个关键词。批次字段和流程包括：

| 关注面 | 当前设计 |
|---|---|
| 批次来源 | 手动创建或未来由选题监控系统传入 |
| 批次输入 | 批次名称、自然语言写作要求、关键词列表 |
| 关键词任务 | 关键词、写作目的、匹配母文章、推荐钩子、建议篇数 |
| 母文章状态 | `ready` 可直接成稿；`needs-mother` 返回母文章铸造 |
| 人工确认 | AI 先完成匹配和建议，用户确认后才生成队列 |
| 成稿队列 | 按“关键词 × 生成篇数”展开，状态为等待、运行中、已完成、需返工 |
| 输出边界 | 输出 Markdown、批次目录和 manifest；不在本系统管理公众号账号与发布排期 |

当前页面不进行真实 LLM 写作，也未连接三套监控系统或老发布系统；代码主要是 `index.html`、`styles.css`、`app.js`。

## 7. GEOProMax：GEO 公域引用观察台

**位置：** `/Users/works14/Documents/zkcode/GEOProMax`

这是面向 AI 搜索回答的事实观察系统。它监控的不是传统搜索结果页，而是“某个问题在某次采集时，AI 如何回答、引用了哪些来源、来源位次怎样变化”。

| 关注面 | 当前功能 |
|---|---|
| 问题库 | 按问题和类目组织观察对象 |
| 回答快照 | 保存采集时间、AI 回答 Markdown、answer hash 和状态 |
| 引用源 | 提取引用标题、链接、域名、平台、作者和引用位次 |
| 时间变化 | 查看同一问题多个快照的回答及引用位次矩阵 |
| 问题视角 | 查看问题、回答原文、引用源榜单、平台分布和历史快照 |
| 引用源视角 | 先看平台，再看平台内作者及其被引用内容和覆盖问题 |
| RedFox 接入 | 支持单问题手动刷新，把结果落为 `Markdown/` 快照 |
| 当前入口 | `official_app.py` 为工程化入口；`geo_demo.py` 为 Demo 数据入口 |

它目前与微信/小红书关键词监控独立，但产出的“问题、追问建议、原始检索词”未来同样可以作为 WritingMoney 的选题输入。

## 9. 当前系统边界

1. 微信搜一搜关键词监控、公众号监控、小红书关键词监控和 GEOProMax 各自拥有独立事实源、存储和页面，不应把其中一个目录当成另一个系统的子模块。
2. 微信关键词系统只代表搜索结果中的文章；公众号监控代表指定账号的文章集合；两者可能出现同一篇文章，但当前没有统一文章主键或自动合并。
3. 三类发现系统目前不会自动向 WritingMoney 投递选题；WritingMoney 只在 Demo 中预留“选题监控来源”和关键词输入。
4. Wiki Viewer 管理 Markdown 和母文章资产；WritingMoney 管理写作决策与生产批次；老 Publish 管理账号、排期和微信公众号后台操作，三者职责不同。
5. 当前最通用的跨系统载体是 Markdown；未来若连接，建议额外传递稳定 ID、来源系统、关键词/问题、原文 URL、发布时间、作者/账号、分类和本地 Markdown 路径。
