# 数据架构与入库说明

## 数据流

```text
原始 JSON
  ↓ 拆解JSON
统一临时数据
  ├─→ 保存Markdown正文 → answers/ + sources/
  └─→ 写入SQLite       → index/geopromax.sqlite
```

原始 JSON 不修改；Markdown 只存正文；SQLite 存稳定元数据、时间快照、序位和双向关系。临时统一数据写入 `~/.geopromax/tmp/`，成功后删除。

## 八张表

### `batches`

一份原始 JSON 对应一个采集批次。

| 字段 | 含义 |
|---|---|
| `id` | 自增主键 |
| `raw_file` | 归档 JSON 路径，唯一 |
| `input_file` / `output_file` | 原采集程序记录的输入、输出路径 |
| `app` | AI APP，当前为豆包 |
| `channel` | 采集端，当前为 `mobile`，未来可为 `web` / `api` |
| `mode` | `quick` / `pro` |
| `new_context` | 本批次是否清空上下文，布尔值 |
| `status` | 批次状态 |
| `started_at` / `finished_at` / `duration_seconds` | 批次时间 |
| `total` / `succeeded` / `failed` | 批次统计 |

### `answers`

一次提问行为对应一行。

| 字段 | 含义 |
|---|---|
| `id` / `batch_id` | 主键、所属批次 |
| `question` | 提问原文 |
| `status` / `error` | 结果状态和错误 |
| `mode` / `new_context` | 该次回答的实际模式和上下文状态 |
| `started_at` / `finished_at` / `duration_seconds` | 该次回答时间 |
| `share_link` | 手机端分享链接 |
| `markdown_path` | AI 回答正文路径 |

### `tools`

回答内有序工具链。`answer_id` 关联回答，`position` 是原数组物理顺序，`type` 和 `content` 保留原工具类型与内容。

### `tool_search_keywords`

搜索工具的关键词序列。`tool_id` 关联工具，`position` 是原数组顺序，`keyword` 是关键词原文。

### `sources`

文章或视频的稳定主体。

| 字段 | 含义 |
|---|---|
| `id` | `sha256(平台 + "\n" + 规范化URL)[:20]` |
| `url` / `platform` | 保守规范化链接和采集平台原名，组合唯一 |
| `title` / `summary` | 最新非空标题和摘要 |
| `favicon` / `cover_image` | 采集时的平台图标、封面原值 |
| `published_at` | 发布时间 |
| `author` / `author_profile_link` | 作者和主页 |
| `markdown_path` / `content_hash` | 最新正文与内容指纹 |

### `source_relations`

回答与来源的事件关系，也是双向索引核心。

| 字段 | 含义 |
|---|---|
| `answer_id` / `tool_id` / `source_id` | 回答、工具、来源关联 |
| `type` | `search_result` / `text_reference` / `image_reference` / `related_video` |
| `position` | 该关系在原数组中的物理顺序 |
| `anchor_index` / `anchor_text` | 原回答中的引用位置和锚点文本 |
| `image_url` | 图片引用的图片链接 |
| `error` | 该来源的解析错误 |

### `suggested_questions`

每次回答返回的多个推荐问题。`answer_id` 关联回答，`position` 保留物理顺序。

### `source_metrics`

来源在某次回答采集时的动态快照：`read_count`、`like_count`、`comment_count`、`favorite_count`、`share_count`。以 `(source_id, answer_id)` 唯一，因此能按时间观察变化，不写入 Markdown。

## 平台聚合

`data/platforms.json` 维护标准平台名、明确别名和可选图标域名。`原子能力/平台归一化` 负责把采集平台名转换为分析口径：

- `raw_platform`：采集原名，属于事实数据。
- `platform`：标准平台名，用于 Web 展示、统计与内容分析。
- `icon_url`：由标准平台的稳定域名生成，只用于展示。

归一化只读取显式别名，不按域名、标题或相似文本自动推断。它不回写 SQLite、不移动 Markdown，也不改变来源 ID。

## 当前边界

- 相同平台和规范化 URL 视为同一来源。
- 平台不同即视为不同来源。
- 不做标题、正文近似判断，不做自动合并。
- 来源 Markdown 固定为 `<平台>/<20位ID>.md`，纯正文。
- 平台聚合是分析层视图，原始平台名仍保留在 SQLite 与 Markdown 路径中。
- RedFox 快照暂存 `data/redfox/`，尚未写入 SQLite 八表；手机端豆包数据是当前主链路。
