# GEOProMax Demo 界面大白话描述

更新时间：2026-07-05  
用途：给后续接手开发者用大白话理解当前 Demo 页面。本文只描述界面和用户看到的东西，不讲代码实现。

---

## 1. 这个页面是什么

这个页面叫 **GEO 公域引用观察台**。它不是内容发布后台，也不是投放后台，更不是一个上来就给“机会”“信任”“建议”的策略平台。它当前最重要的目标，是把豆包这类 GEO 平台在回答问题时引用了哪些公域内容、这些内容来自哪些平台、背后有哪些作者、这些引用在不同时间点怎么变化，尽量直观地摆出来。

用一句话讲，它就是一个 **AI 回答引用源观察仪表盘**。用户来到这里，不是为了立刻决定要写哪篇文章，而是先看事实：问了什么问题，AI 怎么回答，AI 搜了哪些词，AI 引用了哪些链接，这些链接来自会计学堂、抖音、今日头条、官网还是其他网页，哪些作者或机构反复出现。

当前 Demo 用两份豆包批量采集 JSON 做数据，所以内容主要围绕港险问题展开。页面里所有东西都应该按“公域观察”理解：我们还没有假设自己的内容已经发布，也没有假设自己的内容已经被引用。这个阶段更像 SEO 系统早期的关键词观察：先把行业里已经存在的内容、平台、作者、问题结构看清楚。

---

## 2. 整体布局

页面是上下加左右结构。最上面是一条轻量顶部栏，下面主体分成左侧列表和右侧详情。左侧负责“选对象”，右侧负责“看详情”。用户的基本操作非常简单：先在顶部选择看 **问题** 还是看 **引用源**，然后在左侧点一个条目，右侧就展示完整信息。

这个布局故意接近 SEO 系统：白底、灰线、小字体、信息密度高，但不做花哨大屏。它适合长时间看数据，而不是做展示型 PPT。每个数字旁边都有短标签，比如“引用源”“平台”“作者”“时间点”，用户不用猜含义。很多地方用了小标签、小热力格、小位次徽章，让页面能一眼扫出重点。

顶部左边是系统名 **GEO 公域引用观察台**，旁边有三个总数：观察问题、引用源、创作者。这三个数字告诉用户当前这批数据有多大。中间有两个视角按钮：**问题** 和 **引用源**。问题视角从用户提问出发，类似 SEO 里的关键词页；引用源视角从平台和作者出发，类似 SEO 里的公众号板块，但因为来源跨平台，所以先按平台分，再看平台里的作者。

顶部右侧是 Demo 和时间信息，提醒用户这不是正式实时系统。以后接 API 后，这里可以放最近采集时间、批次状态、是否正在刷新、数据源版本等。

---

## 3. 左侧列表怎么用

不管在哪个视角，左侧都是筛选和选择区。上面是搜索框，问题视角可以搜问题、类目、平台；引用源视角可以搜平台或作者。搜索框的作用不是复杂检索，而是让用户快速定位自己关心的对象。

搜索框下面是排序按钮。问题视角里有“最近”“引用源”“快照”：最近表示按最新采集排序，引用源表示按这个问题当前引用了多少来源排序，快照表示按这个问题有多少个时间点排序。引用源视角里有“引用占比”“创作者”“覆盖问题”：引用占比看哪个平台被引用最多，创作者看哪个平台作者生态更丰富，覆盖问题看哪个平台涉及的问题面更广。

再下面是筛选标签。问题视角里是问题类目，比如风控审查词、热门单品、提领成交词、保费融资词。这些不是 AI 推荐标签，而是为了把问题池分桶。引用源视角里的筛选更偏平台和来源类型。

左侧每一行都像一个小排行榜：有排名数字、主标题、若干标签、热力格和右侧关键数字。这个设计让用户即使不点进去，也能大概知道哪个问题材料更多、哪个平台被引用更多。

---

## 4. 问题视角

切到 **问题** 后，左侧每一行就是一个问题，比如“港险分红实现率排名靠谱吗？”“225提领靠谱吗？”“友邦环宇盈活分红实现率有什么缺点？”。问题下面会显示类目、引用源数量、快照数量、最近采集时间。右侧最大数字是“引用源”，表示这个问题当前 AI 回答里拿到了多少来源。它不是机会分，也不是搜索量，只是采集事实。

点击问题后，右侧第一张卡是问题总览。最上面是问题原文，下面有一句解释：这里先记录事实，看每个时间点 AI 怎么回答、引用了哪些来源、引用位次如何变化，价值判断留给人或后续 Agent。这句话是整个产品的基调：页面不是替用户下结论，而是把事实摆平。

总览卡里有一排标签：问题类目、当前快照时间、AI 平台、端侧、观察窗口。下面是一排统计数字：当前引用源、当前平台、当前作者、异常来源、时间点。当前引用源表示这次回答引用多少条来源；当前平台表示这些来源分布在多少个平台；当前作者表示背后有多少作者或机构；异常来源表示抓取失败、404、无链接等情况；时间点表示这个问题有多少次快照。每个数字都是事实数量，不是评分。

---

## 5. 问题详情里的时间

问题总览下面是 **采集快照**。这里横向展示多个时间点，每个时间点像一个小按钮，里面有时间、日期、引用数量和平台数量。用户点击不同快照，右侧回答、引用源榜单、平台分布都会切换到对应时间点。

这块是未来 GEO 系统非常重要的基础。SEO 系统后来最大的价值之一，就是加入时间后能观察“发布—收录—排名变化”。GEO 也一样：同一个问题，上午和下午 AI 引用的来源可能不同；今天和明天回答可能不同；某个平台可能突然出现，某个作者可能突然消失。没有快照，这些变化都会被最新结果覆盖。

当前 Demo 的历史快照有一部分是为了展示 3 小时采集形态而生成的，所以页面会提示 Demo 说明。正式 API 接入后，每个快照都应该是真实采集结果，并且保存自己的回答正文、引用列表、回答 hash 和状态。

采集快照下面是 **引用位次矩阵**。它像一个表格，左边是引用源，右边每一列是一个时间点。格子里的数字代表这个来源在那个时间点排第几位；如果没出现，就显示横杠。用户可以直接看出一个来源是不是稳定被 AI 引用：连续多个时间点都在前 3，说明很稳定；只出现一次，说明可能只是临时来源；从第 8 升到第 2，说明引用位次在变强。这里仍然只展示事实，不写“可信”“机会”之类判断。

---

## 6. AI 回答和引用源榜单

问题详情里最核心的一张卡是 **AI 回答原文**。这里不是一句摘要，而是尽量像真实 AI 回答一样展示完整内容。当前页面已经有基础 Markdown 渲染，能展示段落、标题、加粗、列表。后续如果回答里有表格、链接、引用块，也应该继续按 Markdown 正常渲染。

卡片右上角会显示字段来源、回答字数和 answer hash。字段来源说明这段内容来自原始 JSON 的 `answer` 字段；字数帮助判断回答是否完整；hash 用来判断同一个问题不同时间点的回答是否变化。正式系统里，如果两次回答 hash 不一样，就说明 AI 回答发生了变化。

回答下面还有两类事实：原始检索词和 AI 追问建议。原始检索词来自 `tool.keywords`，说明平台为了回答这个问题实际搜了什么；AI 追问建议来自 `suggested_questions`，说明平台建议用户继续问什么。这两类信息对 GEO 很重要，因为它们是未来扩展问题链的材料。但页面要明确：它们只是采集事实，不是平台评分，也不是推荐标签。

再往下是 **当前快照引用源榜单**。这里列出当前时间点 AI 引用或工具返回的所有来源。每一行左边是位次，中间是平台图标、标题、平台名、作者名、发布时间或异常信息，右边如果有链接就有“原文”按钮。用户可以直接看到：这个问题主要引用会计学堂、抖音、头条，还是官网和其他网页；有没有大量异常来源；哪些作者反复出现。

点击某个引用源行，会切到引用源视角，并定位到它所属的平台和作者。这条交互的意思是：用户可以从一个问题一路追到平台和作者。比如在某个问题里发现一个会计学堂作者经常被引用，就可以点过去看这个作者还覆盖了哪些问题。

问题详情最后是 **当前快照平台分布**。它按平台统计当前回答里的引用数量和占比。这个表很简单，但很有价值：它让用户知道某类问题的资料生态主要集中在哪里。比如有些问题会计学堂权重很高，有些问题抖音视频更多，有些问题普通网页更多。这里也只做统计，不做平台好坏判断。

## 7. 引用源视角

切到 **引用源** 后，页面逻辑变成：先看平台，再看平台里的作者，最后看作者详情。用户之前已经明确过，单个链接太碎，不适合作为一级页面；创作者单独成页又会和引用源重复。所以现在把创作者合并进引用源视角，用三栏结构承载。

最左侧列表变成平台列表，比如会计学堂、抖音、今日头条、网易、搜狐、AIA 官网、找法网、其他网页、异常来源等。每个平台一行，显示作者数、来源数、覆盖问题数、引用占比。如果有真实 logo，就用真实 logo；外面保留细灰边框；找不到图标时才用文字缩写。

平台行右侧的大数字是占比百分比，表示这个平台在所有引用中的大致占比。这个数字不是“平台可信度”，只是“被引用多不多”。比如会计学堂出现很多，说明当前豆包回答港险问题时容易引用会计学堂；抖音多，说明短视频也进入了 AI 引用链；其他网页多，说明还有很多零散域名需要后续归类。

点中一个平台后，右侧会出现平台工作台。工作台左栏是这个平台下的创作者列表，右栏是选中创作者的详情。创作者不是微信公众号意义上的账号，而是跨平台的内容发布主体：会计学堂可能是某个作者或栏目，抖音可能是视频创作者，头条可能是内容作者，官网可能是官方机构。当前 Demo 有些作者名是从链接或标题里推断的，正式系统要保留原始名和识别置信度，不要太早强合并。

创作者列表每一行有作者名、来源数、覆盖问题数、命中快照数、最佳引用位、问题类目标签。右侧最大数字是快照命中，表示这个作者在多少个 answer snapshot 里被引用过。这个数字比单纯来源数更体现时间稳定性。

---

## 8. 创作者详情

点中创作者后，右侧第一张卡是作者总览。标题是平台图标加作者名，下面说明：引用源视角先看平台，再看平台里的作者，最后看作者在不同时间点被 AI 引用的事实。卡片里有平台、问题类目、首次出现、最近出现、快照命中次数，以及引用源数量、覆盖问题数量、最佳引用位、平台异常等统计。

这张卡不是作者评分，只是事实画像。它不说作者好不好，也不说是否值得合作，只说它被引用了多少次、覆盖多少问题、最好排到第几位、最近什么时候出现。

下面是 **被引用内容**。这里列出这个作者旗下被 AI 引用过的具体链接。每行都有引用位次、标题、平台、域名、发布时间、首次引用、最近引用、命中快照次数和原文按钮。它回答的是：这个作者到底靠哪些内容被 AI 看到。未来持续采集后，可以观察某篇内容是长期稳定被引用，还是只短暂出现。

再下面是 **覆盖问题**。这里列出这个作者覆盖了哪些问题。点击问题可以回到问题视角。它回答的是：这个作者的内容被哪些问题复用。一个作者如果覆盖很多问题，说明它在当前问题池里存在感更强；但页面仍然只展示覆盖事实，不替用户下“机会”结论。

最后是 **问题类目分布**，按风控审查词、保费融资词、热门单品、提领成交词等类目统计作者覆盖范围。它帮助用户看作者主要在哪类问题里被引用。

---

## 9. 当前 Demo 的注意点

当前页面虽然像完整产品，但仍是 Demo。第一，历史快照有展示性质，正式接 API 后必须换成真实快照。第二，部分作者名是推断出来的，不能当成绝对真实身份。第三，平台 logo 现在主要靠 favicon 和域名规则补齐，正式系统最好用平台表统一管理。第四，风险标签、回答焦点这类字段只是为了 Demo 易读，不应该进入事实层主数据。第五，AI 回答现在来自 JSON 的 `answer` 字段，正式系统应该把完整回答保存成 Markdown 文件，主数据只存 path 和 hash。

---

## 10. 这套界面最终想给用户什么感觉

用户打开页面，第一眼应该觉得：这不是空泛的 GEO 概念页，而是真的把问题、AI 回答、引用源、平台、作者、时间变化放到一个工作台里了。继续看下去，又会觉得很好理解：左边选问题，右边看回答和引用；切到引用源，左边选平台，右边看作者和内容。

这套界面的核心不是炫技，而是把 GEO 拆成几个用户能立刻理解的事实对象：**问题、回答、快照、引用源、平台、作者、时间**。只要这几个对象稳定下来，后续接 API、做数据库、做 Agent 分析、做内容策略，才有可靠地基。
