③ 事实数据层

原件、正文、索引各司其职
🔒 原始 JSON｜事实原件

保持不可变，方便未来重新解析、核对来源和修正算法。

📝 Markdown｜人和 Agent 可读

🧮 SQLite｜关系和时间索引

保存问题、批次、位次、来源关系、动态指标和双向查询关系。

④ Web 观察台

只展示事实，不替人做判断
🚪 run.py｜唯一入口

默认启动正式站点 8790；加 --demo 启动 8788 演示页。

🧠 web/run.py｜服务与接口

读取数据、聚合统计、提供 API、处理本地导入和单问题刷新。

🎨 web/ui.py｜共用界面

问题页与引用源页共用同一套 UI，展示回答和位次随时间的变化。

🗂️ 数据实际放在哪里？ 三种存储不是重复，而是不同职责

🔒 原始数据

采集设备交回来的原件。原则是尽量不改、不覆盖，出现问题时可以回到这里重新核对。

data/ └── raw/ └── 豆包/ └── mobile/ └── quick/ └── 批次原始 JSON

📱 手机端豆包原始 JSON 与 🦊 RedFox 原始 JSON 分目录存放；RedFox 原始结果目前位于 RedFox 适配器输出区，不进入 data/raw。

📝 Markdown 正文

让人直接打开就能阅读，也方便未来 Agent 批量理解、引用和总结。

data/ ├── answers/ │ └── 豆包／问题／历次回答.md ├── sources/ │ └── 平台／来源ID.md └── redfox/ └── 分类／单问题快照.md

📱 手机端会写入回答正文和来源正文；🦊 RedFox 只写问题快照，暂不并入 answers/ 与 sources/。

🧮 SQLite 事实索引

负责快速查询和串联关系：某个问题引用了谁、某个来源在哪些回答出现、位次如何变化。

📦 批次 💬 回答 🛠️ 工具链 🔎 搜索词 🔗 来源 🧷 回答—来源关系 💡 推荐问题 📈 来源指标快照

📱 手机端数据会进入八表主索引；🦊 RedFox 当前不在 SQLite 主链路，避免污染批次、来源关系与时间位次。

⚡ 两条数据源到底差在哪？ 存储方式、并入主库、展示用途都不一样

📱 手机端豆包 RPA｜主链路

这套数据会完整进入主数据库，是系统当前最可信的事实基础。

data/raw/豆包/mobile/quick/ data/answers/豆包/... data/sources/平台/... data/index/geopromax.sqlite
✅ 并入主库

原始 JSON、Markdown、SQLite 八表都写。

✅ 可做时间比较

回答、引用平台、引用位次、动态指标都能看变化。

🦊 RedFox｜补充链路

这套数据目前只做人辅助观察和局部刷新，不自动并入主索引。

data/redfox/分类/... └── 单问题快照 Markdown
⏸️ 暂存快照

写入本地快照目录，不进入 data/raw，不并入 SQLite 八表。

⏸️ 手动触发

仅在问题详情页手动刷新单个问题，没有全局刷新或自动刷新。

🧠 这套架构最重要的四个原则

① 事实和判断分开

系统记录谁回答了什么、引用了谁、什么时候出现；机会和策略由人或 Agent 后续判断。

② 时间是一等公民

每次采集都是独立快照，回答、引用平台和引用位次都可以沿时间比较。

③ 原始事实不被“聪明处理”污染

平台归一化只作用于分析展示层，不回写原始平台名，不擅自合并相似内容。

④ 主链路与补充链路隔离

手机端豆包是当前主数据；RedFox 是人工触发的补充来源，尚未自动并入八张主表。

✅ 主链路：手机端豆包 🦊 补充链路：RedFox 🔒 事实层：原始数据与 SQLite 👀 展示层：Web 观察台
最简单的记法：📱 采集到原始 JSON → 🧩 原子脚本拆解 → 📝 正文放 Markdown、🔗 关系放 SQLite → 🖥️ Web 页面把问题、回答、引用源和时间变化展示出来。
后台运行：macOS 使用 web/install_macos.sh 管理 com.geopromax.web，正式访问地址为 http://127.0.0.1:8790/。
