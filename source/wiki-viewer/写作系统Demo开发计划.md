# Wiki Viewer 写作系统 Demo 开发计划

最后更新：2026-07-06

## 1. Demo 定位

这个 Demo 不是一个“AI 一键写文章”的按钮，而是一个**新建文章工作台**。

目标是把现在藏在命令行和 Markdown 规则里的写作流程可视化，让用户可以在网页里完成：

- 新建一个文章项目
- 输入选题和写作目的
- 添加 URL 临时素材
- 从本地知识库推荐素材 Markdown
- 对素材做“必用 / 参考 / 不用”判断
- 推荐写作模板 Markdown
- 查看写作方案
- 最终生成一个固定写作包，交给后续 `claude -p` / Agent 写作

第一版是**纯静态可点击 Demo**：页面可以点来点去，任务状态可以变化，但数据全部预设，不真实调用 `claude -p`，也不真实解析 URL。

## 2. 核心判断

> 写作系统的关键不是让 AI 直接替人写，而是记录“人和 AI 一起做写作判断”的过程。

每一篇文章都应该有一个独立工作目录，从选题、素材、模板、方案到最终成稿全部落盘。未来 Agent 可以回看这些目录，学习用户如何选素材、如何否定素材、如何选择模板、如何修改方向。

因此，**工作目录本身就是偏好记录的基础设施**，比单独做一个“偏好设置页”更重要。

## 3. 目录设计

文章工作区统一放在：

```text
WritingSpace/
```

每篇文章一个目录，命名格式：

```text
WritingSpace/YYMMDD_文章关键词/
```

示例：

```text
WritingSpace/260706_友邦环宇盈活提领/
```

建议目录结构：

```text
WritingSpace/260706_友邦环宇盈活提领/
├── job.json
├── topic.md
├── materials.recommendations.json
├── materials.selected.json
├── materials/
│   ├── local_001.md
│   ├── local_002.md
│   └── url_001.md
├── templates.recommendations.json
├── writing_plan.md
├── writing_package.md
├── output.md
└── logs.md
```

第一版静态 Demo 不需要真实生成所有文件，但页面和计划要按这个结构设计。

## 4. job.json 状态文件

每个文章项目必须有一个 `job.json`，用于恢复侧边栏任务列表和当前进度。

建议字段：

```json
{
  "id": "260706_友邦环宇盈活提领",
  "title": "友邦环宇盈活提领",
  "topic": "友邦环宇盈活提领能力怎么样",
  "purpose": "想写一篇针对已看过计划书、关心养老提领的客户文章",
  "status": "waiting_user",
  "stage": "materials_review",
  "created_at": "2026-07-06T10:00:00+08:00",
  "updated_at": "2026-07-06T10:20:00+08:00",
  "archived": false,
  "selected_template": "wiki/创作框架/条款黑盒写作框架.md",
  "output_path": null
}
```

状态枚举建议：

| 状态 | 含义 |
|---|---|
| `collecting_materials` | 正在收集 / 推荐素材 |
| `waiting_user` | 等用户确认下一步 |
| `recommending_template` | 正在推荐写作模板 |
| `planning` | 正在生成写作方案 |
| `writing` | 正在生成全文 |
| `completed` | 已完成 |
| `failed` | 失败 |
| `archived` | 已归档 |

## 5. 页面结构

Demo 建议做成三栏：

### 5.1 左侧：文章项目列表

展示所有 WritingSpace 里的文章项目。

每个项目显示：

- 文章标题 / 关键词
- 当前阶段
- 状态点
- 创建时间

状态视觉：

| 状态 | 页面表现 |
|---|---|
| 正在运行 | 转圈 |
| 等待用户 | 绿色小点 |
| 已完成 | 对勾 |
| 失败 | 红点 |
| 归档 | 灰色 |

Demo 必须体现**多文章并行**：例如一篇正在“素材推荐中”，另一篇处于“等待确认模板”。

### 5.2 中间：当前文章工作台

当前文章的主要操作区，按阶段切换。

阶段一：新建文章

- 字段 1：选题 / 关键词
- 字段 2：写作目的

字段说明：

- 选题是用户自己输入的
- 目的可以自由写，不强制结构化
- 示例：`想写给已经拿到计划书、关心提领养老的人`

阶段二：素材篮

素材来源：

- 用户粘贴 URL，作为临时素材
- AI 从本地知识库推荐 Markdown

AI 推荐范围：

- 只允许在本地知识库内推荐
- 如果本地知识库不够，必须坦诚提示“不够 / 没有相似内容 / 建议补充 URL”
- 不联网搜索，不编造素材

每个素材卡片显示：

- 文件名 / URL 标题
- 推荐语，限制约 100 字
- 可用观点
- 可用图片 / 表格
- 状态按钮：`必用 / 参考 / 不用`

阶段三：模板推荐

模板 Markdown 现阶段来自：

```text
wiki/创作框架/*.md
```

未来用户可以继续扩展这些模板，每个创作框架都是一个 Markdown。

模板推荐最多 3 个，一般推荐 1 个即可。

每个模板卡片显示：

- 模板名称
- Markdown 路径
- 约 100 字推荐理由
- 选择按钮

如果用户认为模板不合适，可以在 Obsidian / Wiki Viewer 外部修改模板，然后回到系统重新跑模板推荐。

阶段四：写作方案确认

写作方案不是最终正文，而是写作前的作战计划。

建议展示 5 样：

- 推荐标题方向
- 核心判断
- 文章大纲
- 重点素材引用
- 结尾引流方案

第一版 Demo 不需要支持页面内直接改写方案，只需要展示：

- 确认方案
- 重跑方案
- 返回修改素材 / 模板

阶段五：生成写作包

确认后生成固定写作包，未来交给 `claude -p`。

写作包应包含：

- 选题
- 写作目的
- 素材篮全文
- 素材状态：必用 / 参考 / 不用
- URL 临时素材 Markdown
- 选中的模板 Markdown
- 写作方案
- AGENTS / 写作规则引用
- 输出路径要求

第一版 Demo 只展示预设的 `writing_package.md` 预览，不真实调用 Agent。

### 5.3 右侧：详情 / 预览面板

根据当前点击对象变化：

- 点击素材：展示素材摘要、可用观点、图片表格
- 点击模板：展示模板摘要和推荐理由
- 点击写作方案：展示完整方案
- 点击写作包：展示最终传给 Agent 的输入包

## 6. 素材推荐 JSON

`materials.recommendations.json` 示例：

```json
{
  "query": "友邦环宇盈活提领能力怎么样",
  "scope": "local_knowledge_base_only",
  "enough_materials": true,
  "items": [
    {
      "id": "local_001",
      "path": "wiki/产品母页/友邦环宇盈活.md",
      "title": "友邦环宇盈活",
      "recommendation": "这篇母页直接给出环宇盈活的核心边界：长期静态收益强，但提领后表现弱，适合作为主事实来源。",
      "usable_points": [
        "环宇盈活适合长期静态持有，不适合高频提领",
        "567 极致提领下可能断单",
        "提领时会更早消耗终期账户"
      ],
      "usable_visuals": [
        "环宇盈活 vs 星河尊享2 提领对比表",
        "四大外资 566 提领对比表"
      ],
      "status": "recommended"
    }
  ]
}
```

用户确认后生成 `materials.selected.json`：

```json
{
  "items": [
    {
      "id": "local_001",
      "path": "wiki/产品母页/友邦环宇盈活.md",
      "usage": "must_use"
    },
    {
      "id": "local_002",
      "path": "wiki/产品母页/港险提领功能横评.md",
      "usage": "reference"
    },
    {
      "id": "local_003",
      "path": "output/260521_友邦环宇盈活产品测评.md",
      "usage": "skip"
    }
  ]
}
```

usage 枚举：

| usage | 含义 |
|---|---|
| `must_use` | 必用 |
| `reference` | 参考 |
| `skip` | 不用 |

## 7. 模板推荐 JSON

`templates.recommendations.json` 示例：

```json
{
  "items": [
    {
      "id": "tpl_001",
      "path": "wiki/创作框架/条款黑盒写作框架.md",
      "title": "条款黑盒写作框架",
      "recommendation": "这个选题核心不是介绍环宇盈活，而是拆提领顺序和账户消耗机制，适合用黑盒框架把销售话术和真实执行规则分开。"
    },
    {
      "id": "tpl_002",
      "path": "wiki/创作框架/产品写作框架.md",
      "title": "产品写作框架",
      "recommendation": "如果文章目标是做单品测评，可以用产品框架，但需要避免写成普通产品介绍，要把提领短板放在主线。"
    }
  ],
  "selected": "tpl_001"
}
```

## 8. 写作方案内容

`writing_plan.md` 建议结构：

```markdown
# 写作方案

## 选题

友邦环宇盈活提领能力怎么样

## 写作目的

写给已经看过友邦计划书、关心未来养老提领的客户，让他意识到环宇盈活不是不能买，而是不能把它当高频提领工具。

## 推荐标题方向

友邦环宇盈活提领，最容易误判的不是收益，而是账户怎么被掏空

## 核心判断

环宇盈活的主场是长期静态复利，不是提领现金流。它赢在不动，输在持续大额提领。

## 文章大纲

1. 开头先拆误会：很多人看到 6.5% 演示，就以为提领也强
2. 承认环宇盈活真实优势：品牌、静态收益、分红实现率
3. 重点拆提领机制：复归红利和终期账户如何被消耗
4. 对比星河尊享2 / 盛利2：不同提领口径下谁更合适
5. 分人群判断：谁适合环宇，谁应该换产品
6. 结尾破窗：同样提领，启动年份和比例不同，20 年后差距可能很大

## 重点素材引用

- `wiki/产品母页/友邦环宇盈活.md`
- `wiki/产品母页/港险提领功能横评.md`
- `wiki/产品母页/永明星河尊享2.md`

## 结尾引流方案

把问题落到读者自己的计划书：什么时候开始领、每年领多少、分红打折后是否还能撑住。引导用户发计划书做 20 分钟提领压力测试。
```

## 9. 偏好记录原则

第一版 Demo 不做偏好学习，但必须保留未来可学习的数据结构。

以下动作都应能在工作目录中留下痕迹：

- 用户选了哪些素材
- 用户拒绝了哪些素材
- 用户把哪些素材标为参考
- 用户选择了哪个模板
- 用户重跑过哪些模板推荐
- 用户确认了哪个写作方案
- 最终写作包长什么样
- 最终成稿和写作包之间有什么差异

未来 Agent 可以通过研究多个 WritingSpace 项目目录，反推出：

- 用户偏好什么类型素材
- 用户不喜欢什么类型素材
- 哪些模板更容易被用户接受
- 哪些标题方向更容易被保留
- 用户常改哪些核心判断
- 用户最终发布的文章和初稿之间差异在哪里

## 10. 第一版静态 Demo 范围

第一版必须实现：

- 左侧多文章任务列表
- 新建文章弹窗 / 页面
- 预设素材推荐列表
- 素材三态切换：必用 / 参考 / 不用
- URL 临时素材添加入口
- 预设模板推荐列表
- 模板选择
- 写作方案确认页
- 写作包预览
- 状态变化：运行中、等待用户、已完成

第一版不实现：

- 不真实调用 `claude -p`
- 不真实解析 URL
- 不真实全文搜索知识库
- 不真实生成文章
- 不做偏好学习
- 不做发布系统
- 不做独立文章编辑页
- 不改造 `output/` 文章编辑流程

## 11. 和现有 Wiki Viewer 的关系

现有 Wiki Viewer 继续负责：

- 浏览知识库
- 编辑 Markdown
- 查看 output 文章
- 图片查看、OCR、删除、未来洗图

写作系统新增的是：

- 新建文章项目
- 管理写作前决策流程
- 生成 WritingSpace 工作目录
- 未来调用 Agent 生成 output 文章

成稿后的编辑仍然回到 Wiki Viewer / output 体系，不单独做写作系统编辑页。

## 12. 后续真实版本接入点

静态 Demo 通过后，再逐步接入真实能力：

1. 读取 `WritingSpace/` 真实任务目录
2. 新建项目时真实创建目录和 `job.json`
3. 调 `claude -p` 推荐素材，输出 `materials.recommendations.json`
4. URL 解析为临时 Markdown，进入 `materials/`
5. 调 `claude -p` 推荐模板，输出 `templates.recommendations.json`
6. 生成 `writing_plan.md`
7. 生成 `writing_package.md`
8. 调 `claude -p` 写完整文章
9. 写入 `output/`
10. 将最终成稿复制到工作目录 `output.md`

## 13. 当前确认结论

- 首页形态：新建文章工作台
- 最少字段：选题 + 写作目的
- URL：临时素材，不自动进入正式 Wiki 主库
- AI 推荐素材：只查本地知识库，不够就直说
- 素材卡片：推荐语、可用观点、可用图片 / 表格
- 素材状态：必用 / 参考 / 不用
- 模板来源：当前默认 `wiki/创作框架/*.md`
- 模板推荐：最多 3 个，一般 1 个即可
- 写作方案：标题方向、核心判断、大纲、重点素材引用、结尾引流方案
- 多线程：必须体现多文章并行和状态点
- 工作目录：`WritingSpace/YYMMDD_文章关键词/`
- 状态文件：`job.json`
- 第一版：纯静态可点击 Demo
