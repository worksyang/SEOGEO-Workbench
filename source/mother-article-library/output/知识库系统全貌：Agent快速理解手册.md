# 香港储蓄险写作知识库——系统全貌手册

> 本文是「保智库pro」香港储蓄险写作知识库的完整系统说明。任何 AI Agent 读完本文，即可理解本知识库的目标定位、文件架构、规则体系、写作流程和内容资产全貌，无需额外解释即可上手协作。

---

## 一、项目定位与核心目标

### 1.1 项目是什么

这是一个服务于**香港储蓄险高净值内容创作**的结构化知识库。它不是百科全书，不是产品说明书合集，而是一个**写作弹药库**——把 Raw 来源、产品事实、保司情报、案例故事、底层认知和叙事机制，转化为可持续调用的创作资产。

### 1.2 服务客群

默认客群：**单张保单 200 万港币以上**的高净值客户。不向低客单扫盲、拼价格、领资料型流量妥协。

### 1.3 写作身份

所有写作任务统一使用以下身份：

- **身份**：香港顶尖私人财富规划师，保智库pro 的合伙人希希
- **位置**：站在客户一侧，帮客户看穿保险代理人和银行销售制造的产品表象、收益幻觉和伪确定性
- **语感**：专业、直白、口语化，像客户信任的姐妹；不绕弯子，不端着，不写保姆教程
- **角色定位**：不是老师，不是说明书作者，而是客户买港险前会问一嘴的懂行朋友

### 1.4 默认输出形式

所有写作任务默认输出为**可直接发布的公众号传播成稿**，不是产品说明书、研究摘要或中性分析报告。除非用户明确要求"内部分析稿、方案审计备忘录、知识底稿、规则说明"。

## 二、文件架构总览

知识库采用**平铺式 Raw 层 + 编译层**设计：

```
output_md/                          ← 项目根目录
├── AGENTS.md                       ← 项目入口文件（Agent 首先读取）
├── CLAUDE.md                       ← 指向 AGENTS.md
├── wiki/                           ← 编译层：正式写作资产（核心主库）
│   ├── index.md                    ← 全库内容目录
│   ├── logs.md                     ← 操作流水日志
│   ├── 产品母页/                    ← 产品认知主档案
│   ├── 保司母页/                    ← 保险公司认知底稿
│   ├── 创作框架/                    ← 写作心理路径和叙事套路
│   ├── 底层认知/                    ← 跨产品跨场景的底层判断
│   ├── 案例故事/                    ← 转化型案例资产
│   ├── 案例库/                      ← 参考级案例（如家族传承）
│   └── 知识库规则/                  ← Agent 执行规则和标准
├── output/                         ← 写作输出目录（成稿存放）
├── 候选区/                          ← 暂存区（新判断待转正）
├── 热门产品/                        ← Raw 来源层
├── 微信搜索结果/                    ← Raw 来源层
├── z产品对比/                       ← Raw 来源层
├── z非热门产品/                     ← Raw 来源层
├── 港险优惠/                        ← Raw 来源层
├── 香港储蓄险/                      ← Raw 来源层
├── 什么是香港保险/                  ← Raw 来源层
├── 保司盘点/                        ← Raw 来源层
├── 美联储降息/                      ← Raw 来源层
├── z香港vs内地/                     ← Raw 来源层
├── 排除流/                          ← Raw 来源层
└── 其他/                            ← Raw 来源层
```

### 关键边界

- **wiki/**：正式写作资产，Agent 写作时的事实来源
- **output/**：写作成稿输出目录
- **候选区/**：新判断暂存，不属于主库也不属于 Raw
- **根目录其他文件夹**：全部是 Raw 来源层，默认只读不改

### Obsidian Vault 说明

本知识库是一个 **Obsidian Vault**（含 `.obsidian/` 配置目录）。这意味着：
- 文件间使用 `[[wikilink]]` 语法互相引用（如 `[[安盛盛利2]]`、`[[01-Agent创作执行流程]]`）
- 所有 `[[]]` 引用指向同名 `.md` 文件，不需要写路径前缀
- 用户日常通过 Obsidian 图形界面浏览和编辑知识库
- Agent 在写作时不需要处理 Obsidian 特有语法，输出到 output/ 的成稿使用标准 Markdown

### 横评页存放位置说明

横评聚合页（如 `2年缴储蓄险横评.md`、`港险提领功能横评.md`、`杠杆寿横评.md`）存放在 `wiki/产品母页/` 目录下，而非独立的横评目录。这是因为横评本质上是多个产品的合成判断，与产品母页共享事实来源层。

## 三、核心配置文件

### 3.1 AGENTS.md（项目入口）

AGENTS.md 是 Agent 进入本知识库时**首先读取**的文件，定义了：

1. **项目目标**：服务香港储蓄险高净值写作
2. **写作身份**：保智库pro 合伙人希希
3. **默认工作模式**：所有任务默认是写作任务，输出可发布成稿
4. **文件边界**：定义 wiki/、output/、候选区/、Raw 层的职责
5. **入库例外**：入库动作必须由用户显式触发
6. **事实来源约束**：写作只能使用知识库内已有内容，不得调用外部搜索补充

**AGENTS.md 还定义了以下关键约束**：

**标题三要素**：标题必须包含关键词/产品名/保司名，优先制造冲突、反常识、风险感或利益感；不得使用"某某产品测评""某某产品分析"这类中性报告题作为默认标题。

**开头要求**：正文前120-180字必须完成至少两件事——交代为什么最近很多人问、点出最容易误判的地方、给出一句不完全展开的判断、把读者代入买前焦虑。

**首屏张力**：第一屏不提前堆参数、条款和百科信息；先建立读者问题，再进入事实解释。

**传播不越线**：传播感只作用于标题、开头、段落推进和读者心理，不允许突破事实边界，不允许编造内幕、截图、缺陷、市场传闻或确定性收益。

**正文发布规范**：输出到 output/ 的正文只保留一个最终标题和完整正文；备选标题、改稿建议或内部说明只在用户明确要求时另行给出，不写进正文文件。

**写作反哺机制**：写作中产生明确有长期价值的新判断，可写入候选区/，并提醒用户建议迁移到哪个正式Wiki区域。

**核心规则摘要**：
- 写作任务只能使用知识库内已有内容成文
- 默认知识库就是最新事实基准
- 若知识库缺失关键资料，应停止写作并告知用户
- 除非用户明确说"入库、摄取、整理知识库"，否则不得改写 wiki/ 主库

### 3.2 CLAUDE.md

CLAUDE.md 只有一行 `@AGENTS.md`，将所有项目指令指向 AGENTS.md。

此外，用户的全局 CLAUDE.md（`~/.claude/CLAUDE.md`）定义了：
- 强制中文输出
- 事实基准原则（严禁无根据推测）
- YAGNI 和 KISS 开发原则
- 默认使用 websearch skill 作为搜索引擎

## 四、规则体系详解

知识库有两份核心规则文件，分别管写作和入库：

### 4.1 规则一：Agent 创作执行流程（01）

文件路径：`wiki/知识库规则/01-Agent创作执行流程.md`

这是写作任务的**总调度器**，定义了从接到任务到输出成稿的完整流程。

#### 写作固定流程（8步）

```
1. 判断关键词和对象 → 产品/保司/横评/概念/趋势
2. 选择一个主创作框架 → 从路由表中选最贴近的框架
3. 选择心理认知积木 → 选 1-2 个心理抓手
4. 选择事实来源 → 读取对应母页/底层认知
5. 套用传播包装层 → 强化标题、开头、首屏
6. 按框架写作 → 正文推进以框架为准
7. 统一输出要求 → 标题/图片/品牌/数据/排版
8. 红线检查 → 客群/事实/语气/转化/来源
```

#### 文章对象粗分（5类）

| 类型 | 示例 | 优先读取 |
|------|------|----------|
| A. 单一产品 | 安盛盛利2、友邦环宇盈活 | 产品母页 |
| B. 单一保司 | 友邦、保诚、宏利 | 保司母页 |
| C. 多产品/横评 | 榜单、PK、现金流对比 | 横评页 |
| D. 概念/机制 | 分红实现率、保费融资 | 底层认知 |
| E. 趋势/流程 | 银行开户、市场趋势 | 通用问题框架 |

#### 创作框架路由表（按优先级排序）

如果一个选题同时命中多个框架，优先选择排位更靠前、对结论影响最大的那个，不混用多个框架。

1. **保费融资** → 保费融资写作框架
2. **分红实现率** → 分红实现率写作框架
3. **IUL** → 产品写作框架（暂无独立框架）
4. **单一产品** → 产品写作框架
5. **多产品/榜单** → 产品横评写作框架
6. **保司/银行** → 保司写作框架
7. **条款/费用/隐藏规则** → 条款黑盒写作框架
8. **外围问题** → 通用问题写作框架

**创作框架配合产品母页使用**：创作框架提供心理转折路径，产品母页提供事实弹药和表达动作。写作时两者配合：框架决定正文推进顺序，产品母页的「写作抓手层」决定怎么翻译术语、先抓什么误会、怎么避免写偏。

#### 公众号传播包装层

所有成稿默认叠加传播包装层，核心要求：

- **标题**：必须有冲突/反常识/风险感/利益感，不用中性报告题
- **开头**（前120-180字）：交代为什么很多人问、点出最容易误判的地方、给出不完全展开的判断
- **首屏**：先建立冲突再解释事实，不堆参数
- **图片弹药库调用**：写作时通读对应产品母页底部「图片弹药库」的插图建议，挑选与当前段落论点匹配的图片；插入时只用 `![](链接)` 格式，不保留插图建议文字
- **传播边界**：传播感可以更强，但事实必须更硬；不编造内幕

#### 破窗机制（结尾转化）

最高宗旨：**看完后，读者应该想咨询我们，而不是觉得"我懂了，可以找原有代理人下单了"。**

三个核心目标：
1. 让读者意识到还有个体化变量需要专业判断
2. 让读者产生"我以为我懂了，但实际上还有一层"的认知
3. 制造"意犹未尽"的感觉，话不能说完

四种实现方法：
- 抛出更高维度的变量
- 制造个体化复杂度
- 用案例复杂度制造咨询需求
- 保留专业判断空间

**结尾模板参考**（摘自01原文）：

模板一（抛出更高维度变量）：
> [总结核心观点] + 但实际上，还有一个关键变量我们没有公开讲：[暗示变量名称，但不展开] + 这个变量，不在任何公开资料里，需要结合你的具体情况判断。

模板二（制造个体化复杂度）：
> [总结核心观点] + 但这些都是基于"标准客户"的假设。如果你的情况涉及 [列举2-3个复杂场景]，那么整个逻辑会完全不同 + 这不是一篇文章能讲清楚的。

模板三（用案例复杂度制造咨询需求）：
> [总结核心观点] + 我见过很多客户，看完文章觉得自己懂了，结果在实际操作中遇到 [列举2-3个真实障碍] + 这些细节，只有真正做过的人才清楚。

**破窗成功标准**：
- **目标达成**：读者看完后产生"我需要进一步咨询"的动机，意识到"我的情况可能有特殊性"，感觉到"还有一些关键信息我没有掌握"
- **目标未达成**：读者觉得"我已经掌握了所有信息"，可以直接根据文章内容自己做决策，或觉得"我可以找原有代理人下单了"

#### 红线检查清单（10条）

1. 客群锚定：200万港币以上
2. 事实边界：不编造
3. 数据红线：可追溯
4. 语气边界：不低俗煽动
5. 传播边界：不用无来源风险制造点击
6. 转化方式：不写"回复XX领取资料"
7. 图片处理：不保留广告二维码
8. 框架隐形：不暴露内部结构
9. 读取边界：不为写作维护主库
10. 搜索词禁露：不暴露搜索动作

### 4.2 规则二：入库核心标准（02）

文件路径：`wiki/知识库规则/02-入库核心标准.md`

这是入库动作的**质量门槛**，定义了什么内容能进主库、什么只能进候选区、什么直接丢弃。

#### 入库目标

**建立可复用的写作资产。** 不是建档案馆，不是做百科，而是把 Raw 来源转化为能反复调用的写作弹药。

#### 客群红线

**硬性拒收**（连 Raw 层都不留，直接丢弃）：
- 扫盲/拼低价类：内地保险和香港保险的基础区别、打工人怎么买、最低门槛
- 羊毛/白嫖类：关注送礼、留邮箱发资料、扫码进群领礼包
- 泛投资失望情绪转化类：先骂股市公募，再忽悠买分红险

#### 来源分类

**主干类型**（建库核心）：
- 产品型：死磕单一产品的文章
- 保司型：专门扒保险公司底细的文章
- 横评型：多个产品横向 PK 的文章

**辅助类型**（外围弹药）：
- 政策监管型、流程工具型、法律税务传承型、案例故事型、邻近产品型

**营销/低增量来源**：只留在 Raw 层当底稿，绝对不准进正式 Wiki

#### 候选区规则

候选区是写作反哺暂存区，三个边界：
- 不属于 wiki/ 主库
- 不属于 Raw 来源层
- 不等于正式事实库

**进候选区的内容**：
- 库里还没有的全新实体、新概念、新框架
- 跨领域但可能有写作价值的素材
- 来源质量存疑、尚未验证的判断
- 不确定是否值得独立建页的探索性内容

**候选页必须包含的头部信息**：
```
状态：待确认 / 已晋升 / 已驳回
建议归属：创作框架 / 底层认知 / 案例故事 / 产品母页 / 其他
创建理由：为什么值得留下
触发来源：来自哪篇 Raw、哪次写作或哪页 Wiki
晋升条件：还需要什么证据或人工判断
风险提示：可能污染主库的地方
```

**晋升必须由用户明确拍板**，Agent 不能自行转正。晋升时需同步完成：
1. 文件移入正式目录
2. 更新 `wiki/index.md`（在对应的分类条目中追加新页面引用）
3. 追加 `wiki/logs.md`（记录晋升操作和原因）

**驳回时**：
1. 删除该候选文件
2. 追加 `wiki/logs.md`
3. 不长期囤积废弃候选页

**候选页命名**：不用加「候选-」前缀，直接用未来可能转正的名字。

#### 入库核心红线

- **不编造**：不编造产品缺陷、内幕、截图、市场传闻、确定性收益
- **不污染主库**：行业通病不能伪装成某个产品的独家硬伤
- **不把行业通病当专属缺陷**："分红不保证"、"提前退保会亏钱"是全行业的毛病，只有在这个产品在这一项上明显比同行差时，才配叫专属缺陷

#### 顶层结构铁律

Agent 不准自行新建 wiki/ 顶层目录，不准擅自重构目录，不准绕过用户修改主库边界。`wiki/index.md` 和 `wiki/logs.md` 的修改必须由用户显式触发。

## 五、创作框架详解

创作框架是**心理转折路径**，不是具体文案模板。它记录的是：读者从什么状态出发、经过什么认知转折、最后产生什么咨询动机。

### 5.1 产品写作框架

**适用**：单一产品测评、爆款产品、朋友推荐、代理人热推、计划书收益漂亮但需要买前判断。

**核心路径**：
```
[客户为什么会问] → [产品为什么吸引人] → [优点短写] → [风险重写] → [分人群判断] → [买前审方案]
```

**作者语气要求**：
- 整体以口语化表达为主，轻松但不失专业感
- 不要有夸张言辞（爆炸、炸裂、UC震惊体）
- 从"产品分析顾问"蜕变成"你身边懂行的闺蜜/兄弟"
- 多用"我都已经被客户问麻了"、"私下交流"、"不敢多评价"这类口头禅

**风险揭露口径**：
- 可以写："这个地方很多人不会看"、"公开场合不方便展开太细，但买前一定要核"
- 必须落到具体变量：保证现金价值托底、非保证分红占比、回本期匹配、提领后账户剩余价值
- 禁止：不编造截图内幕、不写"千万别买""无脑冲"

**分人群判断**（至少3类）：
- 适合：资金能长期放、现金流稳定、目标是传承/长期增值
- 谨慎：被高收益吸引但还没核保证部分的人
- 不适合：短期要用钱、特别在意保证确定性的人

**结尾两条路径**：
- 路径A：把问题留在读者自己的资金期限、提领节奏和家庭安排上
- 路径B：调用需求反查表，引导"你可能应该看的不是这款"

### 5.2 保费融资写作框架

**适用**：保费融资、融资保单、杠杆寿、贷款套利、年化10%以上高收益方案。

**核心路径**：
```
[承认高收益吸引力] → [拆变量联动] → [做压力测试] → [划定适配门槛]
```

**必须拆的变量**：
- 贷款利率：当前利率、浮动机制、未来上行风险
- 分红假设：非保证部分占比、实现率波动
- 现金流：客户是否能持续供款、还息、补仓
- 汇率：保单币种、贷款币种和家庭收入币种是否错配
- 回本期：融资后真实回本时间是否被拉长
- 退场机制：提前退保、降杠杆、还款时怎么处理

**压力测试写法**：把变量往坏的方向推一次，不要只说"有风险"。

**适配门槛**：必须写清楚什么样的客户才适合做融资保单（资产规模、流动性、风险承受、银行关系）。

### 5.3 分红实现率写作框架

**适用**：分红实现率、履约率、过往派息、老保单记录、新产品演示可信度。

**核心路径**：
```
[承认当前数据吸引力] → [指出单点数据失效] → [追溯长期履约] → [回到持有周期判断]
```

**关键判断维度**：
- 不能只看当年实现率，要看5年/10年趋势
- 要区分"总现金价值比率"和"分红实现率"的口径差异
- 新产品没有历史记录时，要看保司整体履约能力

### 5.4 保司写作框架

**适用**：友邦、保诚、宏利、安盛、中银、国寿、太平、万通、周大福人寿等保司和银行系保险评价。

**核心路径**：
```
[承认品牌安全感] → [拆成多维能力] → [同步讲优势和边界] → [回到客户目标]
```

保司文章不能只写"谁最大、谁历史最久"，而是要帮客户判断：这家机构的能力，是否匹配自己的目标。

**推荐评价维度**（至少选4-6个）：
- 历史和市场地位
- 资产实力和偿付能力
- 香港市场市占率和客户基础
- 分红履约和长期派息记录
- 代表产品竞争力
- 服务便利性和渠道覆盖

**必须同时写**：正向标签（评级、偿付、历史）和反向标签（增速、竞品对比、产品短板）。

### 5.5 产品横评写作框架

**适用**：储蓄险榜单、现金流榜、保证榜、提领榜、传承榜、多个产品不知道怎么选。

**核心路径**：
```
[承认选择混乱] → [建立分类维度] → [逐类给出代表产品] → [回到个人目标重排]
```

**必须先定维度**（不要直接给"最值得买"）：
- 收益维度：长期IRR、预期回报
- 保证维度：保证回本期、确定性
- 现金流维度：提领灵活度、派息方式
- 传承维度：身故赔偿、保单分拆
- 流动性维度：早期退保损失
- 保司维度：分红履约、历史记录

### 5.6 条款黑盒写作框架

**适用**：提领顺序、费用结构、货币转换限制、核保规则、合同细则、银行风控、销售说法和执行规则不一致。

**核心路径**：
```
[指出看不见的规则] → [拆销售说法和执行细则] → [定位具体影响] → [引导系统核验]
```

黑盒文章不能空喊"水深"。必须指出具体规则在哪里、怎么影响客户、客户应该核验什么。

**核心价值**：揭示"代理人可能连他自己都不知道"的行业真相，建立信息差优势。

### 5.7 通用问题写作框架

**适用**：银行开户、体检报告、医疗险、夫妻财产分割、离婚资产安排、市场趋势、优惠资讯和其他非标准选题。

**核心路径**：
```
[直接回答标题问题] → [承认表面合理] → [放回真实场景] → [区分不同人群] → [留下个体判断入口]
```

通用问题框架不是泛科普模板，而是把一个看似分散的问题，放回高净值家庭的资产安排、风险边界和真实决策里。

## 六、心理认知积木库

文件路径：`wiki/创作框架/心理认知积木库.md`

心理认知积木是用于激发脑部推演的**心法**，而非需要填空默写的招式。**严禁在最终输出的文章里暴露积木名称**。

### 认知唤醒类（打破既有认知，制造冲突感）

**品牌安全感的塌陷**
- 应用场景：针对只认大品牌的客户
- 心理钩子：点破"大品牌资产雄厚"与"产品分红实现率波动"之间的不平衡
- 叙事原石：调取老保单实现率数据、异常值数据

**静态演示的幻觉**
- 应用场景：针对只盯着计划书大数字的客户
- 心理钩子：揭示漂亮演示是以牺牲"前期保证性"和"流动性"为代价的
- 表达方式：把功能从结果承诺改写成能力证明；揭露功能的使用门槛和时点依赖

**从众效应的阻断**
- 应用场景：爆款产品测评、跟风现象
- 心理钩子：承认"卖得火"是事实，但引入"幸存者偏差"
- 表达方式：先承认热度，再指出理解表面化；借热度开场，立刻切到重审维度

### 理性分析类（提供决策框架，建立信息差优势）

**对比效应**
- 应用场景：产品横评
- 心理钩子：不要孤立看一个数据，通过坐标轴位移让原本"很香"的数字显得"风险巨大"
- 表达方式：多时间轴并置；前中后期分段对比，揭示反转节点

**锚定效应**
- 应用场景：收益拆解
- 心理钩子：将分红险收益锚定在"抗通胀"和"长期传承"上，而非"短期理财"

**信息黑盒效应**
- 应用场景：深度科普、财报拆解
- 心理钩子：告诉读者"有一个行业真相，你的代理人可能连他自己都不知道"

**智力权衡的透明**
- 应用场景：需要展示希希在两个选择之间如何取舍时

### 行为驱动类（触发咨询动机）

**损失规避**
- 应用场景：需要强调回本期、锁定期、保证不足、流动性损失时

**确定性偏差**
- 应用场景：涉及养老、教育金、刚性支出、家庭现金流时

**破窗机制与闭环剥夺**
- 应用场景：结尾需要保留个体审计入口、避免给过度确定答案时

## 九、底层认知体系

底层认知是**跨产品、跨保司、跨场景复用的底层判断**，不是产品母页或保司母页的替代品。

### 9.1 当前底层认知页清单

**基础认知**：
- **港险基础认知与计划书阅读**：港险总纲、计划书阅读、保证/非保证收益、币种与汇率决策
- **跨境投保与保单架构**：赴港投保流程、保单角色结构（投保人/被保人/受益人/后备持有人）、资产隔离
- **CRS与香港保险的税务边界**：CRS申报机制、税务居民身份、保单分红与理赔的税务处理、CRS 2.0变化

**产品判断工具**：
- **分红实现率判断模型**（样板页）：分红实现率、总现金价值比率、保证与非保证之间的判断顺序；英式/美式/混合分红机制解析
- **产品形态谱系**：IUL、保费融资、万用寿险、杠杆寿等形态级概念的路由判断
- **高价值缺陷与漏洞**：跨产品、跨场景复用的高价值反向判断、行业漏洞和销售话术漏洞
- **港险产品-用户需求反查表**：扁平数据库表，按产品名列出适用人群和希希评论，供Agent在文章结尾调用做破窗引导

**选题工具**：
- **搜索意图与高价值词根映射模型**：SEO、微信搜一搜、GEO问题背后的交易距离、词根价值和选题承接模型

### 9.2 分红实现率判断模型核心内容

这是最常被调用的底层认知页，核心判断：

**分红类型**：
- **英式分红（复归红利）**：保险公司赚了钱分保额，形成保额复利。账面价值增长快，但钱不是现金，想取出来有折现率。典型产品：安盛盛利2、保诚信守明天、宏利宏挚传承
- **美式分红（现金红利）**：保险公司赚了钱直接分现金，宣布即确定。落袋为安，提取灵活。典型产品：部分中资保司产品
- **混合模式**：市场进化后的混合设计

**判断顺序**：
1. 先看保证现金价值（合同内确定部分，是抗波动底座）
2. 再看总现金价值比率（实际总现金价值与演示总现金价值的关系）
3. 最后看分红实现率（历史兑现数据，不是未来承诺）

**核心警示**：分红实现率有参考价值，但不能单独决定产品好坏。不能只看当年实现率，要看5年/10年趋势。

### 9.3 港险产品-用户需求反查表使用方法

这张表只有一个目标：**让Agent在文章结尾能直接调出1-2个差异化产品，制造"意犹未尽"。**

破窗三步法：
1. 定位当前产品的核心特征（如：鑫安逸=30年固收锁息）
2. 从表里找差异化最大的1-2个产品
3. 用"如果你的需求是X，那你应该看Y"的句式收束

## 十、案例故事体系

案例故事是**转化核武器**，地位等同于产品页。入库标准不看"有没有故事"，而看它能不能长期反复服务创作、客户筛选、风险解释和保险动作转化。

### 10.1 当前正式案例（S层）

**资产配置选择题**：
- **老王300万闲钱：提前还贷还是配置港险**：40岁中产，手里300万闲钱、房贷剩200万，利率4.2%。核心价值：把"二选一"改写成资产与负债的结构题。铁律：不能写成"港险一定比还贷好"，而是"还一部分、配一部分"的Plan C

**复杂家庭传承题**：
- **73岁父亲给3岁小儿子配置港险：不赴港传承安排**：73岁父亲想给3岁小儿子配置港险，但本人不便赴港，需通过成年大儿子、后备持有人和保单暂托人设计过渡路径

### 13.3 成稿质量标准

**标题**：必须包含产品名/保司名/关键词，有冲突/反常识/风险感/利益感，不用中性报告题。

**开头**（前120-180字）：必须完成至少两件事：
- 交代为什么最近很多人问
- 点出最容易误判的地方
- 给出一句不完全展开的判断
- 把读者代入买前焦虑

**正文**：
- 优势短写（2-4个核心优点），风险重点写
- 至少出现1-2处"读者以为 vs 实际上"的认知反转
- 每夸一个卖点，立刻补一个代价、边界或翻转条件
- 所有"最强""最好""第一"必须补口径

**结尾**：必须使用破窗机制，让读者产生"意犹未尽"的感觉。

**排版**：
- 使用Markdown小标题分层
- 重点信息必须加粗
- 不输出mermaid、HTML复杂组件
- 表格只服务判断，不为了显得完整而堆信息

### 15.3 常见错误与避坑

**错误1：先全库扫描再写作**
正确做法：先判断对象类型，再按路由表选框架，再读取对应母页，不要一上来就全库扫描。

**错误2：把行业通病当产品专属缺陷**
"分红不保证"、"提前退保会亏钱"是全行业的毛病，不能硬塞给某个单品当"专属缺陷"。

**错误3：写作时维护主库**
写作过程中不得改写wiki/主库、更新索引日志或整理Raw来源，除非用户明确要求。

**错误4：暴露内部结构**
最终文章不得暴露"创作框架、心理积木、路由表、写作协议、传播包装层"等内部结构。

**错误5：结尾给完整答案**
结尾必须使用破窗机制，让读者意识到"还有下一步判断"，不要把所有信息都说完。

**错误6：调用外部搜索补充资料**
写作任务只能使用知识库内已有内容，不得调用外部websearch工具补充资料。若知识库缺失关键资料，应停止写作并告知用户。

**错误7：自行新建wiki/顶层目录**
Agent不准自行新建wiki/顶层目录，不准擅自重构目录。

### 15.4 写作任务最小执行口诀

> 先按业务词选框架，再选心理积木，再选事实来源，最后按框架写作。
> 01负责路由、事实来源、公众号传播包装、统一输出要求和红线检查；具体正文推进交给被选中的创作框架。
> **结尾必须使用破窗机制，让读者产生"意犹未尽"的感觉。**

## 十七、知识库治理原则总结

### 17.1 三层防熵增机制

**第一层：入库门槛**
- 客群红线：200万港币以上，低客单内容直接丢弃
- 来源质量：营销/低增量来源只进Raw层，不进正式Wiki
- 缺陷标准：行业通病不能伪装成专属缺陷

**第二层：候选区缓冲**
- 新判断、新框架、灰区素材先进候选区
- Agent可以建议转正，但不能自行转正
- 候选页必须写清楚状态、建议归属、创建理由、晋升条件

**第三层：主库边界**
- Agent不准自行新建wiki/顶层目录
- wiki/index.md和wiki/logs.md的修改必须由用户显式触发
- 写作任务不得触发主库维护动作

### 17.2 写作与入库的分离原则

写作和入库是两个完全独立的工作流：
- **写作**：默认任务，只读wiki/主库，不改主库
- **入库**：必须由用户显式触发，按02-入库核心标准执行

两者不能混用。写作过程中发现的新判断，先写入候选区，不直接转正。

### 17.3 事实来源优先级

```
用户提供的原文/计划书/截图/条款
    ↓ 优先级最高
知识库wiki/主库（产品母页/保司母页/底层认知）
    ↓
Raw来源层（用户明确点名时才读取）
    ↓
外部搜索（写作任务禁止使用，入库任务才可使用）
```

### 17.5 规则演化与历史存档

知识库规则经历过一次重大重构。当前生效的规则只有两份：
- `01-Agent创作执行流程.md`（写作调度）
- `02-入库核心标准.md`（入库质量门槛）

旧版规则（01-候选区总则、02-入库与来源管理、03-产品与案例页面标准、04-创作框架与写作资产、05-质量控制与风险治理、06-选题与搜索意图管理、08-入库与写作核心标准）已归入 `wiki/知识库规则/历史存档/` 目录，仅供人类管理员复盘参考。

此外，`99-历史防御型防线与废弃约束存档.md` 记录了曾经使用但已废弃的微观写作约束（如强制物理动作描写起笔、词汇禁令、强制智力纠结等）。这些约束因导致模型产生"机械合规感"和"词汇沙拉"而被废弃。**严禁任何AI Agent在正常写作流程中读取或执行该文件内的约束。**

**选题价值按交易距离排序**（从高到低）：
1. 产品+动作/优惠/停售（最高价值）
2. 产品+风险/负面/细节
3. 产品+对比
4. 产品名词
5. 人群场景词
6. 负面怀疑词
7. 品类词（最低价值）

- ❌ 错误："你搜了分红实现率，一定是想知道..."（暴露搜索动作）
- ✅ 正确："判断一个新产品的稳定性，盯着短期实现率看其实是个逻辑误区"（提供判断）
