# 补图大规模模型对比测试 Plan

测试时间：2026-06-01
执行者：Opus（主），调度 haiku / sonnet / opus 三种 subagent
目的：① 找出 image-hunter subagent 在不同模型下的缺陷；② 顺带把 0 图 wiki 页面补上。

## 实验设计

变量隔离：同一页面、同一 prompt，只换 model，做三模型头对头。
数据丰度梯度（标题命中文章数）：万通25 > 国寿17 > 富卫16 > 周大福14 > 太平洋7 > 忠意2。
关键假设：数据越稀缺，模型差异越大（划水/编造/放弃最易暴露）。

### A. 头对头区（同页 ×3 模型，直接对比）
- **富卫**（中等丰度 16篇）：haiku + sonnet + opus
- **忠意人寿**（极稀缺 2篇）：haiku + sonnet + opus ← 压力测试核心

### B. 覆盖区（分散 ×1 模型，测广度 + 补图）
- 万通（富 25篇）→ opus
- 中国人寿海外（富 17篇）→ haiku
- 周大福人寿（中 14篇）→ sonnet
- 太平洋人寿（稀 7篇）→ haiku
- 新加坡_宏利Signature Legacy Harvest（产品页）→ sonnet

共 7 个唯一页面，11 次 agent 运行。

## 执行波次（每波 3-4 个并行，同波不撞同一页面，避免 OCR 写冲突）
- Wave 1：富卫-haiku ｜ 忠意-sonnet ｜ 万通-opus
- Wave 2：富卫-sonnet ｜ 忠意-opus ｜ 中国人寿-haiku
- Wave 3：富卫-opus ｜ 忠意-haiku ｜ 周大福人寿-sonnet
- Wave 4：太平洋-haiku ｜ 新加坡宏利-sonnet

## 8 维缺陷评分框架（每次运行打分）
1. 难点清单质量：是否真扫页面、列出真实理解难点
2. 来源消费完整性：是否真读每个来源，还是只 grep 文件名
3. 门禁3判断：是否正确决定要不要微信搜索
4. 链接精度：URL 是否真实存在（编造/抄错率）
5. 凑数倾向：是否堆低价值单数字图
6. 诚实度：缺口如实标注 vs 编造覆盖
7. 远程URL认知：是否误判"远程图不可用"（haiku v1 旧 bug）
8. 数量合理性：图数是否由难点驱动 vs 拍脑袋

## 产出
- 每页：精选插入（我做终筛 + 链接校验）
- 最终：缺陷汇总报告（不改 agent 定义，交用户做价值判断）
- 记录文件：候选区/补图测试/结果记录.md
