# 全域内容资产、搜索观测与原系统保真迁移架构方案 v3.3

> **状态**：新的架构事实基准。
> **替代**：`全域内容资产与观测架构方案_v3.2.md`。v3.2 仅保留为历史讨论记录，不再直接指导开发。
> **执行计划**：`全域内容工作台开发总计划_v2.md`。
> **验收矩阵**：`全域内容工作台验收矩阵_v2.md`（T001–T300）。
> **适用范围**：微信关键词监控、公众号监控、小红书关键词监控、GEOProMax、Wiki/母文章库、WritingMoney 的母文章铸造与批量成稿、旧写作与微信公众号发布，以及统一首页和必要的运维能力。
> **一句话原则**：**Demo 决定统一外壳，原系统决定业务岛屿，v3.3 决定数据底座；革新数据库，不重做已经成熟的产品。**

---

## 0. 本次修订解决什么问题

v3.2 对内容身份、发现、快照、指标、GEO、评论、任务和 Markdown 的语义划分仍然有效，但它没有把“数据统一”和“产品界面保真”之间的边界写清楚，导致执行时出现了错误方向：

- 把“统一入口”实现成“重新设计七套前端”；
- 把 Demo 当成所有页面的内部设计稿，而不是全局外壳和部分新模块的产品稿；
- 让业务页面直接围绕中央 Hub 重新组织，丢失原系统的信息密度和操作习惯；
- 把“七套运行系统”“七个适配器”和“前端一级入口”混为一谈；
- 把 WritingMoney 的母文章铸造与批量成稿压缩成一张宽任务表和一个页面；
- 把正常业务界面的完整迁移放在数据底座之后，导致接口完成而产品仍然偏离。

v3.3 的核心修正是：

1. 冻结三套事实源和优先级；
2. 增加业务岛屿兼容层；
3. 区分跨系统事实、模块运行状态和正文资产；
4. 保持 v3.2 的 14 张跨系统核心表，但不再要求它们承载所有页面状态；
5. 以完整业务纵切片替代“先统一数据、再重做页面”的迁移路线；
6. 将视觉和操作保真提升为 P0 级验收条件。

## 1. 三套事实源与冲突优先级

### 1.1 全局外壳事实源：Demo

`unified-content-platform-demo.html` 永久只读，决定：

- 左侧“内容工作台”全局栏；
- 导航分组、一级入口名称和排列；
- 顶部模块标题与全局动作区域；
- 浅灰全局画布、白色业务岛屿和岛屿边界；
- 首页的入口组织方式；
- 系统之间的跳转语义；
- 母文章铸造、批量成稿和发布中心在缺少成熟 Web 前端时的产品结构。

Demo 中已经明确：

> 每套系统原来的页面结构和使用习惯都保留，只统一入口、颜色和系统之间的跳转。

因此 Demo 不是“重新设计所有业务页面”的授权，而是宿主框架。

### 1.2 业务岛屿事实源：原系统前端

以下页面内部以原系统真实界面和源码为事实源：

- 微信关键词：原关键词监控系统；
- 小红书关键词：原小红书关键词监控；
- GEO 观察：GEOProMax 当前问题/引用源双视角；
- 公众号监控：原 Vue 工作台；
- 母文章 Wiki：原 Wiki Viewer；
- 写作与发布：旧写作/发布能力 + Demo 的可视化产品稿。

原系统决定：

- 页面分栏、页签和信息密度；
- 搜索、筛选、排序、分页和选择逻辑；
- 列表与详情之间的关系；
- 图表、快照、指标和解释文本；
- 按钮、弹窗、抽屉和进度反馈；
- 用户已经形成的操作顺序。

除非用户明确批准，迁移不得：

- 删除原有入口；
- 把多栏页面改成通用卡片列表；
- 把图表降级成数字摘要；
- 把操作改成只读；
- 把正常页面替换成空态、错误态或“待接入”提示；
- 用统一组件库改变关键操作位置。

### 1.3 数据事实源：v3.3

本文件决定：

- 跨系统统一身份和关系；
- 搜索快照、排名、指标和 GEO 引用的语义；
- 模块运行状态与跨系统事实的边界；
- Markdown、SQLite、原始批次和审计各自的职责；
- 数据迁移、双写、对账、备份和恢复；
- 兼容 API 与 Repository 的责任。

### 1.4 冲突优先级

```text
用户最新明确要求
  > Demo 的全局外壳与一级导航
    > 原系统业务岛屿的结构和操作
      > v3.3 数据契约
        > 具体技术实现便利性
```

技术实现不得反向要求产品删功能。

## 2. “系统”“入口”“模块”必须分开计数

### 2.1 七套运行系统

“七套系统”描述的是历史运行和数据来源边界，不等于七个前端页面：

1. 微信搜一搜关键词监控；
2. 公众号监控；
3. 小红书关键词监控；
4. GEOProMax；
5. Wiki/母文章库；
6. WritingMoney；
7. 旧写作与微信公众号发布。

### 2.2 九个工作台页面

统一首页加八个业务入口：

| 导航组 | 页面入口 | 产品事实源 |
|---|---|---|
| 主页 | 统一首页 | Demo |
| 选题发现 | 微信关键词 | 原微信系统 |
| 选题发现 | 小红书关键词 | 原小红书系统 |
| 选题发现 | GEO 观察 | 原 GEO 系统 |
| 内容采集 | 公众号监控 | 原公众号系统 |
| 内容资产 | 母文章 Wiki | 原 Wiki Viewer |
| 内容生产 | 母文章铸造 | WritingMoney + Demo |
| 内容生产 | 批量成稿 | WritingMoney + Demo |
| 内容发布 | 写作与发布 | 旧发布能力 + Demo |

### 2.3 运维页面不是主业务入口

数据治理、系统状态、备份恢复和审计可以存在，但默认进入：

- 首页的二级入口；
- 设置菜单；
- 受控运维 URL；
- 管理抽屉。

不得擅自占用 Demo 左侧主导航。Demo 中 `handoff` 与 `govern` 已标记为 retired/hidden，这一意图必须保留，除非用户另行调整。

## 4. 前端保真迁移架构

### 4.1 宿主只管理外壳

React/Vite 可以继续作为统一宿主，但只负责：

- 全局导航；
- 页面路由；
- 顶部标题；
- 模块生命周期；
- 统一错误边界；
- 跨模块跳转参数；
- 全局健康和入口统计。

宿主不得重新实现业务岛屿。

### 4.2 模块迁入方式

按原技术栈选择最小改动路径：

| 模块 | 原形态 | 迁入策略 |
|---|---|---|
| 微信关键词 | Flask 模板 + 原生 JS/CSS | 复制到 `workbench` 模块目录，保持 DOM/脚本，改 API base |
| 小红书关键词 | Flask 模板 + 原生 JS/CSS | 与微信相同，保持博主/笔记语义 |
| GEO | Python 生成 HTML + 原生 JS | 固化模块 HTML/CSS/JS，数据改由兼容 API 注入 |
| 公众号监控 | Vue | 作为独立模块构建并挂载到宿主岛屿 |
| Wiki | HTML + JS | 原样迁入，API 改写到安全 Repository |
| 母文章铸造 | WritingMoney HTML/JS + Demo | 以现有交互稿为产品基线，后端化状态 |
| 批量成稿 | WritingMoney batch mode + Demo | 独立一级入口，共用运行模型 |
| 发布 | 旧 Python 能力 + Demo 页面 | Demo 页面做前端，旧能力封装为安全命令 |

### 4.3 CSS 隔离

- 外壳 CSS 只控制外部布局；
- 业务岛屿保留原 CSS；
- 使用模块根节点、CSS 前缀或 Shadow DOM 避免污染；
- iframe 只允许作为短期迁移诊断工具，不能作为默认完成形态；
- 不允许全局 reset 破坏原表格、按钮、图表或弹窗。

### 4.4 页面尺寸

- 外壳固定提供可计算的岛屿高度；
- 模块内部保留自己的滚动区域；
- 禁止同时出现页面级和岛屿级无意义双滚动；
- 1440×1000 是 Demo 与现有审计截图的主基准；
- 同时验收 1280×720、1440×900、1920×1080。

## 5. 数据所有权与事实分层

### 5.1 五层数据

| 层 | 保存内容 | 谁可写 | 是否跨系统 |
|---|---|---|---|
| 原始证据层 | raw JSON、旧 SQLite、抓取目录、旧 Markdown | 原系统 | 否 |
| 模块运行层 | 分组、筛选设置、调度、任务、编辑锁、工作流状态 | 模块 Repository | 通常否 |
| 中央事实层 | 身份、发现、快照、排名、指标、关系、信号 | Hub Repository | 是 |
| 正文资产层 | 文章、AI 回答、母文章、成稿 Markdown | 资产 Repository | 是 |
| 审计恢复层 | batch、checkpoint、event、audit、backup、reconcile | 基础设施服务 | 是 |

### 5.2 原始证据层

- 永久保留不可再生历史；
- 迁移期间只读；
- 每批生成 manifest 和 hash；
- Hub 中通过 `source_ref` 定位；
- 不因新数据库上线而删除。

### 5.3 模块运行层

模块运行状态不强行压进 14 张核心表。例如：

- 微信/XHS 的关键词分组、备注、置顶、刷新策略和调度；
- MP 的账号开关、分类、抓取参数、模型选择和二维码流程；
- Wiki 的编辑锁、图片操作任务和 OCR 状态；
- WritingMoney 的项目、素材选择、方案、批次、队列和返工；
- 发布的账号可用性、队列、排期和尝试事件。

首版可与 Hub 共用一个 SQLite 文件，但必须用独立 Repository 和清晰表名前缀保持逻辑边界；后续可拆分为独立运行库。

### 5.4 中央事实层

Hub 只保存稳定、可共享、可分析、可追溯的事实。它不决定某个模块的页面布局，也不保存浏览器瞬时 UI 状态。

### 5.5 正文资产层

- Markdown 是正文事实载体；
- 文件保存使用原子替换和版本快照；
- 数据库保存稳定相对引用，不向 API 暴露绝对路径；
- 动态发布状态、实时排名和累计指标不得回写正文。

## 6. v3.3 的 14 张跨系统核心表

14 张核心表继续保持独立，不因前端保真修订而删除或合并：

| 领域 | 表 | 稳定语义 |
|---|---|---|
| 内容身份 | `contents` | 内容、回答、母文章和成稿的统一身份 |
| 创作者 | `creators` | 平台创作者和公众号身份 |
| 外部标识 | `content_identifiers` | namespace + external_id 映射 |
| 发现关系 | `content_discoveries` | 哪个系统通过什么方式发现内容 |
| 搜索快照 | `search_snapshots` | 平台、关键词、采集时间和快照特征 |
| 搜索命中 | `search_hits` | 快照内排名和原始展示事实 |
| 指标字典 | `metric_definitions` | 指标语义、单位、值类型和累计性 |
| 指标观测 | `metric_observations` | 主体在某时间点的指标值 |
| 评论 | `comments` | 评论当前身份与可见状态 |
| 评论事件 | `comment_events` | visible/missing/deleted 历史 |
| GEO 回答 | `geo_answers` | 问题、回答正文、时间和工具事实 |
| GEO 来源关系 | `geo_source_relations` | 回答到来源的类型、位次和锚文本 |
| 信号 | `signals` | 新收录、排名、阅读、评论和 GEO 异常 |
| 生产任务摘要 | `production_jobs` | 跨模块任务身份、状态和产物关联 |

### 6.1 核心表不承担什么

- 不存全部页面筛选条件；
- 不存完整 WritingMoney 项目 JSON；
- 不存 Cookie、Token、浏览器 profile；
- 不存 MP 登录秘密；
- 不存不可验证的综合热度；
- 不以标题相同自动合并内容；
- 不把公众号库存当微信搜索发现。

Cookie、Token、Profile 只以本机配置引用存在，不进入响应、日志和 Git。

## 8. API 分层

### 8.1 模块兼容 API

业务岛屿优先使用保持原语义的模块 API：

```text
/modules/wechat/api/...
/modules/xhs/api/...
/modules/mp/api/...
/modules/geo/api/...
/modules/wiki/api/...
/modules/writing/api/...
/modules/publishing/api/...
```

兼容 API 可以在不同迁移阶段切换实现：

```text
旧服务代理
  → 旧服务 + Hub 聚合
    → 新 Repository + Hub
      → 完全由新底座提供
```

前端不需要因切换数据源而重写。

### 8.2 跨系统 Hub API

用于统一首页、Agent、跨系统搜索和治理：

- `/api/v1/overview`
- `/api/v1/contents`
- `/api/v1/signals`
- `/api/v1/lineage`
- `/api/v1/systems`
- `/api/v1/reconcile`

Hub API 不替代模块兼容 API。

### 8.3 Command API

刷新、采集、保存、写作、发布等写操作统一使用命令接口：

- 显式操作者；
- 幂等键；
- 确认门；
- 持久任务；
- 审计事件；
- 可重试错误；
- 真实回执。

任何 Toast 都不能单独代表成功。

## 9. 原系统前端复制与来源追踪

原始目录永久只读，新模块代码复制到 `workbench/`。每次复制必须生成 provenance 记录：

```yaml
module: wechat
source_root: /Users/.../wechat-ybxhyyh-top3/
source_files:
  - app/templates/monitor.html
  - app/static/...
source_commit_or_hash: ...
copied_at: ...
allowed_changes:
  - API base
  - 宿主挂载
  - CSS namespace
  - 安全修复
```

允许改：

- API 基地址；
- 绝对路径；
- 宿主挂载方式；
- CSS 作用域；
- 明确的安全缺陷；
- 已由用户批准的 UI 细节。

不允许未经批准改：

- 页面结构；
- 业务页签；
- 核心按钮；
- 主要指标；
- 列表与详情关系；
- 原操作顺序。

## 10. 模块间交接

跨模块只传递稳定信封，不直接读写对方私有表：

```json
{
  "artifact_id": "art_...",
  "artifact_type": "keyword_signal|mother_article|generated_draft|publish_receipt",
  "source_system": "wechat|geo|wiki|writing|publishing",
  "source_id": "...",
  "target_system": "...",
  "content_id": "...",
  "evidence_refs": [],
  "created_at": "...",
  "payload": {}
}
```

关键交接：

- 微信/XHS/GEO 信号 → 母文章铸造或批量成稿；
- Wiki 母文章版本 → WritingMoney；
- WritingMoney 成稿副本 → 发布；
- 发布回执 → Hub 和首页；
- 返工任务 → WritingMoney/Wiki。

交接失败不得修改来源系统唯一文件。

## 11. 迁移路线：界面不动，底层换血

每套系统都执行相同的六阶段迁移：

### 阶段 A：冻结

- 保存原页面截图；
- 提取功能清单；
- 保存原 API 契约；
- 记录原数据计数和关键样本；
- 计算源文件 hash。

### 阶段 B：迁入原页面

- 把原前端复制到业务岛屿；
- 保持旧 API；
- 先证明页面结构和操作不变；
- 此阶段不要求 Hub 接管。

### 阶段 C：兼容层代理

- 所有请求经过模块 BFF；
- BFF 仍可调用旧服务；
- 统一错误、超时、审计和确认门；
- 页面无感。

### 阶段 D：历史事实入 Hub

- 原始批次、SQLite、JSON、Markdown 增量导入；
- 建立身份和关系；
- 对账不一致时仍显示旧系统事实；
- 不用 Hub 数据覆盖旧系统。

### 阶段 E：逐接口切换

- 一个接口一个接口切换到 Repository；
- 每次切换前后做契约对比；
- 页面截图和操作结果必须相同；
- 写操作先双写，再核对。

### 阶段 F：接管与回退

- 连续对账通过；
- Hub/运行层成为新主读写路径；
- 旧服务保留回退开关；
- 原始历史永久保留。

## 12. 安全与不可逆动作

- 默认仅监听 `127.0.0.1:8799`；
- 所有 SQL 参数化；
- Markdown 渲染防 XSS；
- 文件访问防路径穿越和软链接逃逸；
- SQLite WAL + 进程级单写锁；
- Cookie、Token、API Key、Profile 不进入 Git、数据库响应和日志；
- GEO 付费刷新、真实 AI 调用、真实发布必须显式确认；
- 自动化测试只使用 dry-run、草稿或模拟 Provider；
- 原系统正常输出只能经用户显式动作产生；
- 备份恢复演练永不覆盖运行库。

## 14. 验收原则

完整验收以 `全域内容工作台验收矩阵_v2.md` 的 **T001–T300** 为唯一编号基准；旧 T001–T180 不再用于完成声明。

模块完成必须同时满足四类证据：

1. **结构保真**：原页面的主要区域、页签、分栏和信息密度存在；
2. **操作保真**：原按钮、筛选、排序、抽屉、弹窗和任务流程可用；
3. **数据正确**：真实数据、历史迁移、幂等、对账和来源可追溯；
4. **异常真实**：空态、错误态、离线态和阻断态不伪造成功。

其中“正常业务态”优先级最高：

> 页面只有空态和错误态，即使安全且诚实，也不能判定为业务模块完成。

浏览器验收必须逐项对照：

- 原系统截图；
- Demo 宿主截图；
- 新工作台实际截图；
- 操作前后 API/数据库证据。

## 15. v3.3 最终判断

v3.3 不推翻 v3.2 的跨系统事实模型，而是修复了它没有约束产品迁移方式的问题。

最终系统不是：

> 一个围绕中央数据库重新设计的通用管理后台。

而是：

> 一个由 Demo 提供统一入口、由原系统提供成熟业务体验、由兼容层保持操作契约、由 Hub 提供跨系统事实和恢复能力的本地内容工作台。

**外表保持熟悉，底层获得统一；页面不因数据库升级而退化，数据库也不因迁就旧代码而失去长期语义。**
