## 0. 执行规则与不可突破的边界

### 0.1 交付定义

本计划不是一次性的原型清单，而是可连续执行 15 小时以上的工程总计划。完成标准是：七套系统均有统一启动入口；中央 SQLite、适配器、前后端、报表、备份、对账、回滚、测试和浏览器验收链路均能在一台 Mac 上运行；旧系统仍可作为只读事实来源和回退通道。

### 0.2 只读与镜像红线

1. `source/` **只读**：不改、不移动、不重命名、不删除、不在其中写运行时文件。
2. `unified-content-platform-demo.html` **只读**：它是现有演示基线，不接入新系统运行时，不改页面内容。
3. 现有 v3.2 架构文档、AGENTS.md、截图、旧系统源目录均只读。
4. 新代码、数据库、raw、Markdown 资产、日志、缓存、备份、报告统一放入新应用目录；禁止向旧目录回写。
5. 完整镜像是永久保留的灾备事实副本：原始源码、JSON、SQLite、Markdown、媒体、配置、运行日志和每次批次 manifest 均按时间与哈希保存，不因迁移成功而删除。
6. 任何迁移、重跑、对账、回滚操作都必须可在审计日志中回答：谁、何时、对哪个批次、从何处读、写入何处、结果如何。

### 0.3 事实层原则

- Markdown 保存正文事实；SQLite 保存身份、关系、快照、指标、评论事件、GEO 关系和生产任务。
- `content_id` 是跨系统统一身份；WX 搜索、MP 库存、XHS、GEO 等发现渠道通过 `content_discoveries` 隔离。
- `source_ref` 必须能回到 raw 批次及 manifest；`payload_json` 是受控逃生舱，不得成为无 schema 的垃圾桶。
- Agent 默认只读；修复历史事实必须生成 correction job，由适配器重放原始证据。
- 任何派生信号均可重算，不得把信号当作不可追溯的原始事实。

## 3. 14 表、视图、适配器与数据契约

### 3.1 14 张核心表

1. `contents`：正文资产身份、标题、URL、作者、Markdown 路径和内容哈希。
2. `creators`：创作者稳定身份；粉丝数等动态指标不写在此表。
3. `content_identifiers`：平台稳定外部 ID 与 `content_id` 的映射。
4. `content_discoveries`：哪个系统、哪个渠道、何时发现内容；WX/MP 隔离核心。
5. `search_snapshots`：平台、关键词、采集时间、结果数、页面特征。
6. `search_hits`：快照中的排名、原始标题、原始 URL、可选内容身份。
7. `metric_definitions`：指标字典、单位、值类型、累计/瞬时/状态模式。
8. `metric_observations`：主体在某时刻的指标观测；不覆盖历史。
9. `comments`：评论稳定身份、正文和当前可见状态。
10. `comment_events`：评论 visible/missing/deleted/hidden 等状态事件。
11. `geo_answers`：独立 AI 回答身份、问题、回答哈希、工具链和回答 Markdown。
12. `geo_source_relations`：文本引用、搜索来源、图片引用的有序关系。
13. `signals`：可重算的异常与日报信号。
14. `production_jobs`：选题、成稿、发布等生产任务状态。

### 3.2 必须建立的标准视图

| 视图 | 用途 |
|---|---|
| `v_content_latest_metrics` | 每个主体、指标的最新观测 |
| `v_metric_recent_pairs` | 最近两次真实观测点，供增速计算 |
| `v_first_keyword_appearance` | 同平台、同关键词、内容首次出现 |
| `v_rank_changes` | 最近两次排名、上升、下降、掉榜 |
| `v_new_creators` | 首次出现的创作者及首次发现上下文 |
| `v_comment_anomalies` | 评论缺失、疑似隐藏、确认删除事件 |
| `v_daily_overview` | 每日信号数量、最高等级、未消费数 |
| `v_geo_citation_timeline` | GEO 回答引用、位置、关系类型回放 |
| `v_reconcile_open_issues` | 未解决对账问题与 source_ref |
| `v_production_job_board` | 生产任务状态、输入信号、输出内容 |

视图必须是 SQL 契约：有版本、说明、示例查询和快照测试；前端不得复制另一套业务计算逻辑。

### 3.3 统一适配器契约

每个适配器必须实现：

```text

persist_raw(raw_batch, manifest) -> SourceRefSet
emit(events) -> AppendResult
reconcile(batch_id) -> ReconcileResult

```

统一事件类型：

```text

CreatorUpsert
IdentifierUpsert
DiscoveryAppend
SnapshotAppend
SearchHitAppend
MetricObservationAppend
CommentUpsert

GeoAnswerAppend
GeoRelationAppend
```

适配器禁止直接写跨平台热度、建议分数或无法追溯的聚合值；跨平台信号只能由 `signals` 计算器生成并记录 `model_version`。

## 4. 前后端与运行入口

### 4.1 后端边界

建议使用 Python 3.11+、FastAPI、SQLite、Pydantic、Typer、pytest；具体依赖必须锁版本并写入 `pyproject.toml`。后端分为：

- `ingestion`：读取旧系统、落 raw、生成 manifest、规范化事件。
- `db`：迁移、事务、仓储、只读查询、视图。
- `identity`：外部 ID、URL、正文哈希、低置信候选和人工合并队列。
- `signals`：按业务时区计算新收录、排名变动、阅读暴增、新账号、评论异常、GEO 引用。
- `reconcile`：文件、身份、事实连续性、引用完整性四类对账。
- `backup`：SQLite Online Backup API、完整镜像、恢复演练。
- `api`：健康检查、只读查询、信号、任务、对账、运行控制和回滚状态。

### 4.2 前端首版页面

1. **总览**：今日数据批次、系统健康、总览信号、未消费数。
2. **系统页**：七套系统状态、最后成功批次、快照数、错误数、重跑入口。
3. **搜索观测**：平台/关键词/日期筛选，结果排名时间线，WX 与 MP 明确分区。
4. **内容详情**：正文 Markdown、稳定身份、发现渠道、指标时间线、评论事件、GEO 引用。
5. **信号中心**：按类型、严重度、日期、状态筛选，标记消费或创建生产任务。
6. **生产看板**：WritingMoney 与发布任务状态、输入信号、输出文章、失败重试。
7. **对账与备份**：开放问题、source_ref、manifest、备份时间、恢复点、只读/双写/回滚开关。

前端只调用统一 API；原始 Markdown 查看使用安全的本地文件服务或受控内容接口，禁止拼接任意路径。

### 4.3 CLI 与服务入口


默认访问：

- 新工作台：`http://127.0.0.1:8788/`
- 健康检查：`http://127.0.0.1:8788/health`
- API 文档：仅本机调试开启，生产默认关闭或只读保护。
- 现有 Demo：直接打开 `unified-content-platform-demo.html`，只做基线验证，不被新系统改写。

### 4.4 Mac 单机调度

- 开发阶段使用 `launchd` 或一个可观测的本地 scheduler，不使用隐式后台进程。
- 单写者锁：每个 ingest/reconcile/backup job 先获取 `data/locks/*.lock`。
- 每日任务顺序：镜像 → 采集 → raw manifest → 入库 → signals → 报表 → 对账 → backup。
- 任何任务失败都不得阻塞其他系统的只读访问；失败批次可单独重跑。
- 所有时间显式传 `report_date` 与 `timezone=Asia/Shanghai`，库内保存带时区 ISO 8601。

## 5. 完整镜像、迁移、双写与回滚

### 5.1 永久完整镜像

首次上线前建立 `mirror-0001`，以后每次采集、迁移、发布都生成不可变快照：

```text
data/mirrors/<mirror_id>/
├── manifest.json
├── checksums.sha256
├── source/                       # source/ 的只读完整副本
├── legacy/                      # 七套旧系统可读文件/数据库/配置副本
├── demo-baseline/               # Demo 只读基线与截图
├── content-workbench/           # 新系统指定版本
└── restore-instructions.md
```

镜像规则：先完成临时目录写入，再 `fsync`、校验、原子改名；镜像内不保存密钥；manifest 记录文件数、字节数、哈希、来源时间、Git commit、schema 版本和工具版本。永久镜像不参与普通清理策略，删除需要人工双确认且首版默认禁止。

### 5.2 迁移阶段

| 阶段 | 目标 | 旧系统角色 | 退出条件 |
|---|---|---|---|
| M0 契约冻结 | ID、时间、命名空间、指标、schema、权限 | 运行事实源 | 契约评审通过 |
| M1 只读索引 | 从 JSON/SQLite/Markdown 导入 hub | 运行保障与核验 | 14 表可查询、raw 可追溯 |
| M2 适配器旁路 | 每个新批次写 raw + hub，不改旧输出 | 主事实源仍是旧系统 | 连续 7 天批次可对账 |
| M3 双写对账 | 采集器同时写旧结果和 hub | 旧系统继续运行 | 连续 14 天核心计数和关键指标一致 |
| M4 查询切换 | Agent/前端默认 hub，旧系统只读回退 | 回退/证据源 | 连续 30 天无阻断级差异 |
| M5 派生层冻结 | 停止旧派生报表作为事实源 | 永久归档 | 恢复演练通过，保留策略生效 |

### 5.3 双写设计

双写不是两个数据库互相覆盖，而是同一批 `batch_id` 由同一份 raw manifest 派生两份结果：

1. raw 落盘并封存；
2. 旧适配器生成 legacy 输出；
3. 新适配器从相同 raw 生成统一事件；
4. 两边分别提交；
5. 对账器按 batch_id、系统、日期比较：
   - 快照数、命中数、内容数、创作者数；
   - 外部 ID 覆盖率；
   - 关键指标总数与样本值；
   - 评论可见状态变化；
   - GEO 回答 hash 与引用关系；
6. 差异写报告和 `signals`，不人工静默修库。

### 5.4 回滚等级

| 等级 | 场景 | 动作 |
|---|---|---|
| R0 查询回退 | hub API/视图异常 | 前端切换只读 legacy 查询，禁止写入旧系统 |
| R1 适配器回退 | 新适配器输出异常 | 停止新适配器，保留 raw，恢复旧采集输出 |
| R2 数据批次回退 | 单批次污染 | 标记 batch invalid，从上一份备份恢复到隔离库，重放 raw |
| R3 schema 回退 | 迁移引入阻断问题 | 停服、备份当前库、执行反向迁移或恢复同版本库 |
| R4 发布回退 | 新版本整体不稳定 | 恢复上一个 release、数据库备份、配置和静态资源 |

回滚必须先 dry-run，再执行；不能 force push、不能覆盖永久镜像、不能删除原始批次。每次回滚生成事件、影响范围、恢复点和验收记录。

## 7. 依赖、风险与应对

| 风险 | 触发信号 | 预防 | 处置 |
|---|---|---|---|
| 旧系统路径不可读 | permission/路径变化 | 启动时探测、能力报告 | 该系统标 unavailable，不阻塞其他系统；等待修复后重跑 |
| 旧 JSON/SQLite 结构漂移 | schema/hash 变化 | adapter contract fixture、版本探测 | raw 封存，适配器失败隔离，不污染 hub |
| WX/MP 数据混淆 | discovery channel 错误 | 强制枚举和隔离查询测试 | 阻断发布，按 batch 回滚并重放 |
| 同文多身份误合并 | hash/URL 冲突 | 多级身份解析、低置信人工队列 | 禁止自动合并，保留映射和证据 |
| SQLite 写锁 | busy timeout、锁等待 | 单写者、队列、WAL、事务小批提交 | 退避重试，超时告警，读服务继续 |
| 采集成功但数据为空 | snapshot=0/字段骤降 | batch completeness 规则 | 标记 partial/failed，不产生掉榜信号 |
| 评论消失被误判删除 | 采集局部失败 | missing 与 deleted_confirmed 分离 | 需要连续证据或二次采集确认 |
| GEO 回答 hash 变化 | 同问题不同回答 | answer_hash、captured_at、mode | 新回答追加，不覆盖旧回答 |
| Markdown 被手工移动 | md_path 不存在 | content_id 与文件哈希对账 | 生成 reindex/repair job，不改身份 |
| 完整镜像过大 | 磁盘低于阈值 | 容量告警、压缩只做副本 | 扩盘/转离线介质；永久策略不自动删除 |
| Mac 睡眠/断电 | job 中断 | checkpoint、原子批次、在线备份 | 恢复后按 batch 状态重跑 |
| 前后端字段漂移 | API schema 不一致 | OpenAPI snapshot/契约测试 | 版本化 API，不让前端猜字段 |
| 任意文件读取 | path traversal | allowlist、realpath、根目录校验 | 立即关闭接口并审计日志 |
| 外部内容含恶意脚本 | HTML/Markdown 注入 | 前端 sanitize、CSP、纯文本默认 | 隔离渲染、清理缓存、回滚版本 |
| 依赖安装失败 | 无网络/版本冲突 | lockfile、离线 wheel 缓存 | 使用已锁版本；禁止临时升级 |
| 任务重复执行 | duplicate batch/job | idempotency key、唯一约束 | 返回已存在结果，不能重复写事实 |

## 9. 主代理最终浏览器验收脚本

主代理不得只看接口返回，必须实际打开浏览器并完成以下顺序：

1. 启动 `./scripts/start.sh`，打开 `http://127.0.0.1:8788/`。
2. 查看首页总览，确认日期、批次、七系统健康状态和信号数量。
3. 逐一点击七系统卡片，确认每个系统有状态、最后批次、错误/重跑入口。
4. 进入 WX 搜索页，筛选关键词，打开一条命中，检查排名时间线、正文、source_ref。
5. 返回并进入 MP 页，确认同一内容可以有 MP discovery，但 WX 选题筛选不出现纯 MP 文章。
6. 进入 XHS 页，打开笔记和创作者指标时间线。
7. 进入 GEO 页，展开回答、文本引用、搜索引用和图片引用位置。
8. 进入 Wiki 内容详情，确认 Markdown 正文可读、编辑/重建索引状态可见。
9. 从信号中心创建一个 WritingMoney production job，检查输入信号和输出状态。
10. 打开发布看板，检查成功、失败、重试和回执路径。
11. 打开对账页，查看至少一条已解决或开放问题，确认 source_ref 受控可达。
12. 打开备份页，确认最近备份 hash、镜像 ID 和恢复演练结果。
13. 模拟 API/数据库异常或使用故障 fixture，确认前端显示错误态而非白屏。
14. 执行浏览器刷新、深链接、返回、筛选清空、空态和错误重试。
15. 直接打开现有 `unified-content-platform-demo.html` 做基线检查，确认页面仍可用且 hash 未变。
16. 保存浏览器截图、控制台日志、网络错误、release verify 输出和最终验收结论。

浏览器验收判定：

- 核心路径任何白屏、数据混区、详情打不开、误写只读源、回滚入口失效，均为阻断级失败。
- 字段展示小问题可以记录为非阻断问题，但必须有 issue 编号、责任模块和修复时限。
- 主代理确认“通过”前，不得宣布首版完成。

## 11. 版本演进与长期维护

- v1 首版只允许增加字段/适配器能力，不静默改变既有字段语义。
- 每次 schema、指标、视图或 API 变化都增加 migration、CHANGELOG、契约 fixture 和回滚说明。
- 每周检查镜像完整性、备份可恢复性、未解决对账问题和磁盘容量。
- 每月执行一次全量恢复演练和七系统端到端抽样。
- 连续 30 天对账稳定后，才可停止旧派生层；旧原始数据与完整镜像仍永久保留。
- 只有在 SQLite 写锁、规模、权限或高可用需求满足 v3.2 的 PostgreSQL 迁移条件时，才启动数据库迁移评估；迁移不得改变 `content_id`、Markdown 路径、source_ref 和适配器事件契约。

> **最终形态**：人能读 Markdown，Agent 能查视图，脚本能写事实，旧系统能回退，raw 能追证据，镜像能恢复，浏览器能验收；七套系统首版全部可运行，但没有任何一个新模块以修改旧系统或现有 Demo 为代价。
