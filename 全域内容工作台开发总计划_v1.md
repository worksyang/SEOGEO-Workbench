> 总负责人：Codex 主代理

> 协作边界：总计划、核心架构、数据契约、合并与最终验收均由 Codex 主代理亲自负责；Luna/子代理只执行边界明确的探索或局部实现

## 0. 项目结论

本项目不把七套旧系统粗暴拼成一个单体，也不继续让前端依赖各自巨大的 JSON。最终采用：

1. **一个新工作台应用**：FastAPI 后端、React + TypeScript 前端、同源部署。
2. **一个中央事实库**：`content_hub.sqlite`，严格实现 v3.2 的 14 张核心表。
3. **七个适配器**：旧系统保持原样运行，新系统通过 HTTP、SQLite、JSON、Markdown 和受控子进程接入。
4. **两个内容层**：
   - `_local_full_backup/`：完整历史恢复层，永不上传 GitHub。
   - `asset_store/`：去重后的统一可读 Markdown 资产层。
5. **三条运行通道**：
   - 历史导入：从旧文件与数据库增量灌入 Hub。
   - 实时操作：统一前端调用适配器执行刷新、采集、编辑、写作和发布。
   - 对账恢复：旧系统与 Hub 每日核对，可随时重放。
6. **安全默认值**：
   - 搜索刷新、GEO 刷新、AI 写作和发布都必须显式触发。
   - 真发布默认关闭；浏览器验收只使用 dry-run、草稿箱或模拟回执，不自动群发。
7. **验收定义**：不是声称“绝无 Bug”，而是做到 **180 项测试全部通过、零已知 P0/P1、数据可回滚、七系统真实链路均有证据**。

## 3. 技术选型

| 层 | 选择 | 原因 |
|---|---|---|
| 后端 | Python 3.11 + FastAPI | 与现有 Python 系统一致，API 和异步任务适合 |
| 数据库 | SQLite WAL | 单 Mac、单写者、Agent 多读者 |
| 数据访问 | 标准 `sqlite3` + 显式 Repository | 避免 ORM 隐式迁移，保持 SQL 对 Agent 可读 |
| 前端 | React + TypeScript + Vite | 七模块复杂状态需要组件化与类型约束 |
| 样式 | CSS Variables + 模块化 CSS | 直接继承 Demo 配色和视觉，不引入第二套设计系统 |
| 后台任务 | SQLite 持久任务 + 单写队列 | 避免内存任务重启丢失 |
| 测试 | pytest + Vitest + Codex 内置浏览器 | 自动化验证与主代理真实交互验收；不使用 Playwright CLI 替代 |
| 部署 | 一个 `python3 workbench/run.py` | 自动检查依赖、建库、启动 API 并服务前端 |
| 默认地址 | `http://127.0.0.1:8799/` | 避开现有 8765、8766、8887、18923、28765、5173 |

### 4.2 必要的工程支持表

这些是工程、注册表与审计设施，不计入 v3.2 的 14 张业务核心表，也不改变核心契约：

| 表 | 用途 |
|---|---|
| `schema_migrations` | 数据库版本 |
| `ingestion_batches` | 历史导入、增量导入、来源范围、计数与错误状态 |
| `ingestion_checkpoints` | 每个适配器的游标、文件哈希和最后成功时间 |
| `audit_log` | 人工操作、敏感动作、事实修正与系统写入审计 |
| `keywords` | 跨搜索系统关键词注册表；不取代快照中的原始关键词 |
| `platforms` | 标准平台、别名与图标注册表；保留来源 raw platform |
| `identity_merge_candidates` | 自动发现但未经批准的身份合并候选与证据 |
| `identity_merge_map` | 旧 ID 到统一 ID、人工合并和撤销证据 |
| `signal_consumption` | 信号被日报、写作或人工流程消费的逐次事件 |
| `job_events` | 持久化任务进度、日志和错误 |
| `system_connections` | 七套系统连接状态和能力声明，不保存秘密 |
| `correction_jobs` | 历史事实修复请求，不直接改事实 |
| `publish_attempts` | 发布幂等键、草稿回执、失败与重试 |

### 4.3 ID 规则

- 已有稳定平台 ID：确定性 ID。
- canonical URL：规范化后确定性 ID。
- Markdown 只有路径：使用 `namespace + 相对路径 + 首次文件哈希`。
- 新建 Wiki/生成稿：ULID。
- 所有 ID 均带类型前缀：
  - `cnt_`、`crt_`、`dsc_`、`snp_`、`hit_`、`obs_`
  - `cmt_`、`cev_`、`ans_`、`rel_`、`sig_`、`job_`
- 合并不删除旧 ID，只写 `identity_merge_map`。

### 4.4 时间规则

- 数据库存 ISO 8601 UTC。
- 同时保留来源原始时间字符串于 `payload_json`。
- 前端默认按 `Asia/Shanghai` 展示。
- 日报必须显式传入 `report_date` 与 `timezone`。

### 4.5 正文规则

- 同一内容只有一个 `content_id`。
- 多次被 WX、MP、GEO 发现，只增加 discovery/relation。
- 外部正文导入后不可人工静默改写。
- Wiki 和生成稿允许编辑，每次保存更新 `file_hash` 并产生版本记录。
- 资产文件名：`YYYYMMDD_可读标题_ID短码.md`。

### 4.6 唯一性、JSON 与指纹规则

- SQLite 的 `NULL` 不参与普通 `UNIQUE` 冲突判断，因此：
  - `content_discoveries` 使用 `COALESCE(snapshot_id, 'no-snapshot')` 的表达式唯一索引。
  - `metric_observations` 使用 `COALESCE(snapshot_id, 'no-snapshot')` 的表达式唯一索引。
  - `geo_source_relations` 对 `position`、`url_raw`、`anchor_text` 使用明确哨兵值的表达式唯一索引，未解析来源也不能重复写入。
- `geo_answers` 为 `(app, question_raw, captured_at, answer_hash)` 建立 `answer_hash IS NOT NULL` 的部分唯一索引，保证历史重放幂等，但不把不同时间的相同回答错误合并。
- 所有 `payload_json`、`features_json`、`tools_json`、`recommended_json`、`entities_json`、`intents_json` 必须在适配器边界按仓库内 JSON Schema 校验；校验失败进入 `ingestion_batches` 错误记录，不写半条事实。
- URL 统一经过平台感知的 canonicalizer：规范 host/scheme/fragment/trailing slash，删除受控追踪参数，但保留可能影响内容身份的平台参数；raw URL 永远保留。
- `file_hash = SHA-256(原始文件字节)`；`content_hash = SHA-256(规范化正文语义文本)`；`answer_hash = SHA-256(规范化回答正文)`，三者不得混用。
- 所有标准时间必须通过 ISO 8601 UTC 校验并以 `Z` 结尾；无法解析的来源时间保留在 JSON 中，标准时间字段不得写伪值。
- `production_jobs.status` 使用受控状态 `queued/running/succeeded/failed/cancelled/blocked`，并具有 `attempt_count`、`max_attempts`、`last_error`、`locked_by`、`locked_at`、`next_attempt_at`，防止重启或多进程重复执行。

### 4.7 单写者、备份与恢复协议

- 所有 Hub 写入者必须先对 `data/hub/content_hub.lock` 获取进程级 `flock`，再执行 `BEGIN IMMEDIATE`；未获得锁时排队或明确失败，禁止绕过。
- SQLite 固定开启 `foreign_keys=ON`、`journal_mode=WAL`、`busy_timeout=5000`；读取可并发，写入统一经过单写队列。
- 备份顺序固定为：获取写锁 → `wal_checkpoint(FULL)` → SQLite online backup 到临时文件 → `integrity_check` → 原子改名。
- 每日自动备份并保留轮换；每月执行一次隔离目录恢复演练，记录恢复时间、校验结果和历史重放差异。
- 对账发现源文件删除、修改或身份冲突时，只创建 `correction_jobs`，不得静默改写历史事实。

### 4.8 安全边界

- Markdown 渲染必须经过 HTML 清洗，链接和图片协议使用白名单，禁止脚本、事件属性和危险内嵌内容。
- 文件读取、Wiki 保存、导入与媒体访问都必须解析真实路径并限制在配置的允许根目录内，拒绝 `..`、软链接逃逸与绝对路径越权。
- SQL 全部参数化；动态排序、字段名和表名只能来自代码白名单。
- Cookie、Token、API Key、浏览器配置只从本机秘密配置读取，永不进入 Hub payload、日志、审计正文或 Git。

## 11. 迭代与返工规则

1. 每个里程碑先实现最小真实链路，再扩展 UI。
2. 子代理只负责边界明确的探索或实现；总体架构、计划、合并和验收由主代理负责。
3. 每批代码必须由主代理检查 diff、运行测试和浏览器验收。
4. 测试失败必须记录证据，派发修复后重新执行相关测试和回归测试。
