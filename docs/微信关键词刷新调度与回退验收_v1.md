## 幂等、并发和持久化

- 显式同键同输入返回同一 `job_id`/`batch_id`，不重复创建命令或快照。
- 显式同键异输入返回 **409**；省略键的请求使用独立隐式键。
- 两个线程并发提交同一键只执行一次 provider；不同键可独立创建任务。
- 进程/应用实例重建后，刷新状态、调度状态、命令和审计投影均从 SQLite 恢复。
- SQLite WAL 运行库通过 `BackupService.snapshot()` 和隔离 `restore_drill()` 验证 `PRAGMA integrity_check=ok`，运行库本身保持不变。

## R19/R20 兼容与切换回退

对 `refresh-all-history`（R19）和 `scheduler-status`（R20）分别执行：

```text
legacy → compare → hub → legacy
```

每一阶段都校验：

1. HTTP 状态码和 JSON 响应精确相等；
2. 旧响应不泄露内部 `runtime_subtype`；
3. compare 使用 recorded reference，不触发网络；
4. 回到 legacy 后旧响应恢复；
5. scheduler 的 `enabled`、间隔、预算、上下次运行、最近运行和错误字段保持兼容。

本次最小修复：取消信号已持久化的 running 单词/批次不再被 `_active_single()` 误判为仍在执行，允许回退后的新幂等键继续进行受控恢复。
