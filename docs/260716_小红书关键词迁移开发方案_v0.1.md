## 3. 哪些不能照抄微信

## 3.1 指标语义完全不同

| 微信 | 小红书 |
|---|---|
| `read_count` | 通常没有可靠公开阅读量 |
| `like_count` | `liked_count` |
| `friends_follow_count` | `collected_count`、`comment_count`、`shared_count` |
| 公众号账号 | 博主账号与丰富画像 |
| 微信正文 | 图文笔记或视频笔记 |

不能把小红书点赞映射成微信阅读量，也不能为了复用微信页面计算而伪造曝光量。

## 3.2 TikHub 有成本、限流和两阶段采集

小红书搜索通常分为：

```text
关键词搜索
  → 得到榜单和 note_id
  → 按需抓笔记详情
  → 按需抓博主详情
```

如果每次搜索都全量拉取所有笔记和博主详情，会造成：

- API 成本扩大；
- 429 限流；
- 批次时间不可控；
- 重复详情请求；
- 单词失败拖垮整批。

因此必须单独设计：

- 搜索结果缓存；
- note_id 详情缓存；
- user_id 博主缓存；
- TTL；
- 请求预算；
- 限流退避；
- 部分成功语义。

## 3.3 笔记身份不能使用微信 URL 逻辑

小红书优先身份：

1. `note_id`；
2. 规范化小红书 URL；
3. TikHub 外部标识；
4. 内容哈希只作为补充证据。

博主优先身份：

1. `user_id`；
2. 小红书号；
3. 平台主页 URL；
4. 昵称只能作为显示字段，不能作为唯一身份。

## 3.4 关键词 ID 必须平台隔离

旧微信与小红书都曾使用 `kw_<md5(keyword)>`，同一个文本会产生相同 ID。此次串台中，“自主未来2”正是因为 ID 相同而被微信详情接口错误命中。

新内部身份必须满足至少一种方案：

```text
xhs_keyword_<hash>
wechat_keyword_<hash>
```

或：

```text
PRIMARY KEY(system_key, platform, external_keyword_id)
```

兼容层可以继续回显旧 ID，但 Repository 查询不能只凭 `keyword_id`。

## 3.5 热度和趋势必须重新定义

小红书热度只能基于明确可观察值，例如：

- 点赞；
- 收藏；
- 评论；
- 分享；
- 排名；
- 在榜天数；
- 新进榜；
- 多关键词覆盖。

综合热度必须保存：

- 公式版本；
- 输入指标；
- 归一化方法；
- 置信度；
- 采样窗口。

不能直接照搬微信阅读量预测。

## 3.6 图文、视频、图片和封面是独立问题

小红书独有：

- 图文/视频类型；
- 多图；
- 视频封面；
- 图片防盗链；
- 详情接口可能暂不可用；
- 原链接与临时媒体 URL 生命周期不同。

媒体代理、缓存和失败占位需要单独验收。

## 3.7 `snapshot_terms` 当前为空

微信有下拉词和关联词历史，小红书当前没有。新系统必须显示真实空态，不能：

- 复制微信关联词；
- 用标题分词伪装搜索建议；
- 因为空就删除前端字段。

## 7. 验收重点

### 7.1 防串台门禁

每个小红书响应必须能证明：

- `platform=xiaohongshu`；
- Provider 为 TikHub/XHS；
- `note_id/user_id` 符合小红书身份；
- 不出现微信文章 URL、公众号头像和微信阅读量字段；
- 活动批次只来自 `xhs-search`；
- 浏览器刷新状态键带 `xhs` 前缀。

### 7.2 指标门禁

- 点赞、收藏、评论、分享不能互换；
- 缺失值不能补成 0，除非原契约明确；
- 综合热度可追溯到公式和原始观测；
- 视频与图文类型正确；
- 互动观测按时间追加。

### 7.3 刷新门禁

- Token 缺失明确阻断；
- 429 有退避；
- 5xx/超时可重试；
- 部分成功不能显示全成功；
- raw、快照、命中、指标、任务和审计一致；
- 全量刷新不能因为页面重复点击启动两次。

## 8. 当前建议

> **先不要立刻照搬微信全部实现。**

最稳妥的第一步是：

1. 保持当前小红书旧服务继续只读提供真实数据；
2. 冻结小红书接口和页面基线；
3. 把 `/api/v1/xhs` 扩展为完整只读契约；
4. 全量导入并完成读取影子比较；
5. 再决定 TikHub 刷新接管深度。

这样既能复用微信已经验证过的迁移骨架，又不会把微信的数据语义、身份算法和刷新 Provider 强塞给小红书。
