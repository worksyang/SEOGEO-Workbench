## 1. 范围与不可变边界

### 1.1 本计划范围

首版交付一个运行在 `127.0.0.1:8799` 的 FastAPI + React/TypeScript 工作台，以中央 SQLite Hub 统一承载身份、发现、快照、关系、指标、GEO、评论、信号和生产任务；以 Markdown 承载正文；以适配器读取七套旧系统；以历史导入、实时操作、对账恢复三条通道逐步切换查询入口。

范围包括：

- v3.2 的 14 张业务核心表、13 张工程支持表、迁移器、单写锁、WAL、备份恢复、Schema/ID/时间/URL/路径校验。
- 微信搜一搜、公众号监控、小红书、GEOProMax、Wiki/母文章库、WritingMoney、旧写作与发布七套系统的分阶段接入。
- 首页、七业务模块、数据治理、系统状态及 Demo 一致性的前端壳和真实数据态。
- 增量导入、幂等重放、身份解析、来源追溯、对账报告、信号和持久任务。
- T001–T180 自动化、契约、恢复、安全、性能和 Codex 内置浏览器验收。

### 1.2 永久只读边界

以下内容不得修改、不得把新代码写入其中、不得回滚其他代理或用户改动：

- `source/**`、`source/_local_full_backup/**`。
- `unified-content-platform-demo.html` 及七套原始项目中的历史事实文件。
- 本次计划文档编写仅改动该文件；后续各里程碑可在 AGENTS.md 允许的 `workbench/`、`data/`、`asset_store/`、`docs/` 范围内修改。

允许读取旧系统 HTTP/API、SQLite、JSON、CSV、Markdown、日志、批次目录和本地完整镜像；新代码只能写 `workbench/`、`data/`、`asset_store/`、`docs/` 和测试临时目录。Cookie、Token、API Key、浏览器配置、原始大数据、Hub 运行库不得进入 Git。

### 1.3 不可变数据与安全原则

- Markdown 是正文事实源；SQLite 是身份、关系、快照、指标、任务、审计和状态事实源；Frontmatter 只保存稳定身份字段。
- 同一正文只有一个 `content_id`；WX、MP、GEO 的发现记录保持独立，公众号库存不得参与微信搜索供需判断。
- 外部正文不静默覆盖；历史事实不直接删除或修正，只能通过 correction job 重放原始证据。
- JSON 只能承载平台边界字段，必须在适配器边界做 JSON Schema 校验；SQL 全部参数化；Markdown 渲染清洗 XSS；文件路径禁止穿越和软链接逃逸。
- 所有 Hub 写入必须经过进程级 `flock` + `BEGIN IMMEDIATE` 单写路径；真实刷新、计费 GEO、AI 写作和发布必须显式确认，自动化只用 dry-run、草稿或模拟提供方。

## 5. 迁移、幂等、对账、备份与恢复

### 5.1 迁移顺序

1. M0：冻结 ID 前缀、namespace、时间、指标字典、URL canonicalizer、Markdown/JSON Schema、目录和核心/支持表边界。
2. M1：为各适配器建立 `ingestion_batches` 与 checkpoint；先导入关键词/平台/创作者/内容/标识，再导入 discovery、snapshot/hit、指标、评论/GEO、信号。
3. 每批原始文件先落盘并生成 manifest，再解析正文和身份，再写事实；任一 JSON Schema/时间/路径错误整条记录拒绝并登记错误。
4. 双写阶段旧系统仍是运行保障；Hub 只接收适配器事实，禁止人工补写历史快照、排名和观测。
5. 连续 14 天核对日报/信号；连续 30 天核心数据一致后，旧派生文件停止作为事实源但原始文件不删除。

### 5.2 幂等与身份规则

- 稳定平台 ID、规范 URL、Markdown 路径 + 首次哈希分别按约定生成确定性 ID；新建 Wiki/生成稿使用 ULID。
- `(namespace, external_id)`、规范 URL、快照平台+关键词+UTC 时间、snapshot+rank、带 `no-snapshot` 哨兵的 discovery/observation 等唯一约束必须在数据库实现。
- `file_hash`、`content_hash`、`answer_hash` 均为 SHA-256，但语义不能混用；重放同一 manifest 不得新增事实。
- 同标题/正文相似只能生成 `identity_merge_candidates`；人工合并写 `identity_merge_map`，保留旧 ID，可撤销。
- 任务用 `job_events`、状态、锁、attempt_count/last_error/next_attempt_at（实际字段以当前迁移和测试复核为准）支持崩溃恢复；失败不自动标为成功。

### 5.3 对账

每日/每批生成 `data/hub/reports/reconcile/YYYY-MM-DD.md`，至少检查：

- 文件完整性：`md_path` 存在、file hash、source_ref 和 manifest 可定位。
- 身份完整性：外部 ID 一对多、同 URL 多内容、孤立 hit、placeholder 回填、跨系统复用。
- 数量与事实连续性：关键词、快照、hit、内容、指标、回答、来源、关系和任务计数；缺采集与真实空结果分开。
- 指标语义：累计/瞬时/状态不混算；异常倒退和单位变化进入 signal/correction job。
- 引用/关系完整性：GEO relation、comment event、production output 有存在主体。

对账差异必须保留原始行/批次/错误码和置信度；修复走 correction job，不能删除差异证据。

### 5.4 备份与恢复

- 每日：取得写锁 → `wal_checkpoint(FULL)` → SQLite Online Backup → 临时文件 `integrity_check` → 原子改名 → 轮换保留。
- 每周：`PRAGMA integrity_check`、备份可读性、manifest 可定位性检查。
- 每月：隔离目录恢复演练，记录恢复时间、Schema 版本、计数差异、重放差异和结论。
- Hub 损坏：优先从最近合格备份恢复；空库或备份不可用时按 manifest/checkpoint 从历史镜像重放；不得用假数据填充缺失事实。

## 6. 前端 Demo 一致性方案

`unified-content-platform-demo.html` 是唯一视觉事实源，只读引用，不直接改写。React 实现必须复用其语言：浅灰全局外壳、白色圆角主画布、左侧分组导航、顶部工具区、黑色主按钮、低饱和状态色、紧凑字号/间距/边框/圆角；业务内容只替换为真实 Hub 数据。

## 9. 风险、外部阻塞与禁止伪成功

| 风险/阻塞 | 识别信号 | 处理方式 | 禁止行为 |
|---|---|---|---|
| 旧服务未启动/端口变化 | 连接探测失败、超时、端口不符 | 记录绝对时间、端口、原始错误；页面显示 offline/blocked；恢复后重试 | 默认标 healthy、用静态数量冒充在线 |
| WeRSS 登录不一致/二维码失效 | auth check 与 runtime 不一致 | 显示不一致，保留二维码/任务错误；不启动伪采集 | Toast “成功”、伪造账号数据 |
| API Key/Token/Cookie 缺失 | 配置校验或上游 401/403 | blocked/failed + 脱敏错误；只允许 dry-run | 返回秘密、假成功、写入 payload/log |
| 上游限流/5xx/半成功 | HTTP 429/5xx、批次 partial_failed | 持久化错误、可重试、保留已入库事实与失败清单 | 把部分结果标成全量成功 |
| 历史数据字段不确定/身份冲突 | schema 错、同 URL 多内容、一对多 ID | raw 保留、候选合并、correction job、标待验证 | 猜测合并、静默覆盖历史 |
| GEO 计费/外部刷新不可逆 | 需要真实 Provider 或付费动作 | 二次确认、dry-run、回执和审计；凭据不足保持 blocked | 自动调用真实刷新 |
| 发布风险 | Cookie/账号/渠道异常、回执缺失 | 先预览/草稿/dry-run；幂等键；显式确认 | 自动群发、重复发布 |
| 内置浏览器插件故障 | 原始插件错误 | 保存错误证据，后续重试 | 改用 Playwright/Chrome 伪装验收 |

> **禁止伪成功原则**：系统只能显示“成功”当且仅当存在可定位的上游回执 + 持久任务/批次结果 + Hub/文件变化或明确的真实只读响应。否则只能显示 `unknown`、`degraded`、`offline`、`blocked`、`failed`、`partial_failed` 等真实状态，并保留重试入口和原始证据。

## 10. Git 提交流程

每个完整里程碑交付前，主代理按以下顺序执行，不覆盖其他人的提交：

1. `git fetch origin`。
2. `git status --short --branch`。
3. `git log --oneline HEAD..origin/main`；若远端领先，先审查差异，再决定 rebase 或人工合并，禁止 force push。
4. 检查 `git diff`、`git diff --cached` 和目标文件列表；确认没有秘密、原始大数据、运行库、缓存、本地镜像。
5. 执行适用的语法、类型、单元、契约、迁移/恢复测试；涉及页面时由主代理使用 Codex 内置浏览器验收。
6. 只暂存本里程碑实际改动，使用准确中文 commit message，例如 `M1 完成历史统一索引与幂等对账`。
7. `git commit` 后再次检查 `git status --short --branch`，再 `git push origin main`；不得 `git push --force`。
8. 推送后复核 `git fetch origin`、`git log --oneline HEAD..origin/main` 和工作区，确认本地与 `origin/main` 一致。

本次计划文件只允许修改：`docs/首版实施计划_v1.md`。实施计划本身完成后，交付汇报必须列出修改文件、核心内容、验证命令与未解决的待验证/外部阻塞项；不得把计划中的预期误报为已完成事实。
