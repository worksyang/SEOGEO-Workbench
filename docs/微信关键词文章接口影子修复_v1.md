# 微信关键词文章接口影子修复 v1

## 1. 修复范围

本轮只修改 Hub 的旧接口只读重建层：

- R12 `GET /api/article-hit-detail`
- R15 `GET /api/keyword-manage`
- R16 `GET /api/keyword-discovery`
- R21 `GET /api/articles`
- R22 `GET /api/articles/accounts`

未修改 `features/wechat/service.py`、状态写、刷新、AUX、冻结 payload、旧源码或 8774 reference；未访问 8765、未触发抓取、Aidso 或外部付费动作。

## 2. Shadow 差异根因

### R12

- 详情静态投影使用了 monitor account 的 `name` 数据，却按 `canonical_name` 读取，导致账号名为空。
- 投影生成时把 `batch_id` 固定写成空字符串，没有从冻结 `snapshot.source_file_path` 恢复 `web_*` 批次。

### R21

- 原实现从 6,334 条 canonical `contents` 重建列表，错误代替了旧系统 6,364 条 source article 身份；30 组重复 URL 被合并。
- 命中按 canonical `content_id` 汇总，导致重复 URL 文章互相串词、在榜天数和账号归属串联。
- `published_at` 使用 Hub UTC 字段，旧接口实际输出冻结源的 Asia/Shanghai 无 `Z` 字符串。
- 完整 shadow `run_20260716T054626` 证明前述身份、总数、排序、分页和筛选已经全部收敛，唯一剩余差异是 `hit_keywords` 混入 `kw_*`。
- 冻结 10,191 个 snapshots 覆盖 457 个 source keyword ID，其中 145 个历史/归档词不在 312 个 monitor active 词内。导入 snapshots 时这些词先以 `keyword=keyword_id` 建立占位行；随后 runtime registry 只在关键词不存在时插入，未覆盖该占位文本。
- 旧源码并不读取 snapshot 的 Hub 展示字段，而是严格执行 `snapshot.keyword_id → keyword_registry.keyword_text`；精确文本仍完整保存在导入后的 `search_keyword_settings.payload_json`。
- 排序额外增加 `article_id/account_id` tie-breaker，破坏了旧 Python 稳定排序和原 `articles.json` 顺序。
- 查询存在逐文章、逐 hit 查询，真实冻结上既慢又放大差异。

### R22

- 原实现按 canonical content 去重计数，得到 2,141 个账号候选；旧系统按 source article 身份聚合，应为 2,143 个。
- `headimg_url=None` 被强制改成空字符串，破坏旧 null 语义。

### R15

- `keyword_manage` 是导入时静态投影，`next_refresh_at`、`refresh_age_days`、`is_refresh_due` 没有按请求时刻重算。
- 观察期自动词应使用 3 小时间隔，普通词和人工周期应使用 `refresh_frequency_days × 24` 小时。
- 完整 shadow `run_20260716T054626` 中动态字段已全部收敛，只剩 JSON 数组下标 17（第 18 组 `grp_ced9f780`）两对位置交换：`国寿海外保险/国寿保费融资产品`、`宏利历史分红实现率/宏利分红达成率`。
- 冻结旧 `KeywordRegistryRepository.load_payload()` 的准确顺序是 `COALESCE(keyword_order, 999999), keyword_text`，再按 `group_id` 分桶；导入 projection 的无序 `SELECT *` 结果不能代表旧页面顺序。
- 第二次完整 shadow `run_20260716T063442` 中组内顺序及所有关键词字段均已收敛，唯一差异只剩顶层 `/updated_at`：Hub 仍返回冻结 projection 时间，8774 每次请求返回当前本地时间。
- 冻结旧源码的准确实现是 `datetime.now().isoformat(timespec="seconds")`：本地 naive 时间、秒精度、无微秒、无 `Z` 或 UTC offset。

### R16

- Hub 表的 `payload_json/payload/source_ref` 包装字段被直接泄露。
- probes/candidates 的排序、状态筛选、summary 结构和 JSON 列展开方式没有还原旧 repository。

## 3. 实现方式

### R12

- 保留已生成的详情投影，仅从现有 Hub 事实补齐缺口：
  - `content_identifiers(namespace=wechat_article).payload_json` 恢复原 article/account 身份；
  - `creators.payload_json` 恢复 canonical account；
  - account 投影作为旧 monitor fallback；
  - `search_snapshots.payload_json.source_file_path` 恢复批次号。

### R21 / R22

- 以 `content_identifiers(namespace=wechat_article)` 为文章身份清单，按 `rowid` 保留冻结 `articles.json` 导入顺序。
- 以 `search_hits.payload_json.article_id/snapshot_id` 连接每条 source article，禁止用 canonical `content_id` 串联重复 URL。
- 以 snapshot 原始 payload 恢复 `snapshot_date`；关键词文本按旧 source ID 精确映射：
  1. `search_keyword_settings.payload_json.keyword_text`（冻结 runtime registry，含 145 个历史词）；
  2. `keywords.payload_json` / `keywords.keyword` 的非占位文本；
  3. snapshot 的非 ID 文本 fallback。
- 任一 fallback 与 source `keyword_id` 相同即视为占位 ID，不进入 `hit_keywords`；旧接口只输出文本。
- 标题、URL、账号、指标、封面、正文路径、发布时间全部取 article identifier 原始 payload。
- 逐行还原旧 7 种排序、筛选、30 天 todayReads、accountScore 每账号最多 3 篇、分页及稳定排序。
- R22 直接复用同一 source article 集合聚合，保留旧 null 语义。
- 增加按数据库路径、导入 checkpoint、事实计数/rowid、关键词、runtime setting、账号和投影版本组成的进程内缓存；首次构建后不再重复扫描 101,348 条 hits，相关事实版本变化时自动失效。

### R15

- 每次请求从 `search_keyword_settings.payload_json` 取得冻结 runtime 字段并动态计算：
  - `last_refresh_at = last_refresh_at or last_seen_at`
  - observing + 非 manual：3 小时
  - 其他：`max(1, refresh_frequency_days) × 24` 小时
  - `next_refresh_at`、完整天数 age、是否到期均按请求时钟计算。
- 返回前使用同一 runtime payload 的 `keyword_order`、`keyword_text` 对每组关键词重新排序，准确复现旧 repository 的排序，而不是对 group 17 写死特例。
- 顶层 `updated_at` 不再沿用冻结 projection；每次 `keyword_manage()` 请求按旧源码动态执行 `self._now().isoformat(timespec="seconds")`。该值独立于每个关键词的动态刷新字段，不改写关键词级 `updated_at/last_refresh_at/next_refresh_at`。

### R16

- candidates：按 `validation_score DESC`、`related_parent_probe_count DESC`、`related_occurrence_count DESC`、`first_seen_at` 排序。
- probes：按 `proposed_at`、`probe_id` 排序。
- 重复状态参数分别作用于 probes/candidates；limit 保持 1–500。
- 只返回旧行字段，并展开：
  - probe `warming_facts`、`replacement_candidate_ids`
  - candidate `source_probe_ids`、`source_article_ids`
- summary 返回全部状态计数，不返回 evidence。
