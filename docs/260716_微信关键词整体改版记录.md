> **日期**：2026-07-16
> **用途**：给以后接手微信关键词、其他搜索监控模块迁移或兼容改造的 Agent 阅读。
> **事实基准**：`全域内容资产与观测架构方案_v3.3.md`、`全域内容工作台开发总计划_v2.md`、`全域内容工作台验收矩阵_v2.md`。
> **一句话总结**：这次改版没有重做成熟前端，而是把旧系统散落在 Markdown、JSON、SQLite 和运行目录里的事实，迁入可去重、可观测、可恢复、可审计的新底座，再用兼容 API 把原页面完整“演回来”。

## 1. 为什么要留下这篇记录

微信关键词迁移表面上像一件简单的事：

1. 旧接口返回什么；
2. 新接口也返回什么；
3. 前端不改；
4. 对比一致后切换。

真正做起来才发现，旧接口只是露在水面上的一小部分。接口背后还隐藏着：

- 搜索结果 Markdown；
- 文章正文 Markdown；
- 数万次搜索快照和排名命中；
- 阅读量、点赞、在看等动态观测；
- 关键词分组、置顶、备注、topic、bucket；
- 单词刷新、批量刷新、取消、历史和调度；
- 文章去重、账号别名、惩罚信号、阅读量估算；
- 页面排序、空态、错误状态和历史兼容行为。

因此，这次改版最重要的经验不是某段代码，而是：

> **不要把“页面能显示”误认为“系统迁移完成”。迁移完成意味着每一份历史事实、每一种动态状态、每一个副作用和每一种失败都能解释、落盘、重放、对账和回退。**

## 2. 旧系统原来是什么结构

旧微信系统不是单一数据库应用，而是多个事实层叠加形成的运行系统。

### 2.1 搜索结果 Markdown

旧抓取任务会为关键词批次生成搜索结果目录和 Markdown。它们同时承担：

- 一次搜索发生过的证据；
- 搜索结果列表；
- 文章排名；
- 文章标题、账号、摘要和链接；
- 部分任务与批次信息。

问题在于，同一篇文章会在多个关键词、多个批次、多个日期中反复出现。搜索结果 Markdown 更适合作为**原始不可再生证据**，不适合继续承担实时查询数据库的职责。

### 2.2 文章正文 Markdown

文章正文 Markdown 是真正需要长期保存的内容资产。它与搜索结果 Markdown 的性质不同：

- 搜索列表是一次采集事实；
- 正文是内容本体；
- 阅读量、排名等动态值不是正文的一部分。

旧系统中同一篇文章可能被多个关键词重复抓取，从而产生多份路径不同、内容相同或接近的 Markdown。

### 2.3 标准化 JSON

`normalized/` 中保存了：

- `keywords.json`
- `snapshots.json`
- `ranking_hits.json`
- `snapshot_terms.json`
- `articles.json`
- 指标观测和阅读变化
- 聚合后的 `monitor-data.json`

这些 JSON 已经比 Markdown 更接近数据库结构，但仍有几个问题：

- 大文件每次完整解析成本高；
- 缺少事务和并发写保护；
- 增量导入、幂等重放和跨文件约束困难；
- 任务运行态与事实数据仍然分散。

### 2.4 SQLite 与运行目录

旧 `app.db`、调度配置、刷新任务目录和 ledger 主要保存：

- 关键词注册表；
- 分组和页面设置；
- 刷新策略；
- 单词和批量任务状态；
- 调度状态；
- 部分附加业务数据。

因此旧系统的真实答案从来不在一个地方，而是：

```text
Markdown 原始证据
  + 标准化 JSON
  + SQLite 设置
  + 任务运行目录
  + 页面端兼容逻辑
```

## 3. 为什么必须改成新结构

### 3.1 重复内容越来越多

旧结构以“抓取批次目录”为中心，而新结构必须以“内容身份”为中心。同一篇文章可以被很多关键词、很多批次发现，但正文资产只应有一个稳定身份。

### 3.2 动态数据不能继续塞进 Markdown

阅读量、排名、点赞、在看、趋势和预测会持续变化。如果把它们写进正文 Frontmatter：

- Markdown 会不断被改写；
- 很难知道某个数字是什么时候观测到的；
- 历史值被最新值覆盖；
- 正文哈希和内容去重会被动态字段干扰。

所以新结构明确：

> **Markdown 保存正文；SQLite 保存身份、关系、动态观测和运行状态。**

### 3.3 旧任务难以恢复和解释

仅保存“成功/失败”不够。批量刷新必须知道：

- 哪些词已完成；
- 哪些词失败；
- 当前跑到哪个词；
- 失败原因；
- 是否请求取消；
- 是否允许恢复；
- 是否已经调用过外部 Provider；
- 重启后如何继续。

### 3.4 写操作必须可防重

页面重复点击、网络重试和进程重启都可能让同一刷新被执行多次。新系统要求写接口带幂等键，同一个业务请求只能产生一次真实副作用。

### 3.5 必须能影子比较和一键回退

迁移不是“相信新代码写对了”，而是让旧系统演一遍、新系统演一遍，再比较：

- HTTP 状态码；
- JSON 字段；
- 排序；
- 数字；
- 错误；
- 数据库落盘；
- 任务和审计记录。

## 4. 新系统的数据理念

## 4.1 四层结构

```text
原始证据层
  搜索结果、原始响应、旧 Markdown、冻结包

标准事实层
  contents / creators / search_snapshots / search_hits / metric_observations

模块运行层
  keyword settings / refresh jobs / refresh items / scheduler

兼容投影层
  按旧页面需要还原原 /api/... 响应
```

这四层不能混为一层。

### 4.2 v3.3 的核心事实表

微信主要使用以下跨系统核心表：

| 表 | 微信语义 |
|---|---|
| `contents` | 微信文章的统一内容身份 |
| `creators` | 公众号或作者身份 |
| `content_identifiers` | 旧 article_id、微信外部 ID、URL 等标识 |
| `content_discoveries` | 哪个关键词、批次或系统发现了文章 |
| `search_snapshots` | 某关键词在某次采集时刻的完整快照 |
| `search_hits` | 快照中的排名与原始展示事实 |
| `metric_definitions` | 阅读、点赞、在看等指标的明确语义 |
| `metric_observations` | 某文章在某时间点观测到的指标值 |
| `signals` | 新收录、排名变化、阅读异常等信号 |
| `production_jobs` | 后续内容生产任务的跨模块摘要 |

14 张核心表保持独立；微信不能为了页面方便重新合并成“大宽表”。

### 4.3 搜索模块运行表

以下是运行状态，不应混称核心事实：

- `search_keyword_groups`
- `search_keyword_settings`
- `search_refresh_jobs`
- `search_refresh_items`
- `search_refresh_events`
- `search_scheduler_state`
- `command_runs`
- `job_events`
- `audit_log`

微信和小红书可以共享这些表结构，但必须用 `system_key/platform` 隔离。

### 4.4 Markdown 的新职责

标准正文 Frontmatter只保存相对稳定字段，例如：

```yaml
schema_version: "content-md/1.1"
content_id: "..."
content_type: "external_article"
title: "..."
canonical_url: "..."
published_at: "..."
author: "..."
```

以下内容不得写入正文：

- 当前排名；
- 历史命中次数；
- 阅读量及增量；
- 点赞、在看；
- 趋势、预测和置信度；
- 刷新任务状态；
- 关键词分组和置顶状态。

## 5. 兼容性为什么会比较强

### 5.1 前端契约没有被强迫重写

原页面继续调用旧 `/api/...`。新系统在兼容层把 Hub 数据重新投影为旧响应形状，因此：

- 原 DOM 不需要重做；
- 原排序和字段可以逐项核对；
- 可以按接口切换而不是一次重写整页；
- 回退时只切迁移模式，不需要回滚页面。

### 5.2 旧标识和旧路径没有被抛弃

新系统保留：

- 旧 `keyword_id`；
- 旧 `article_id`；
- 原始 URL；
- 旧 Markdown 相对路径；
- 批次、快照和来源引用；
- 原始 payload 的安全副本。

旧页面需要哪个标识，兼容投影就可以还原哪个标识。

### 5.3 去重不等于删除历史

正文可能从很多份合并为一个 `content_id`，但这些关系必须保留：

```text
多个旧文件路径
多个旧 article_id
多个关键词命中
多个搜索快照
多个动态观测
        ↓
同一个标准 content_id
```

因此标准正文数量可以减少，但历史排名、来源和观测不能减少。

### 5.4 新旧读取模式可切换

迁移支持：

- `legacy`：读取冻结参考答案；
- `compare`：同时读取新旧并记录差异；
- `hub`：读取新底座。

这比直接删除旧实现安全得多。

## 6. 实际开发过程中的坎坷

### 6.1 “接口能返回”不等于契约一致

早期最容易犯的错误是只比较 JSON 大致结构。后来发现差异可能来自：

- 字段缺失与 `null` 的区别；
- 数字和字符串类型差异；
- 数组顺序；
- 默认排序；
- Flask 旧错误页与新 JSON 错误；
- ETag、缓存头和 gzip；
- 时间格式和时区；
- 空结果到底是 200、404 还是 500。

最终必须建立接口专用的规范化和差异规则，而不是只比哈希。

### 6.2 大 JSON 不能成为实时查询路径

旧 `monitor-data.json` 体积很大。如果每次请求都重新解析，会出现：

- 响应慢；
- 内存压力大；
- 多请求重复计算；
- 一次局部查询也要读完整文件。

因此新系统把事实导入 SQLite，通过 Repository 按需查询，再还原旧响应。

### 6.3 写接口不能直接代理旧系统

如果原 iframe 的 POST/PATCH/DELETE 直接打到旧系统，就可能：

- 绕过新 Hub；
- 双写两边但没有一致性证据；
- 迁移模式失效；
- 无法统一审计和回退。

后来统一采用：

```text
页面写请求
  → Hub 命令
  → 幂等检查
  → 单写锁
  → 新运行表
  → 隔离副本对账（需要时）
  → 审计回执
```

### 6.4 真实刷新不能为了测试调用两次

影子比较可以把同一个 GET 发给两套系统，但真实搜索抓取、Aidso 或其他外部副作用不能简单执行两遍。正确方法是：

1. 外部 Provider 只调用一次；
2. 保存原始结果；
3. 旧转换器和新转换器分别消费同一份结果；
4. 比较两边的标准化产物。

### 6.5 文章去重不能只看标题

标题相同不代表同一篇文章，标题变化也不代表不同文章。身份优先级必须是：

1. 稳定外部 ID；
2. canonical URL；
3. 公众号与文章外部 ID；
4. 正文哈希；
5. 标题、作者、发布时间只能产生人工合并候选。

### 6.6 刷新状态比想象中复杂

前端只显示一条进度，但后端至少有：

- command；
- batch job；
- keyword item；
- Provider request；
- snapshot；
- hit；
- event；
- audit。

如果只维护一个内存变量，进程一重启，页面就不知道任务发生过什么。因此状态必须持久化。

### 6.7 最大的一次教训：模块同名接口串台

微信和小红书旧页面都使用：

```text
/api/monitor-data/*
/api/keyword-manage
/api/articles
/api/refresh-all/*
/api/keywords/*
```

微信迁移时新增的显式兼容路由优先于旧总代理，结果把小红书页面的部分请求截成了微信请求：

- 小红书首页显示 151 个词；
- 关键词详情却显示微信数据；
- 文章 List 返回微信文章；
- 关键词管理显示微信 312 个词；
- 刷新进度显示微信 24/312。

这次事故说明：

> **共享表结构不等于共享 API 命名空间；共享前端代码不等于共享运行状态；任何搜索模块都必须在入口处先确定 module/platform，再进入 Repository、任务和缓存。**

2026-07-16 已完成第一层解耦：

- 小红书 iframe 的旧根 `/api/*` 在进入微信路由前统一分流到小红书来源；
- `/api/v1/xhs/*` 与 `/api/v1/wechat/*` 保持独立；
- 小红书与微信的刷新 `localStorage` key 分开；
- 小红书旧写请求继续 fail-closed，不会写入微信。

## 7. 最终形成的开发方法

### 7.1 先冻结标准答案

没有稳定参考端，就无法判断差异来自新代码还是旧数据继续变化。

### 7.2 先读后写

顺序应为：

```text
bootstrap
→ keyword/account detail
→ article list/content
→ keyword settings
→ single refresh
→ batch refresh
→ cancel/resume/history
→ scheduler
```

### 7.3 接口逐条影子验收

每条接口至少比较：

- 状态码；
- content-type；
- 字段和类型；
- 数组顺序；
- 数值；
- 时间；
- 空态；
- 非法输入；
- 数据库落盘。

### 7.4 页面必须在真实正常态验收

不能只验收错误页或空态。至少要走：

```text
真实数据加载
→ 原版操作
→ 写入落盘
→ 刷新恢复
→ 异常态
→ 对账与回退
```

## 8. 给未来 Agent 的硬性提醒

1. 不要修改 `source/**` 和旧事实源。
2. 不要把 Demo 当业务页面设计稿。
3. 不要为了统一而删除原页面字段。
4. 不要把动态指标写回正文 Markdown。
5. 不要只用标题自动合并文章。
6. 不要让微信和小红书共用无平台隔离的 Repository 查询。
7. 不要让两个模块共用刷新状态、浏览器缓存键或活动批次。
8. 不要把 Toast 当成功证据。
9. 不要在影子测试中重复执行真实外部副作用。
10. 不要在没有备份、对账和回退能力时切换到 Hub。

## 9. 这次改版真正改变了什么

过去系统更像：

> “抓一次，生成一堆文件，再由页面把这些文件拼起来。”

改版后系统更像：

> “每次采集都形成可追踪的快照、命中、内容身份和指标观测；Markdown 继续保存正文，数据库负责关系、历史和任务；原页面通过兼容 API 继续工作。”

最大的收益不是页面变新，而是：

- 数据不再依赖某一个目录结构才能解释；
- 同一篇文章可以跨关键词去重；
- 动态指标保留完整历史；
- 刷新任务可恢复；
- 写操作可防重；
- 新旧结果可比较；
- 出错可以回退；
- 其他模块可以复用方法，但不会再复用错业务语义。
