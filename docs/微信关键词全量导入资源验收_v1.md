# 微信关键词全量导入资源验收 v1

## 结论口径

大冻结源不得把全部 JSON 数组、`accepted_rows`、6364 个正文兼容投影和单一超大事务同时驻留内存。全量导入现在按源文件流式读取、**单 writer lock + 单业务 transaction** 执行；仍可按批 `execute`，但不在阶段之间提交。任一中段异常会回滚核心表、运行表和兼容投影到导入前状态；命令失败状态与 audit 在独立事务保留，随后可用新幂等键重试，同一成功幂等键可安全重放。文章详情投影按批生成并以 zlib+base64 存储，读取时由 `WechatLegacyRepository` 解压，语义不变。

**严禁把不同事实集合混成一个总数：**

| 维度 | 源事实 | Hub 对账 |
|---|---:|---:|
| monitor active scope 关键词 | 312 | 312 |
| runtime registry 关键词 | 457 | 457 |
| `wechat_article` 身份标识 | 6364 | 6364 |
| canonical 去重后的 `contents` | 6334 | 6334 |
| search snapshots | 10191 | 10191 |
| search hits | 101348 | 101348 |
| `wechat.*` metric observations | 80665 | 80665 |

`ingestion_batches.status` 只有在分层对账全部通过且无 rejected rows 时为 `succeeded`；失败/中断保留 `running` 或 `partial_failed`，不得伪造原子成功。`records_written` 按实际源记录减去 rejected rows 记录，不再固定写 0。

> **457 条底层历史注册词 ≠ R15 页面 312 个活跃词。** 冻结 app.db 的 registry 为 312 条 `status='active' AND archived_at IS NULL` 加 145 条 archived；导入和 reconcile 必须保留全量 457，`GET /api/keyword-manage` 只能返回 18 个分组中的 312 个活跃可见词。

## 正文资产完整性口径

冻结 `normalized/articles.json` 的 6364 条文章身份中：

| 正文维度 | 数量 | 验收含义 |
|---|---:|---|
| 有 `content_file_path` | 4328 | 必须全部产生 `wechat_article_paths` 和可读 asset |
| `payload/<relative>` 可定位 | 4319 | 优先来源 |
| 仅 `payload/code-snapshot/<relative>` 可定位 | 9 | 安全 fallback 来源 |
| 源本来无 `content_file_path` | 2036 | 记为 `source_no_content_path`，不计迁移丢失 |
| 有路径但导入后无 asset | 0 | `missing_path_asset_count` 必须为 0 |

正文解析只允许两条候选：先读 `root/<relative>`，不存在时再读 `root/code-snapshot/<relative>`；两条路径均须为 root 内 `.md` 相对路径并逐级拒绝软链接、绝对路径与 `..`。`wechat_article_paths.relative_path` 保留冻结原始路径，`source_ref` 必须指向实际命中的路径，因此 fallback 记录以 `code-snapshot/...` 结尾。导入验收还必须逐条重新读取实际 source 并与 asset blob 比较 SHA-256；一字节不一致即 `verified=false`，`sha256_mismatches` 保留可审计证据。fallback 正常命中的 source hash 纳入 `source_unique_blob_count`，不能从唯一 blob 统计中排除。
