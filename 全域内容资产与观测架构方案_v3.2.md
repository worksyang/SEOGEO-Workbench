# 全域内容资产、搜索观测与 Agent 协作架构方案 v3.2

> 适用范围：微信搜一搜、公众号监控、小红书关键词监控、GEOProMax、Wiki Viewer、WritingMoney、旧写作系统、微信公众号发布系统，以及未来接入的抖音、视频号、百度和其他 AI 搜索平台。  
> 版本定位：保留 v3.0 的语义边界与可追溯能力，吸收 v3.1 的工程简化思路，面向 SQLite、本地脚本和 Agent 日常分析场景进行收敛。  
> 一句话原则：**内容让人和 Agent 好读，事实让数据库可靠保存，系统之间共享身份但不抹平语义，Agent 有自由但不靠猜。**

---

## 一、结论先行

v3.2 不再选择两个极端：既不采用 v2.0 的“少数万能表承载全部平台”，也不照搬 v3.0 的 30 多张高度正规化表。最终采用以下结构：

1. **四个不可删除的核心概念**：内容 Content、发现 Discovery、快照 Snapshot、观测 Observation。
2. **一个中央 SQLite，14 张核心表**：一个人可以理解和维护，关键关系仍可直接查询。
3. **Markdown 与数据库分工明确**：Markdown 是正文事实源；SQLite 是元数据、关系和时序事实源；Frontmatter 只是稳定身份证。
4. **Agent 双通道访问**：优先直查 SQL 视图并阅读 Markdown；可选工具接口用于提效，但不是唯一入口。
5. **五个日报标准视图**：阅读量暴增、关键词新收录、排名异动、新账号、评论异常开箱即用。
6. **原始文件不强制逐条登记进数据库**：保留可重建目录和批次 manifest；关键事实表记录 `source_ref`，兼顾追溯和轻量维护。
7. **渐进迁移、双写对账、可随时回滚**：旧系统在一致性验证完成前仍是运行保障。

> **v3.2 的核心不是“表越少越好”，而是让每张表都承担一种稳定语义，让 Agent 不必在 30 张表里迷路，也不必在几个 JSON 字段里盲猜。**

## 第一阶段：从“怎么存Markdown”开始的朴素直觉

**用户**：咱们先不聊复杂的，就只说一个HTML里面有小红书、有抖音、有视频号、有GEO、有微信公众号。我都是基于搜索。你就说我们这些字段，文章的Markdown文件，排序的一些情况，时间的一些情况，怎么存呢？每个平台假设HTML页面都是不一样的，但是它们可以互相切换。然后它们中间是不是有一些link的页面之类的？我有点懵。我们目前的文章都是遵循这样的储存方式：日期加上名称，因为Agent可以直接搜索名称就很方便。

**Agent核心观点**：物理文件保持扁平直存，但给每个文件套一个Frontmatter“身份挂牌”，平台隔离和互相跳转全靠挂牌，不靠文件夹。

---

**用户**：你这个命名为什么不是分文件夹存放，而是直接要放个叫内容总库的东西？这放进去不乱吗？假设两个文章重名，查到的日期也一样，那这两个文件会不会有问题？

**Agent核心观点**：（1）Agent搜索逻辑是“全盘扫描”，分文件夹会导致Agent猜路径猜错就找不到；（2）文件名末尾加“平台_唯一ID后四位”作为物理防撞锁，永不冲突。

## 第二阶段：平台区分的关键纠结——WX与MP

**用户**：WX和MP本质上你觉得需要区分吗？我很不希望我的搜索关键词系统里面混入非搜索结果公众号文章。因为我的Agent会每天分析搜索流需要写哪些内容，如果一旦混进来，Agent分析不知道会不会乱。

**Agent核心观点**：必须严格区分。WX证明的是“需求与供给的交汇”（搜索结果），MP证明的是“创作者的日常表达”（公众号存量）。MP数据绝对不能参与搜索热度计算。WX是雷达，MP是军火库，可以放同一个基地但不能互相干扰。

---

**用户**：可是那意味着，假设今天有篇文章叫《香港保险分红实现率》，同一天我们爬也爬到它了，搜索引擎也抓到了，那么它有WX和MP两篇文章是一模一样的，你觉得合理吗？怎么感觉有点怪怪的？

**Agent核心观点**：绝对不能出现两份物理副本。正确的做法是“单源存储，多标签挂载”：物理文件只存一份，platform字段变数组`[wx, mp]`。系统读取时按需过滤，Agent分析搜索流时只认包含wx的文件。

## 第三阶段：动态数据存哪——Markdown还是数据库？

**用户**：现在微信搜一搜的Markdown文件是无限熵增的。今天爬到一个关键词有10个文章，明天这10个文章依旧在，依旧存一遍。这样的话阅读量变化、点赞量变化、评论区变化都能知道。但是头部的标签里面能加这些字段吗？

**Agent核心观点**：绝对不能把这些字段写死在Markdown文件头里。每抓一次改一次文件头会导致知识库和Agent索引崩溃。Markdown是“静态资产”只存事实，动态指标是“观测日志”必须外挂。提出两种方案：同名JSON观测日志，或统一观测日志库。

---

**用户**：从数据库查询的角度，什么样的方式最干净？是不是建一个数据库来记录这些快照比较好？

**Agent核心观点**：SQLite是绝对最优解。JSON文件有查询效率极低和数据一致性问题。提出单表`observations`方案，用`UNIQUE(content_id, captured_at, keyword)`防重复。

---

**用户**：快照是有排名信息的，这个是不是也要存到数据库里面？

**Agent核心观点**：必须存，这是核心字段。同一天同一文章同一关键词只留一条记录（每天最后一次抓取），算“经典存活”和“时效动能”才最干净。

## 第四阶段：评论区的生死留痕问题

**用户**：假设这些内容的评论区被读者删了，但我们前面抓到它没删的状态，第二次取的时候没取到了，但上下文其实还有一些关系能看得出来没抓错。这存数据库吗？还是存在Markdown里？存在数据库的话，我怕AI Agent在分析的时候没有反查数据库就不知道评论情况了。

**Agent核心观点**：正文Markdown保持绝对纯净，但在文件头钉一张“评论动态铭牌”（状态、存亡数、被删样本原话），完整评论尸体存SQLite。Agent读文章时扫一眼头部就知道发生过什么，不需要查库；需要深度复盘时再查数据库。

## 第五阶段：GEO回答和搜索结果特征存哪

**用户**：豆包的SEO搜索结果Markdown回答，是单独存成Markdown文件，还是存到数据库当成JSON字段？任何搜索结果都应当有一张表，有排序、下拉词、关联词，还有各平台的AI总结。这些是存进数据库对吧？

**Agent核心观点**：AI回答必须存成独立Markdown文件（它是机器生成的“百科母文章”）。搜索结果的榜单、下拉词、AI总结框这些“快照事实”，全部结构化存入SQLite，建`search_snapshots`和`search_features`两张表。

## 用户核心思维脉络总结

1. **从直觉出发**：文件名要让人和Agent都好搜，不瞎建文件夹。
2. **对混乱高度敏感**：重名怎么办、WX和MP混了怎么办、一模一样的两份文件怪不怪。
3. **对Agent可靠性极度关注**：Agent分析搜索流时不能被MP数据污染，Agent读文章时不能因为没查数据库而错过评论信号。
4. **对“熵增”有本能厌恶**：Markdown无限覆盖导致知识库崩溃，JSON文件查不动还容易坏。
5. **务实但追求干净**：不做过度工程化，但坚持“不乱、不脏、不互相污染”的底线。

**架构方案最终收敛方向**：Markdown管正文（静态资产），SQLite管动态事实（排名、指标、评论生死、搜索快照），Frontmatter只是身份证不是第二数据库，平台来源不是内容属性而是一次发现事实，Agent有读的自主权但没有乱写的权限。


## 二、设计目标与非目标

### 2.1 必须满足的业务目标

- 同一篇文章被公众号监控、微信搜索和 GEO 引用发现时，只有一个 `content_id` 和一份正文，但保留三条独立发现记录。
- 微信搜一搜选题分析只能使用真实搜索发现数据，公众号库存不得混入搜索供需判断。
- 能按时间回放关键词排名、互动量、阅读量、朋友在看、收藏、评论状态和 GEO 引用位置。
- 日报 Agent 能快速发现：**阅读量暴增、关键词新收录、排名大幅变化、新账号出现、评论删除或失踪、GEO 新引用**。
- 正文、AI 回答、Wiki 母文章和成稿都以可读 Markdown 保存，支持人类浏览、全文检索和 Agent 定点阅读。
- 新平台接入时，绝大多数情况只需增加适配器、指标字典和少量配置，不需要重构中央模型。
- 数据发生异常时，能追溯到原始采集文件或批次目录，并支持重新解析。

### 2.2 本阶段不追求

- 不把七套现有系统立即改写成一个单体应用。
- 不为所有未来可能出现的字段提前建表。
- 不让 SQLite 承担高并发任务队列、全文向量数据库或媒体对象存储职责。
- 不要求所有 Agent 必须通过某个 API 中间件访问数据。
- 不立即删除旧 JSON、旧 SQLite、历史 Markdown 或原始抓取目录。

## 三、五类核心对象

### 3.1 内容 Content

回答“这份东西本身是什么”。例如公众号文章、小红书笔记、视频文字稿、AI 回答、Wiki 母文章和生成稿。内容主体保存稳定身份、标题、作者、发布时间、正文路径和内容指纹，不保存会持续变化的阅读量和排名。

### 3.2 发现 Discovery

回答“哪个系统通过什么方式发现了这份内容”。同一篇公众号文章可以同时具有：

- `mp-monitor / account-feed`
- `wechat-search / keyword-result`
- `geopromax / text-reference`

发现记录是隔离 WX 与 MP 业务语义的关键。**平台来源不是内容属性，而是一次发现事实。**

### 3.3 快照 Snapshot

回答“某个平台在某个时刻呈现了怎样的完整现场”。搜索快照包含关键词、时间、结果列表和页面特征；GEO 回答快照包含问题、回答、工具调用和引用关系。

### 3.4 关系 Relation

回答“两个对象在一次现场中如何关联”。例如文章在搜索快照中排名第 3，来源文章在 AI 回答中位于第 2 个文本引用，评论 B 回复评论 A。

### 3.5 观测 Observation

回答“某个对象在某一时刻测得什么值”。不同平台指标使用不同 `metric_key`，即使表面上都叫阅读或收藏，也不在存储层提前合并。

### 4.1 “核心表”与“扩展表”的边界

14 张核心表是所有部署必须具备的稳定契约。某个平台出现复杂且高频查询的独有结构时，可以增加带命名空间的扩展表，例如 `geo_tool_calls`、`publish_attempts`，但必须满足两个条件：

1. 该结构已经被至少两个真实查询场景反复使用；
2. 放在 JSON 中已经明显影响正确性、索引或查询性能。

因此可扩展性不是“永远不加表”，而是**先进入 JSON 逃生舱，证明价值后再晋升为正式字段或扩展表**。

## 五、文件系统与 Markdown 规范

### 5.1 推荐目录

```text
project_root/
├── data/
│   ├── hub/

│   │   ├── AGENT_GUIDE.md
│   │   ├── SCHEMA.md
│   │   └── reports/reconcile/
│   ├── raw/
│   │   ├── wechat-search/2026/07/14/
│   │   ├── mp-monitor/2026/07/14/
│   │   ├── xhs-search/2026/07/14/
│   │   └── geopromax/2026/07/14/
│   └── manifests/
├── asset_store/
│   ├── content/2026/07/
│   ├── ai_answers/doubao/2026/07/
│   ├── wiki/
│   ├── generated/2026/07/
│   └── media/
└── adapters/
```

原始文件按“系统 + 日期 + 批次”保存，批次生成一份 manifest。中央表只需记录可以定位原始证据的 `source_ref`，不要求为每个小文件维护复杂血缘图。

### 5.2 文件命名

```text
{日期}_{可读标题}_{content_id短码}.md
```

示例：

```text
20260710_香港保险分红实现率_cnt_7F4K.md
20260712_香港储蓄险有什么缺点_ans_91J3.md
```

文件名用于人类浏览和 `grep`，唯一身份仍是 `content_id`。改名或移动文件不得生成新的内容身份。

### 5.3 Frontmatter：稳定身份证，不是第二数据库

```yaml
---
schema_version: "content-md/1.1"
content_id: "cnt_01J2Z8T6P7F4K"
content_type: "external_article"
title: "香港保险分红实现率"
canonical_url: "https://mp.weixin.qq.com/s/..."
published_at: "2026-07-10T09:30:00+08:00"
author: "保智库"
---
```

Frontmatter 只保留 7 个稳定字段。不得写入阅读量、排名、发现渠道、实体意图、生产状态或不断增长的来源数组。

### 5.4 两类 Markdown 的可变性

- **外部内容与 AI 回答**：原则上不可人工改写；解析修复应生成新版本或留下变更记录。
- **Wiki 与生成稿**：允许人工编辑正文；数据库通过文件哈希发现变化并更新索引时间。

正文事实源是 Markdown，元数据、标签、关系、指标和状态事实源是 SQLite。标题等少量镜像字段出现不一致时，以数据库作为列表和查询展示依据，以 Markdown 正文作为内容阅读依据，并由对账报告提示修复。

## 六、14 张核心表

### 6.1 身份层：4 张

#### 1. `contents`

```sql
CREATE TABLE contents (
    content_id       TEXT PRIMARY KEY,
    content_type     TEXT NOT NULL,
    title            TEXT,
    canonical_url    TEXT,
    creator_id       TEXT,
    author_name      TEXT,
    published_at     TEXT,
    first_seen_at    TEXT NOT NULL,
    updated_at       TEXT NOT NULL,
    md_path          TEXT,
    file_hash        TEXT,
    content_hash     TEXT,
    domain           TEXT,
    entities_json    TEXT NOT NULL DEFAULT '[]',
    intents_json     TEXT NOT NULL DEFAULT '[]',
    payload_json     TEXT NOT NULL DEFAULT '{}'
);

CREATE UNIQUE INDEX ux_contents_canonical_url
ON contents(canonical_url)
WHERE canonical_url IS NOT NULL;

CREATE INDEX ix_contents_type_time
ON contents(content_type, published_at);
```

`entities_json` 和 `intents_json` 在当前阶段采用受控 JSON 数组，词条由仓库中的 `taxonomy.yaml` 管理。它们不是自由文本标签；适配器和人工编辑器必须校验词典。未来出现层级关系、别名合并或百万级关联时，再升级为独立词典表。

#### 2. `creators`

```sql
CREATE TABLE creators (
    creator_id       TEXT PRIMARY KEY,
    canonical_name   TEXT,
    platform         TEXT,
    external_id      TEXT,
    profile_url      TEXT,
    first_seen_at    TEXT NOT NULL,
    updated_at       TEXT NOT NULL,
    payload_json     TEXT NOT NULL DEFAULT '{}',
```

## 七、JSON 使用规则：逃生舱，不是垃圾桶

允许 JSON 的场景：

- 字段结构仍在快速变化；
- 查询通常按整条记录读取；
- 暂时没有索引、聚合和 JOIN 需求；
- 数据可通过 schema 校验。

必须升级为正式列或扩展表的触发条件：

- 同一个 JSON 路径出现在三个以上核心查询中；
- 需要建立索引、排序、聚合或外键；
- JSON 内元素具有独立生命周期或状态；
- 数据错误会直接影响日报、选题或生产决策。

仓库应为重要 JSON 提供 JSON Schema，例如：

```text
schemas/search_features.schema.json
schemas/geo_tools.schema.json
schemas/production_job_payload.schema.json
```

> **JSON 负责吸收变化，正式表负责承载已经被业务证明稳定的关系。**

## 八、事实不漂移机制

### 8.1 权威矩阵

| 数据类型 | 唯一事实源 | 镜像或缓存 |
|---|---|---|
| 正文与人工编辑内容 | Markdown | 搜索索引、摘要缓存 |
| 内容身份、URL、作者关联 | SQLite | Frontmatter 少量镜像 |
| 实体、意图、发现渠道 | SQLite | 导出清单 |
| 排名、指标、评论事件 | SQLite | 日报、信号表 |
| 原始平台响应 | `data/raw/` | manifest 与 `source_ref` |
| 派生异常判断 | `signals`，可重算 | 日报消费状态 |

这不是“两个事实源互相同步”，而是**不同类型事实分别只有一个权威位置**。

### 8.2 写入规则

标准采集链路：

```text
原始抓取落盘
  → 生成批次 manifest
  → 身份解析与正文 Markdown 落盘
  → 写核心事实表
  → 写指标观测
  → 计算 signals
  → 刷新日报导出
```

- Frontmatter 由入库器生成，不作为人工维护元数据的入口。
- Wiki Viewer 修改正文时，只写 Markdown；保存后触发轻量 `reindex` 更新 `file_hash`、标题缓存和全文索引。
- 标签、实体、意图通过数据库编辑命令或管理页面修改，不通过手改 Frontmatter 修改。
- 外部文章正文若发生变化，不静默覆盖；记录新哈希并产生 `external_content_changed` 信号。

### 8.3 对账不是只看文件哈希

`reconcile.py` 每晚执行四类检查：

1. **文件完整性**：`md_path` 是否存在，文件哈希是否变化。
2. **身份完整性**：外部 ID 是否出现一对多、同 URL 多内容或孤立 hit。
3. **事实连续性**：关键采集源当天是否缺快照，指标是否出现异常倒退或单位变化。
4. **引用完整性**：GEO relation、comment event、production output 是否指向存在的主体。

对账报告写入：

```text
data/hub/reports/reconcile/2026-07-14.md
```

严重问题同时写入 `signals`。因此漂移不仅能被发现，还能被日报 Agent 看见。

### 8.4 实时监听的取舍

不强制依赖 `fswatch`。Wiki Viewer 正常保存时主动调用 `reindex`，覆盖 95% 的人工编辑；每晚全量对账负责兜底。这样既没有常驻监听器的运维负担，也不会把漂移发现时间无限拖长。

## 九、Agent 友好协议

### 9.1 Agent 的三层访问自由

**第一层：标准视图，适合日报和常见分析。** Agent 优先查视图，不必记忆复杂 JOIN。

**第二层：核心表直查，适合探索性分析。** Agent 可以自由写只读 SQL，不被某个 API 锁死。

**第三层：Markdown 与原始文件保底。** 数据库异常时仍可通过文件名、Frontmatter 和 `grep` 找正文；需要核查证据时可顺 `source_ref` 打开原始文件。

可选的 `get_content_context`、`get_daily_signals` 等工具只是便利层，不是强制入口。

### 9.2 `AGENT_GUIDE.md` 必须包含什么

控制在约 200 行以内，至少包含：

- 14 张表一句话说明；
- 五个核心概念；
- 五个日报视图及示例 SQL；
- Markdown 目录和 `source_ref` 解析方式；
- 指标字典查询示例；
- 只读红线和写入命令；
- WX 搜索与 MP 库存隔离规则；
- 时间字段统一采用 ISO 8601，日报按业务时区计算；
- 数据置信度与缺失值处理方式。

Agent 进入项目的推荐顺序：

```text
读 AGENT_GUIDE.md
  → 查 v_daily_overview
  → 查具体标准视图或核心表
  → 打开 2~5 篇相关 Markdown
  → 必要时追溯 source_ref
```

### 9.3 Agent 写权限

默认 Agent 对核心事实表只读。允许通过受控命令执行以下操作：

- 标记 signal 已消费或忽略；
- 新建 production job；
- 更新内容实体与意图；
- 创建人工备注或修复任务。

不允许 Agent 直接修改历史观测、搜索快照、发现记录和原始文件。需要修复时写 correction job，由适配器重放原始证据。

> **Agent 的自由应该体现在“可以自由观察和组合事实”，而不是“可以随意改写事实”。**

## 十、日报标准视图

SQLite 的日期函数对时区和不规则采集频率容易出错，因此 v3.2 不使用简单的“今天减昨天固定日期 JOIN”。标准视图优先选取每个业务日最后一次观测，并允许由日报脚本传入业务日期和时区。

### 10.1 `v_content_latest_metrics`

每个主体、每个指标的最新值。

```sql
CREATE VIEW v_content_latest_metrics AS
WITH ranked AS (
  SELECT o.*,
         ROW_NUMBER() OVER (
           PARTITION BY subject_type, subject_id, metric_key
           ORDER BY observed_at DESC
         ) AS rn
  FROM metric_observations o
)
SELECT * FROM ranked WHERE rn = 1;
```

### 10.2 `v_daily_read_spikes`

建议由脚本或参数化查询计算：对累计型阅读指标，比较最近两个有效观测点，除以真实时间差得到增速，再结合账号规模、发布时间和历史基线产生 signal。不能只按自然日机械相减。

视图提供基础数据：

```sql
CREATE VIEW v_metric_recent_pairs AS
WITH ranked AS (
  SELECT o.*,
         ROW_NUMBER() OVER (
           PARTITION BY subject_type, subject_id, metric_key
           ORDER BY observed_at DESC
         ) AS rn
  FROM metric_observations o
)
SELECT cur.subject_type, cur.subject_id, cur.metric_key,
       cur.observed_at AS current_at, cur.numeric_value AS current_value,
       prev.observed_at AS previous_at, prev.numeric_value AS previous_value
FROM ranked cur
JOIN ranked prev
  ON prev.subject_type = cur.subject_type
 AND prev.subject_id = cur.subject_id
 AND prev.metric_key = cur.metric_key
WHERE cur.rn = 1 AND prev.rn = 2;
```

### 10.3 `v_new_keyword_entries`

“新收录”必须以**同平台、同关键词、历史从未出现**为准，同时排除未解析身份的重复 URL。

```sql
CREATE VIEW v_first_keyword_appearance AS
SELECT s.platform, s.keyword, h.content_id,
       MIN(s.captured_at) AS first_appeared_at,
       MIN(h.rank) AS best_initial_rank
FROM search_hits h
JOIN search_snapshots s ON s.snapshot_id = h.snapshot_id
WHERE h.content_id IS NOT NULL
GROUP BY s.platform, s.keyword, h.content_id;
```

日报用业务日期筛选 `first_appeared_at` 即可。

### 10.4 `v_rank_changes`

对每个“平台 + 关键词 + 内容”取最近两次排名，计算上升、下降和掉榜。掉榜不能仅看没有 hit，还必须确认该关键词当天采集成功。

### 10.5 `v_new_creators`

直接筛选 `creators.first_seen_at`，同时联结首次发现内容和 discovery，避免只报一个没有上下文的账号名字。

### 10.6 `v_comment_anomalies`

从 `comment_events` 查询近期开启的 `missing`、`deleted_confirmed`、`hidden_suspected`，并联结文章标题与评论正文。

### 10.7 `v_daily_overview`

日报入口视图或导出表，汇总当天各类 signal 数量、最高 severity 和未消费数量。Agent 先看 overview，再钻取具体信号。

推荐日报过程：

```text
1. 查 v_daily_overview
2. 打开 read_spike / new_keyword / rank_change / new_creator / comment_anomaly
3. 去重相同主体的多条信号
4. 阅读最值得解释的 2~5 篇 Markdown
5. 必要时核查原始 source_ref
6. 生成“发生了什么—为什么值得注意—建议动作”日报
7. 把已采用 signal 标记为 consumed
```

## 十一、身份解析与去重

### 11.1 自动解析优先级

1. 平台稳定外部 ID；
2. 规范化 canonical URL；
3. 平台账号 ID + 文章 ID 组合；
4. 完整正文哈希；
5. 标题、作者、发布时间和正文相似度综合候选。

### 11.2 禁止自动合并的情况

- 同标题但作者不同；
- AI 回答问题相同但时间、模式或上下文不同；
- 同一外部文章发生实质正文更新；
- 视频搬运、转载和原作关系不明确；
- 仅凭标题相似或 URL 参数相似。

低置信候选进入人工合并队列。合并操作需要保存旧 ID 到新 ID 的映射，不能直接删除历史身份。

## 十二、平台接入契约

每个适配器只需实现六类输出：

```text
RawBatch
ContentUpsert
IdentifierUpsert
DiscoveryAppend
SnapshotAppend / RelationAppend
MetricObservationAppend
```

新平台接入步骤：

1. 在 `metric_definitions` 注册指标；
2. 定义外部 ID 命名空间；
3. 实现原始批次落盘和 manifest；
4. 实现内容与发现适配；
5. 平台独有字段先进入 schema 校验的 `payload_json`；
6. 加入对账检查和一个最小日报查询。

适配器不得直接生成跨平台综合热度。综合分数由 signals 计算层产生，并记录 `model_version`。

## 十三、部署与运行建议

### 13.1 SQLite 设置

```sql
PRAGMA journal_mode = WAL;
PRAGMA foreign_keys = ON;
PRAGMA busy_timeout = 5000;
PRAGMA synchronous = NORMAL;
```

- 单写者或写入队列串行化，避免多个脚本同时长期写锁。
- 读 Agent 使用只读连接：`file:content_hub.sqlite?mode=ro`。
- 每日使用 SQLite Online Backup API 或 `.backup`，不能只复制正在写入的主文件。
- 每周执行 `PRAGMA integrity_check`，每月检查索引和数据库体积。

### 13.2 时间标准

数据库保存带时区的 ISO 8601 时间或统一 UTC；每个 snapshot 同时明确平台和业务时区。日报脚本传入 `report_date` 和 `timezone`，禁止依赖服务器本地时间或 SQLite 默认 `now` 推断业务日。

### 13.3 何时迁移 PostgreSQL

出现以下任一情况时再迁移：

- 多个采集器需要持续并发写；
- 数据库写锁频繁影响任务；
- 时序观测达到数亿级；
- 需要多人远程权限、审计和高可用；
- 复杂分析查询明显阻塞运行写入。

迁移时保持表契约、ID、Markdown 路径和适配器事件结构不变。

## 十四、渐进迁移路线

### 阶段 0：冻结契约

- 建立 ID、时间、命名空间和 metric key 规范；
- 创建 `taxonomy.yaml`、JSON Schema 和 `AGENT_GUIDE.md`；
- 不改旧系统输出。

### 阶段 1：只读统一索引

- 从现有 JSON、SQLite 和 Markdown 灌入 14 张表；
- 构建五类日报查询与 overview；
- Agent 先使用 hub 只读分析；
- 所有问题仍可回退旧系统核验。

### 阶段 2：适配器双写

- 各采集器完成原始落盘后，同时向 hub 追加事实；
- 每日对比旧系统与 hub 的快照数量、hit 数量、内容数量和关键指标；
- 不一致只修适配器，不让 Agent 人工补历史事实。

### 阶段 3：信号与日报接管

- 阅读暴增、排名变化、新收录、新账号、评论异常写入 signals；
- 连续 14 天与人工日报核对；
- 修正阈值、时区和采集缺失判断。

### 阶段 4：查询入口切换

- Wiki Viewer、分析脚本和 Agent 默认读取 hub；
- 可选工具接口基于相同视图构建；
- 旧系统继续保留只读回退能力。

### 阶段 5：停止旧派生层

- 连续 30 天核心数据对账一致；
- 停止旧 `monitor-data.json` 等派生文件作为事实源；
- 原始文件与旧数据库按保留策略归档，不立即删除。

## 十五、验收标准

### 15.1 身份与发现

- 同一文章被 WX、MP、GEO 发现：一个 `content_id`、一份 Markdown、三条 discovery。
- 搜索选题查询不出现只被 MP 抓到、从未进入搜索结果的文章。
- 外部标识冲突可被对账检测并进入修复队列。

### 15.2 搜索与指标

- 任意平台、关键词和时间可回放当时的排名与页面特征。
- 不同平台同名指标仍保留独立 `metric_key`。
- 阅读暴增计算使用真实观测间隔，不把采集延迟误判成增长。
- 掉榜判断能区分“未出现”与“当天采集失败”。

### 15.3 Markdown 与漂移

- Agent 能通过标题或 ID 找到正文，图片相对路径有效。
- Wiki 人工修改后能触发 reindex；漏触发时每晚对账可发现。
- 外部不可变正文被修改时产生报警和信号。
- 元数据只通过 SQLite 修改，不依赖手改 Frontmatter。

### 15.4 GEO 与评论

- 每个 AI 回答有独立 Markdown 和 `answer_id`。
- 文本引用、搜索结果和图片引用保持不同关系类型与位置。
- 评论从 visible 到 missing 再到 deleted_confirmed 的历史可查询。

### 15.5 Agent 与容错

- 中间件全部关闭时，Agent 仍能只靠只读 SQL + Markdown 生成完整日报。
- 数据库无法打开时，Agent 能通过 Markdown 和 raw manifest 完成有限内容检索与证据核查。
- `AGENT_GUIDE.md` 足以让一个首次进入项目的 Agent 在无需口头解释的情况下完成标准日报。

## 十六、实施优先级

### P0：先建立能产生价值的最小闭环

1. 建 14 张核心表和基础索引；

## 十七、最终判断

v3.2 保留了 v3.0 最重要的专业性：内容、发现、快照、关系和观测不再混为一谈；同时接受 SQLite 和一人团队的现实，把独立 Markdown 登记、完整血缘图、过早拆分的生产表和低价值正规化结构收拢起来。

它也修正了 v3.1 中几个潜在风险：

- 不把评论状态历史塞进难查询的 JSON；
- 不声称仅凭 Markdown 和 raw 文件就一定能完整重建全部关系，而是通过 manifest、source_ref 和适配器重放保证可恢复性；
- 不使用对不规则采集不可靠的“自然日固定 JOIN”计算暴增；
- 不让 JSON 成为不可治理的万能容器；
- 不把 Agent 锁在中间件里，也不赋予它任意改写历史事实的权限。

> **最终形态是一套“人能读、Agent 能查、脚本能写、出错能追、未来能拆”的本地内容数据底座。它不追求企业架构的形式完整，而追求业务语义在三年后仍然不需要推倒重来。**
